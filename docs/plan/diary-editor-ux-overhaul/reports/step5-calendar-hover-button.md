# Step 5: 캘린더 날짜에 마우스를 올리면 "작성" 버튼이 뜬다

## 이 단계를 왜 했나?

지금까지 다이어리를 쓰려면 캘린더에서 날짜 칸을 클릭해야 했다. 동작 자체는 문제가 없지만,
날짜 칸이 그냥 숫자만 적힌 네모라서 "여기를 누르면 일기를 쓸 수 있다"는 사실이 화면에
드러나지 않았다. 처음 보는 사람은 날짜가 클릭되는지조차 알기 어렵다.

그래서 날짜 칸에 마우스를 올렸을 때만 작은 파란색 "작성" 버튼이 떠오르게 했다.
버튼이 보이니 "누르면 뭔가 쓸 수 있구나"가 즉시 전달된다. 중요한 건, 이건 **새 기능이
아니라 새 입구**라는 점이다 — 버튼을 눌러도 결국 기존의 "날짜 클릭"과 완전히 같은 일이
일어난다. 날짜 칸을 직접 클릭하는 기존 방식도 그대로 살아 있다.

## 무엇을 만들었나?

캘린더 화면 파일(XPCalendar.jsx) 하나만 수정했다. 바뀐 내용은 세 가지다.

1. **어느 날짜에 마우스가 올라와 있는지 기억하기** — 캘린더가 "지금 마우스가 올라간
   날짜"를 하나 기억한다. 마우스가 칸에 들어오면 그 날짜를 기록하고, 나가면 지운다.
2. **그 날짜 칸에만 "작성" 버튼 겹쳐 그리기** — 기억된 날짜와 같은 칸에만, 칸의 하단
   중앙에 작은 버튼을 겹쳐서(오버레이) 표시한다. 버튼 색은 캘린더 상단 헤더와 같은
   파란 그라디언트를 재사용해서 화면 톤이 튀지 않는다.
3. **버튼 클릭이 두 번 처리되지 않게 막기** — "작성" 버튼은 날짜 칸 위에 겹쳐 있어서,
   버튼 클릭이 아래 칸 클릭으로도 번질 수 있다(개발 용어로 "이벤트 버블링" — 클릭 신호가
   겹쳐진 요소들을 뚫고 아래로 전파되는 현상). 그러면 편집 화면 진입이 두 번 실행된다.
   버튼 클릭 시 전파를 차단해서 정확히 한 번만 실행되게 했다.

한 가지 구조적 결정: HTML에서는 버튼 안에 또 버튼을 넣는 게 규격 위반이다. 날짜 칸
자체가 이미 버튼이므로, 그 안에 "작성" 버튼을 넣는 대신 날짜 칸을 투명한 상자로 한 번
감싸고, 그 상자 안에 날짜 버튼과 "작성" 버튼을 나란히(겹쳐서) 두는 방식을 썼다.

캘린더의 날짜 계산(이 달이 며칠까지 있는지, 1일이 무슨 요일인지)과 월 이동(◀ ▶) 로직은
한 줄도 건드리지 않았다.

## 데이터가 어떻게 흐르나?

```
마우스가 15일 칸에 진입
        |
        v
캘린더가 기억: "지금 호버 = 2026-07-15"
        |
        v
15일 칸 하단에 [작성] 버튼 표시  (다른 30개 칸은 그대로)
        |
        +-- 마우스가 칸을 떠나면 --> 기억 삭제, 버튼 사라짐
        |
        v
[작성] 클릭
        |
        v
클릭 전파 차단 (아래 날짜 칸까지 클릭이 번지지 않게)
        |
        v
onSelectDate("2026-07-15") 호출  <-- 날짜 칸을 직접 클릭했을 때와 100% 동일한 함수
        |
        v
2026년 7월 15일 편집 화면으로 이동
```

숫자 예시: 2026년 7월은 칸이 31개다. 마우스가 15일 위에 있는 동안 "작성" 버튼은
화면 전체에서 딱 1개만 존재한다. 버튼을 클릭하면 편집 화면 진입 함수가 정확히 1회
호출된다 — 전파 차단이 없었다면 2회(버튼 1번 + 밑의 칸 1번) 호출될 수 있는 상황이다.

## 쉽게 말하면

종이 달력의 날짜 칸 위에, 손을 가까이 가져가면 "여기에 일기 쓰기"라고 적힌 작은
포스트잇이 살짝 떠오르는 것과 같다. 포스트잇을 떼어내도(클릭해도) 달력 자체를 짚었을
때와 똑같은 페이지가 펼쳐진다. 포스트잇은 안내판일 뿐, 문은 원래 있던 그 문 하나다.

## 제대로 됐는지 어떻게 확인했나?

먼저 "이렇게 동작해야 한다"를 자동 테스트 5개로 적어 놓고(이 시점엔 당연히 실패),
그 다음 구현해서 전부 통과시켰다 (개발 용어로 TDD — 테스트를 먼저 쓰고 코드를 나중에
쓰는 방식).

- 마우스를 올리기 전에는 "작성" 버튼이 없다 — 통과
- 날짜 칸에 마우스를 올리면 "작성" 버튼이 나타난다 — 통과
- 마우스가 벗어나면 버튼이 사라진다 — 통과
- "작성" 클릭 시 편집 화면 진입이 정확히 1회, 올바른 날짜(2026-07-15)로 호출된다 — 통과
- 날짜 칸을 직접 클릭하는 기존 동작이 그대로 작동한다 — 통과

이후 프로젝트 전체 검사를 돌렸다: 코드 규칙 검사(lint) 통과, 배포용 빌드(build) 성공,
기존 것을 포함한 자동 테스트 전체 37개(기존 32 + 신규 5) 전부 통과. 기존 기능이 깨지지
않았음을 확인했다.

## 전체 그림에서 지금 어디?

```
[0] 캔버스/창 확대      완료
[1] 배경 커스텀         완료
[2] 툴바 고정 메뉴      완료
[3] 버튼 아이콘         완료
[4] 다이어리 갤러리     완료
[5] 날짜 호버 작성 버튼 완료  <-- 지금 여기
[6] 메인화면 마감       남음
```

다음은 마지막 단계인 메인화면 마감이다. 메인 화면(캘린더 + 위젯들)의 여백을 줄여 창을
꽉 차 보이게 하고, 위젯 글자를 키우고, 날씨 위젯의 "불러오는 중" 텍스트를 회전 스피너로
바꾼다. 이번 단계까지가 개별 화면 요소들의 정비였고, 마지막 단계가 전체 첫인상을
다듬는 마무리라서 순서상 맨 뒤에 있다.

## 아직 안 된 것 (솔직하게)

- 자동 테스트는 마우스 진입/이탈을 신호로만 흉내 낸 것이라, 실제 브라우저에서 버튼의
  위치·크기가 눈에 예쁘게 보이는지는 검증하지 않았다. 실행해서 눈으로 확인하는 게 좋다.
- 터치 화면(태블릿·폰)에는 "마우스 올리기"라는 개념이 없어서 이 버튼이 뜨지 않는다.
  다만 날짜 칸을 직접 탭하면 기존대로 편집 화면에 들어가므로 기능이 막히지는 않는다.
- 날짜 칸이 아주 작아지는 화면 크기에서는 "작성" 버튼이 칸 밖으로 살짝 삐져나올 수 있다.
  현재 창 크기(step 0에서 확대한 크기)에서는 문제없다.
- 메인화면 여백 축소와 날씨 스피너(step 6)는 아직 안 했다.
