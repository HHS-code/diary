# Step 4: 다이어리 모아보기 (갤러리)

## 이 단계를 왜 했나?

지금까지 다이어리를 보려면 캘린더에서 날짜를 하나씩 눌러 들어가야 했다. 내가 어떤 날에 뭘 그렸는지 한눈에 훑어볼 방법이 없었다. 이번 단계는 "작성한 다이어리 전부를 작은 미리보기 이미지(썸네일)로 모아 보여주고, 하나를 누르면 바로 그 날짜의 편집 화면으로 들어가는" 모아보기 화면을 만드는 것이다.

## 무엇을 만들었나?

세 가지다.

1. **썸네일 생성 로직** (`useDiaryThumbnails.js`): 저장된 다이어리 전체를 읽어서, 다이어리가 실제로 작성된 날짜만 골라낸 뒤, 각 날짜의 캔버스를 화면에 보이지 않는 임시 캔버스에 복원하고 원본의 20% 크기 PNG 이미지로 찍어낸다. 예를 들어 1400×900 크기로 저장된 다이어리는 280×180짜리 작은 이미지가 된다. 이미지를 만든 임시 캔버스는 곧바로 폐기해서 메모리를 낭비하지 않는다. 이 과정은 전부 "읽기만" 한다 — 저장된 데이터를 건드리는 코드는 한 줄도 없다.

2. **갤러리 화면** (`DiaryGallery.jsx`): 위에서 만든 썸네일들을 격자(그리드)로 늘어놓는다. 각 칸에는 미리보기 이미지와 날짜(예: 2026-07-11)가 붙어 있고, 클릭하면 그 날짜의 편집 화면으로 이동한다. 작성된 다이어리가 하나도 없으면 "아직 작성된 다이어리가 없습니다"라고 안내한다. 패널 테두리와 파란 제목 막대는 캘린더와 똑같은 XP 스타일을 그대로 썼다.

3. **진입 버튼** (`DiaryApp.jsx` 수정): 메인 화면(캘린더+위젯) 오른쪽 위에 "모아보기" 버튼을 추가했다. 누르면 갤러리 화면으로 전환되고, 갤러리에는 ◀ 뒤로가기 버튼이 있어 메인으로 돌아올 수 있다.

## 데이터가 어떻게 흐르나?

```
localStorage (브라우저 저장소)
     │  전체 다이어리 데이터 읽기 (읽기 전용)
     ▼
날짜별로 골라내기 ── diary 탭에 그림이 있는 날짜만, 최신 날짜부터
     │  예: 30일치 저장 중 다이어리 작성은 7일 → 7개만 통과
     ▼
날짜마다 반복:
  임시 캔버스 생성 (저장 당시 크기, 예: 1400×900)
     → 저장된 그림 복원
     → 20% 축소 PNG로 찍기 (1400×900 → 280×180)
     → 임시 캔버스 폐기
     ▼
썸네일 목록 [{ 날짜, 이미지 }] 완성
     ▼
갤러리 화면이 격자로 표시
     │  썸네일 클릭
     ▼
기존의 "날짜 선택" 함수 호출 → 그 날짜의 편집 화면으로 이동
```

- "임시 캔버스"는 화면(DOM)에 붙이지 않는 오프스크린 캔버스다. 사용자 눈에는 아무것도 깜빡이지 않고, 뒤에서 이미지만 만들어낸다.
- 옛날 버전에서 저장돼 크기 정보가 없는 데이터는 당시 기본 크기였던 800×600으로 간주한다. 이 경우 썸네일은 160×120이 된다.
- 갤러리 화면을 도중에 닫으면(컴포넌트 언마운트) 남은 썸네일 생성을 중단하는 플래그를 뒀다. 다 만들지도 않을 이미지를 계속 만드는 낭비를 막기 위해서다.

## 쉽게 말하면

앨범 정리와 같다. 원본 그림(저장된 다이어리)은 서랍에 그대로 두고, 각 그림을 복사기에 넣어 20% 크기로 축소 복사한 뒤 앨범(갤러리)에 날짜순으로 꽂아둔 것이다. 앨범의 사진을 가리키면 서랍에서 원본을 꺼내 책상(편집 화면)에 펼쳐준다. 원본은 절대 오려 붙이거나 고치지 않는다.

## 만든 파일

- `src/hooks/useDiaryThumbnails.js` — 신규. 썸네일 생성 로직(날짜 골라내기 + 축소 이미지 만들기 + 화면용 훅).
- `src/hooks/useDiaryThumbnails.test.js` — 신규. 위 로직의 자동 테스트 5개.
- `src/components/DiaryGallery/DiaryGallery.jsx` — 신규. 썸네일 격자 화면.
- `src/components/DiaryApp/DiaryApp.jsx` — 수정. "모아보기" 버튼과 갤러리 화면 전환 추가.
- `docs/plan/diary-editor-ux-overhaul/steps/index.json` — step 4 상태 갱신.

## 제대로 됐는지 어떻게 확인했나?

약속한 검증 기준(AC)은 "lint, build, test 전부 통과"였고, 셋 다 직접 실행해 통과를 확인했다.

- `npm run lint` (코드 스타일·실수 검사기): 통과.
- `npm run build` (실제 배포용으로 조립): 통과.
- `npm run test` (자동 테스트): 32개 전부 통과. 이 중 5개가 이번에 새로 쓴 테스트다.

새 테스트는 구현보다 먼저 작성해 일부러 실패를 확인한 뒤(TDD), 구현으로 통과시켰다. 특히 썸네일 크기 검증은 눈대중이 아니라, 만들어진 PNG 파일의 머리 부분(파일 맨 앞에 가로·세로 픽셀 수가 기록되는 규격 영역)을 직접 읽어 "800×600 캔버스 → 정확히 160×120", "1400×900 캔버스 → 정확히 280×180"임을 숫자로 확인했다. 또 "다이어리를 안 쓴 날짜, 영화 탭만 있는 날짜는 갤러리에 안 나온다"와 "최신 날짜가 먼저 온다"도 테스트로 못 박았다.

## 전체 그림에서 지금 어디?

```
[완료] step 0  캔버스/창 크게 키우기
[완료] step 1  캔버스 배경 색상·이미지 커스텀
[완료] step 2  오브젝트 조작 메뉴 고정 위치로
[완료] step 3  버튼들 아이콘으로 교체
[완료] step 4  다이어리 모아보기 (갤러리)   ← 지금 여기
[예정] step 5  캘린더 날짜에 마우스 올리면 "작성" 버튼
[예정] step 6  메인 화면 여백 줄이기 + 날씨 로딩 스피너
```

다음은 step 5다. 갤러리로 "이미 쓴 다이어리"에 빨리 가는 길을 만들었으니, 이제 "새로 쓰러 가는" 길을 빠르게 하는 차례 — 캘린더 날짜에 마우스를 올리면 바로 작성 버튼이 뜨게 한다.

## 아직 안 된 것 (솔직하게)

- **썸네일은 매번 새로 만든다.** 갤러리를 열 때마다 모든 다이어리의 축소 이미지를 처음부터 다시 생성한다. 다이어리가 수십 개를 넘어가면 갤러리가 뜨는 데 몇 초씩 걸릴 수 있다. 캐싱(한 번 만든 이미지 재사용)과 페이지 나누기는 이번 범위에서 하지 않기로 합의된 사항이라 일부러 뺐다.
- **갤러리에서 지우거나 관리할 수 없다.** 보기와 이동만 된다. 삭제/일괄 관리는 계획 문서에서 범위 밖으로 못 박은 항목이다.
- **로딩 중 표시가 단순 텍스트다.** 썸네일 생성 동안 "불러오는 중…"이라는 글자만 보인다. 회전 스피너 같은 시각 효과는 넣지 않았다(step 6의 날씨 위젯 스피너와는 별개 사안이며, 갤러리 쪽은 요구사항에 없었다).
- **영화 탭은 갤러리에 안 나온다.** 이번 phase 전체가 다이어리 탭만 다루기로 합의돼 있어, 갤러리도 diary 탭 데이터만 읽는다.
