# Step 3: weather-widget

## 이 단계를 왜 했나?

메인 화면 우측 위젯 3개(할 일, 날씨, 시계) 중 날씨 위젯을 이번 단계에서 만든다. 지금 있는 곳의 실시간 기온과 날씨 상태(맑음/흐림/비 등)를 보여주는 부품이다. 지난 단계에서 만든 시계 위젯과 마찬가지로, 이 위젯 하나만 독립적으로 완성하고 실제 diary 화면에 붙이는 건 하지 않는다(5단계에서 한다).

## 무엇을 만들었나?

`src/components/WeatherWidget/WeatherWidget.jsx`라는 새 화면 부품을 만들었다.

- 화면에 놓이면 곧바로 브라우저에 "위치 정보를 알려줘도 될까요?"라고 물어보는 권한 프롬프트가 뜬다(Geolocation API — 브라우저가 위치를 물어보는 표준 기능).
  - 사용자가 **허용**하면: 실제 현재 위치 좌표로 날씨를 가져온다.
  - 사용자가 **거부**하거나, 브라우저가 위치 기능 자체를 지원하지 않으면: 서울 좌표(위도 37.5665, 경도 126.978)로 대신 가져온다. 에러 화면을 띄우지 않고 조용히 서울로 바꿔치기한다.
- 좌표가 정해지면 Open-Meteo(무료, 회원가입·API 키 불필요한 날씨 API)에 "지금 날씨 알려줘"라고 요청한다.
- 응답으로 받은 기온(섭씨, 정수로 반올림)과 날씨 코드(숫자)를 화면에 보여준다. 날씨 코드는 세세한 전체 표를 다 만들지 않고, 맑음/구름/안개/비/눈/뇌우 정도의 큰 그룹으로만 나눠 이모지와 한글 한 단어로 바꿔 보여준다.
- 요청이 끝나기 전에는 "날씨 정보를 불러오는 중..."이라는 문구를, API 호출이 실패하면 "날씨 정보를 불러올 수 없습니다"라는 안내 문구를 보여준다.
- 이 위젯도 바깥에서 아무 값도 받지 않는다(props 없음) — 그냥 놓으면 알아서 지금 날씨를 보여준다. 어떤 날짜를 편집 중인지와는 무관하게 항상 "지금 이 순간"의 날씨만 보여준다(과거·미래 날씨는 다루지 않음).
- 전체를 시계 위젯과 같은 톤(진한 파란 테두리 + 베이지 배경 + 그림자)의 XP 스타일 카드로 감쌌다.

## 데이터가 어떻게 흐르나?

```
브라우저 위치 권한 프롬프트
        │
        ├─ 허용 ──────────────► 실제 좌표 (예: 위도 35.18, 경도 129.08)
        │
        └─ 거부/미지원 ───────► 서울 좌표 (위도 37.5665, 경도 126.978)
        │
        ▼
Open-Meteo에 "이 좌표 지금 날씨 뭐야?" 요청
  https://api.open-meteo.com/v1/forecast?latitude=..&longitude=..&current_weather=true
        │
        │  응답 예시: { temperature: 31.2, weathercode: 0 }
        ▼
화면 코드 변환
  temperature 31.2 → 반올림 → "31°C"
  weathercode 0 → "☀️ 맑음"
        ▼
카드 안에 "☀️ / 31°C / 맑음" 표시
```

실제로 브라우저에서 위치를 부산 좌표(위도 35.18, 경도 129.08)로 설정하고 테스트했을 때 "☀️ / 31°C / 맑음"이 떴고, 위치를 거부하도록 설정했을 때는 자동으로 서울 좌표로 바뀌어 "☀️ / 30°C / 맑음"이 떴다. 두 경우 모두 몇 초 안에 결과가 나타났다.

## 쉽게 말하면

이 위젯은 "창밖을 대신 내다봐 주는 작은 창문"과 같다. 스스로는 눈이 없으니 먼저 "내가 지금 어디 있는지" 브라우저에 물어보고, 위치를 알려주기 싫다고 하면 그냥 서울 하늘을 대신 보여준다. 그다음 진짜 기상대(Open-Meteo)에 전화를 걸어 "거기 지금 날씨 어때요?"라고 물어보고, 들은 대답을 이모지 한 개와 숫자 하나로 요약해서 보여줄 뿐이다.

## 만든 파일

- `src/components/WeatherWidget/WeatherWidget.jsx` (신규) — 현재 위치(또는 서울)의 실시간 날씨를 보여주는 위젯.

## 제대로 됐는지 어떻게 확인했나?

1. **린트(코드 스타일 자동 검사) 통과**: `npm run lint`를 돌려 에러 없이 통과했다.
2. **브라우저에서 직접 확인**: 개발 서버를 띄우고 diary 창 안에 날씨 위젯을 임시로 띄워 자동화 브라우저(Playwright)로 확인했다(확인 후 임시 코드는 원상복구해 이번 커밋에는 남기지 않았다).
   - **위치 허용 경로**: 위치를 부산 좌표로 설정하고 diary 창을 열었더니, 네트워크 요청 목록에 `api.open-meteo.com`으로 나간 요청이 200(성공) 응답으로 찍혔고, 화면에는 몇 초 안에 "☀️ / 31°C / 맑음"이 나타났다 — 확인됨.
   - **위치 거부 경로**: 위치 요청이 거부되도록 만들고 diary 창을 다시 열었더니, 서울 좌표(37.5665, 126.978)로 자동 전환되어 같은 방식으로 `api.open-meteo.com` 요청이 200 응답을 받았고, 화면에 "☀️ / 30°C / 맑음"이 나타났다 — 확인됨.
   - 두 경로 모두 콘솔에 에러가 없었다.

## 전체 그림에서 지금 어디?

```
[0] storage-todos (완료)
[1] xp-calendar (완료)
[2] analog-clock-widget (완료)
[3] weather-widget (완료)   <- 지금 여기
 └─ 4  todo-widget
 └─ 5  main-screen-wiring    <- 여기서 이 날씨 위젯을 실제 diary 화면에 끼운다
```

날씨 위젯 하나를 독립적으로 완성했다. 다음 단계는 할 일 체크리스트 위젯(step 4)을 같은 방식으로 따로 만들고, 마지막 5단계에서 캘린더 + 위젯 3개를 한 화면에 모아 조립한다. 지금 diary 창을 열어도 아직 이 날씨 위젯은 보이지 않는 게 정상이다.

## 아직 안 된 것 (솔직하게)

- 실제 diary 화면에는 아직 연결되지 않았다. 지금 diary 창을 열면 날씨 위젯이 보이지 않는다. 연결은 5단계에서 한다.
- 도시를 직접 검색하거나 다른 지역으로 수동으로 바꾸는 기능은 없다 — 계획 문서에 "자동 감지 또는 서울 고정만"으로 범위가 정해져 있어서 그 범위만 지켰다.
- 위치 권한 프롬프트에 사용자가 아무 응답도 하지 않고 계속 무시하면(허용도 거부도 안 누르면) 위젯이 "불러오는 중..." 상태에서 멈춰 있을 수 있다. 보통은 사용자가 허용/거부 중 하나를 누르므로 실사용에서 문제될 가능성은 낮지만, 계획 문서에 이 경우에 대한 시간제한(타임아웃) 처리는 명시되어 있지 않아 별도로 만들지 않았다.
