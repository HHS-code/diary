# ADR — 스티커 스튜디오 주요 결정

## ADR-1. 독립 데스크톱 앱으로 진입 (다이어리 캔버스 사이드바 확장 아님)

**맥락**: "스티커 전용 에디터 메뉴"를 다이어리 캔버스 사이드바에 패널/모드로 추가할지, 완전히 별도의 데스크톱 앱(Window)으로 만들지 정해야 한다.

**선택지**:
- (A) 데스크톱에 새 아이콘 추가 → `App.jsx`에 `isStickerStudioOpen` state → 다이어리와 동일한 패턴으로 독립 `Window` 열림
- (B) `DiaryCanvas` 사이드바에 "새 스티커 만들기" 버튼 → 같은 화면 안에서 편집 모드로 전환

**결정**: (A) 독립 데스크톱 앱.

**결과(트레이드오프)**: 다이어리 캔버스 상태와 완전히 분리되어 구현이 단순하고, 로드맵이 원래 표현한 "에디터 메뉴"라는 이름에도 부합한다. 대신 `Window` 컴포넌트가 다중 창 스택을 지원하지 않으므로(현재 다이어리 Window 하나만 동시에 열리는 구조), 다이어리를 열어둔 채 스티커 스튜디오를 동시에 열 수 있는지는 기존 `Window` 구현의 제약을 그대로 따른다 — 이 phase에서 다중 창 지원 자체를 새로 만들지 않는다.

## ADR-2. useFabricCanvas에 캔버스 크기 파라미터 추가 (새 훅을 만들지 않음)

**맥락**: 기존 `useFabricCanvas`는 `LOGICAL_CANVAS = { width: 1600, height: 1000 }`가 모듈 상수로 하드코딩되어 있다. 스티커 스튜디오는 작은 정사각형(예: 512×512) 캔버스가 필요하다.

**선택지**:
- (A) `useFabricCanvas`에 `options.logicalSize` 같은 파라미터를 추가해 다이어리·스티커 스튜디오가 같은 훅을 공유
- (B) 스티커 스튜디오 전용 새 훅(`useStickerCanvas` 등)을 별도로 만듦

**결정**: (A) 기존 훅에 크기 파라미터 추가.

**결과(트레이드오프)**: 캔버스 생명주기(생성/dispose/저장 디바운스/에셋 참조 해석)를 한 곳에서 유지보수하고, `usePaintTools`/`useCanvasHistory`/`useCanvasKeyboardShortcuts` 등 `fabricCanvasRef` 기반 훅들이 캔버스 크기와 무관하게 그대로 재사용된다. 대신 `useFabricCanvas`의 기존 시그니처가 확장되므로, 다이어리 쪽 호출부가 새 파라미터를 몰라도 기존 동작(1600×1000)이 그대로 유지되도록 기본값 처리가 필요하다.

## ADR-3. 스티커도 기존 assetStorage에 새 타입('sticker')으로 저장 (별도 저장소 신설 안 함)

**맥락**: 완성된 스티커를 어디에 저장할지 정해야 한다. 기존 `assetStorage.js`(IndexedDB)는 `type: "image" | "font"`만 지원한다.

**선택지**:
- (A) `assetStorage`의 `type`에 `'sticker'`를 추가해 같은 object store를 공유
- (B) 스티커 전용 별도 IndexedDB(별도 DB_NAME) 신설

**결정**: (A) 기존 저장소에 타입 추가.

**결과(트레이드오프)**: `saveAsset`/`getAsset`/`listAssets`/`createAssetObjectURL`/마이그레이션 로직을 그대로 재사용하고, `AssetImportPanel`이 이미지·폰트와 같은 방식으로 스티커도 노출할 수 있다. 대신 `useAssetLibrary.js`가 `images`/`fonts` 두 상태만 관리하던 구조에 `stickers` 상태가 추가되어야 하고, `AssetRecord.type`의 JSDoc 유니온 타입(`"image" | "font"`)도 `"image" | "font" | "sticker"`로 넓혀야 한다 — 이 타입을 참조하는 기존 코드(마이그레이션, 배치 로직)가 `'sticker'`를 다른 타입과 혼동하지 않도록 각 분기를 점검해야 한다.

## ADR-4. 저장은 항상 "다른 이름으로 저장" — 덮어쓰기 없음

**맥락**: 저장된 스티커를 다시 열어 편집했을 때, 원본을 덮어쓸지 새 항목으로 남길지 정해야 한다.

**선택지**:
- (A) 항상 새 `assetId`로 저장(원본 보존), 편집을 아무리 반복해도 원본은 최초 저장 상태 그대로 남음
- (B) "덮어쓰기"와 "다른 이름으로 저장"을 사용자가 선택하는 UI 제공

**결정**: (A) 항상 새 항목으로 저장.

**결과(트레이드오프)**: 저장 로직이 단순해지고(항상 `saveAsset` 신규 호출), 실수로 원본을 잃어버리는 상황이 구조적으로 발생하지 않는다. 대신 같은 스티커를 여러 번 다듬으면 편집 버전마다 별도 항목이 쌓여 "내 스티커" 목록이 빠르게 늘어난다 — 이번 phase는 목록 정리(삭제 등) UI를 범위에 넣지 않았으므로(PRD Out-of-scope), 목록이 길어지는 것 자체를 문제로 다루지 않는다.

## ADR-5. 누끼따기는 Fabric clipPath로 미리보기, 최종은 rasterize해서 새 이미지로 교체

**맥락**: 사용자가 올가미로 그린 자유곡선 안쪽만 남기고 바깥을 투명하게 만들어야 한다. Fabric.js에는 오브젝트에 다른 오브젝트를 마스크로 지정하는 `clipPath` 기능이 있다.

**선택지**:
- (A) 사용자가 그린 `Path`를 이미지의 `clipPath`로 지정해 실시간 미리보기 → 확정 시 `toCanvasElement()`로 실제 픽셀을 구워(rasterize) 새 `FabricImage`로 캔버스 오브젝트를 교체
- (B) 처음부터 오프스크린 Canvas 2D에서 `globalCompositeOperation`으로 직접 픽셀을 마스킹

**결정**: (A) clipPath로 미리보기, 확정 시 rasterize.

**결과(트레이드오프)**: 곡선을 그리는 동안 Fabric의 표준 렌더 파이프라인 안에서 자연스러운 실시간 미리보기가 가능하다. 대신 확정 시점에 `clipPath`가 적용된 상태를 실제 픽셀로 구워야(rasterize) 이후 단계(흰색 테두리 생성 등 순수 이미지 처리)와 파이프라인이 이어진다 — clipPath 자체는 오브젝트 트리에 의존하는 "표현"이라 흰색 테두리 생성처럼 픽셀 단위로 처리하는 다음 단계와 직접 합성할 수 없기 때문이다. 사용자가 그린 곡선이 완전히 닫히지 않은 경우(시작점과 끝점이 안 만남), Fabric Path의 fill 규칙상 자동으로 직선으로 이어져 채워지는 동작을 그대로 받아들인다(별도 보정 로직을 만들지 않음).

## ADR-6. 흰색 테두리는 오프스크린 캔버스의 다방향 스탬프 기법으로 생성

**맥락**: 알파 채널이 있는 이미지의 실루엣을 따라 N픽셀 두께의 흰색 테두리를 만들어야 한다. Fabric.js 필터(컨볼루션 등)는 색상 변환용이라 이 용도에 직접 맞지 않는다.

**선택지**:
- (A) 오프스크린 캔버스에서 이미지를 흰색 실루엣으로 단색화한 뒤, 여러 각도(예: 16~32방향)로 반지름만큼 반복해서 `drawImage`로 스탬핑해 확장된 테두리를 만들고, 그 위에 원본을 다시 그려 순수 테두리만 남긴다
- (B) 픽셀 단위 거리 변환(distance transform) 알고리즘을 직접 구현

**결정**: (A) 다방향 스탬프 기법.

**결과(트레이드오프)**: 스탬프 방향 수가 수십 회 수준의 `drawImage` 호출로 끝나 이 프로젝트 규모(작은 스티커 캔버스)에서 실시간 슬라이더 조작에도 충분히 빠르다. 대신 두께가 두꺼워질수록 방향 수를 늘리지 않으면 테두리 외곽이 다각형처럼 각져 보일 수 있어, 두께에 비례해 방향 수를 조정하는 균형이 필요하다. (B)의 거리 변환은 더 매끈한 결과를 내지만 CPU 픽셀 루프 비용이 이미지 크기에 비례해 커져, 슬라이더로 실시간 조절하기엔 부담이 크다고 판단했다.

## ADR-7. 그리기/history/키보드 단축키 훅은 다이어리와 100% 재사용

**맥락**: `usePaintTools`, `useCanvasHistory`, `useCanvasKeyboardShortcuts`는 모두 `fabricCanvasRef`(그리고 일부는 `registerAndPlaceImage` 콜백)만 받는 독립적인 훅으로, 다이어리 캔버스 전용 로직에 결합되어 있지 않다.

**선택지**:
- (A) 세 훅을 그대로 import해 스티커 스튜디오 캔버스에 연결
- (B) 스티커 스튜디오 전용으로 복제해 별도 유지보수

**결정**: (A) 그대로 재사용.

**결과(트레이드오프)**: 코드 중복 없이 다이어리와 동일한 그리기·undo/redo·복사/붙여넣기 동작을 그대로 얻는다. 대신 이 세 훅에 향후 변경이 생기면 두 화면(다이어리, 스티커 스튜디오) 모두에 영향을 미치므로, 다이어리 전용으로 만든 가정(예: `LOGICAL_CANVAS` 참조 여부)이 섞여 있지 않은지 이번 phase에서 확인이 필요하다 — 조사 결과 세 훅 모두 캔버스 크기에 의존하지 않는 것을 확인했다.
