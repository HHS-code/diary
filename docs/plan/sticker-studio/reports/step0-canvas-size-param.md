# Step 0: canvas-size-param

## 이 단계를 왜 했나?

다이어리 캔버스는 항상 가로 1600, 세로 1000짜리 고정 크기로 그려진다. 이 크기는 코드 한 곳(`useFabricCanvas`라는 부품)에 못 박혀 있었다.

그런데 앞으로 만들 "스티커 스튜디오"는 512×512짜리 작은 정사각형 캔버스가 필요하고, 배경도 흰색이 아니라 투명해야 한다. 크기와 배경색을 못 박아 둔 부품을 그대로 쓰면 스티커 스튜디오도 강제로 1600×1000 흰 배경이 되어버린다.

그래서 이번 단계는 스티커 스튜디오 화면 자체는 아직 만들지 않고, 그 화면이 나중에 쓸 수 있도록 "캔버스 크기와 배경색을 바꿔 달라고 요청할 수 있는 문(옵션)"만 부품에 새로 뚫었다.

## 무엇을 만들었나?

`useFabricCanvas`는 화면에 그림판(캔버스)을 하나 만들고 관리해 주는 코드 조각(리액트에서는 이런 걸 "훅"이라고 부른다 — 화면 생명주기에 붙는 재사용 가능한 로직 뭉치)이다. 캔버스를 새로 만들 때 크기·배경색을 어떻게 할지 등, 추가 옵션을 4번째 자리로 넘겨받는데, 여기에 두 가지를 새로 추가했다.

- `logicalSize`: `{ width, height }` 형태로 캔버스 크기를 지정. 안 주면 원래대로 1600×1000.
- `backgroundColor`: 배경색 문자열. 안 주면 원래대로 흰색(`'#ffffff'`).

이 두 옵션을 아예 안 주면(기존 다이어리 화면이 하는 대로) 이전과 완전히 똑같이 동작한다 — 이번 단계에서 가장 중요하게 지킨 약속이다.

그리고 캔버스에 그림을 그릴 때마다 0.5초 뒤 자동 저장이 실행되는데("디바운스" — 짧은 시간 안에 여러 번 일어나는 이벤트를 마지막 것만 남기고 몰아서 처리하는 방식), 이 저장 함수가 두 번째 값으로 "지금 캔버스가 몇×몇 크기인지"도 함께 넘겨준다. 이 값도 이제 실제로 넘겨받은 크기(예: 300×300)를 반영하도록 고쳤다.

## 데이터가 어떻게 흐르나?

```
[호출하는 쪽]                        [useFabricCanvas 내부]

다이어리 화면
  useFabricCanvas(el, json, onSave)  --옵션 없음-->  logicalSize = 1600x1000 (기본값)
                                                      backgroundColor = '#ffffff' (기본값)
                                                      → 캔버스 1600x1000, 흰 배경으로 생성
                                                      → 저장 시 onSave(그림데이터, {1600,1000})

(다음 단계) 스티커 스튜디오
  useFabricCanvas(el, json, onSave,
    { logicalSize: {512,512},
      backgroundColor: 'transparent' })  -->        logicalSize = {512,512}
                                                      backgroundColor = 'transparent'
                                                      → 캔버스 512x512, 투명 배경으로 생성
                                                      → 저장 시 onSave(그림데이터, {512,512})
```

숫자 예시: 이번 테스트에서는 `{ width: 300, height: 300 }`을 넘겨 실제로 캔버스 가로/세로가 300으로 만들어지는지, 배경색을 `'transparent'`로 넘기면 진짜 투명으로 설정되는지 확인했다.

## 쉽게 말하면

기존 부품은 "무조건 1600×1000, 흰 도화지만 주는 도화지 자판기"였다. 이번에 그 자판기에 "크기/색깔 선택 버튼"을 새로 붙였다. 버튼을 안 누르면(옵션을 안 주면) 예전처럼 그냥 1600×1000 흰 도화지가 나온다. 스티커 스튜디오는 다음 단계에서 이 버튼을 눌러 "512×512, 투명"을 주문하게 된다.

## 만든 파일

- `src/hooks/useFabricCanvas.js` — `logicalSize`/`backgroundColor` 옵션 처리 추가 (기존 로직 구조는 그대로 두고 하드코딩된 값 두 곳만 옵션 기반 변수로 교체)
- `src/hooks/useFabricCanvas.test.js` — 신규 테스트 4개 추가 (기존 테스트는 전혀 건드리지 않음)

## 제대로 됐는지 어떻게 확인했나?

- `npm run lint`: 통과 (경고 없음)
- `npm run build`: 통과 (정상 빌드됨)
- `npm run test`: 전체 25개 테스트 파일, 151개 테스트 모두 통과
  - 기존 `useFabricCanvas.test.js`의 모든 테스트(8개)가 수정 없이 그대로 통과 — 옵션을 안 주는 기존 호출부(다이어리)는 동작이 조금도 바뀌지 않았다는 뜻.
  - 신규 테스트 4개도 모두 통과:
    - `logicalSize: {300,300}`을 주면 실제 캔버스 크기가 300×300이 되는지
    - `backgroundColor: 'transparent'`를 주면 실제 배경색이 투명으로 설정되는지
    - 옵션 없이 저장하면 저장 콜백이 여전히 `{1600,1000}`을 받는지 (회귀 확인)
    - `logicalSize: {300,300}`을 주고 저장하면 저장 콜백이 `{300,300}`을 받는지

## 전체 그림에서 지금 어디?

```
[0] canvas-size-param        <- 지금 여기 (완료)
[1] sticker-asset-type
[2] sticker-studio-shell
[3] sticker-image-upload-crop
[4] sticker-lasso-cutout
[5] sticker-outline
[6] sticker-save-and-reuse
```

이번 단계는 스티커 스튜디오 화면을 만들기 전에, 그 화면이 쓸 "작은 투명 캔버스"라는 재료를 부품 쪽에서 미리 준비해 둔 것이다. 다음 단계(`sticker-asset-type`)는 완성된 스티커 이미지를 저장할 저장 창고(assetStorage) 쪽에 `'sticker'`라는 새 종류를 추가하는 작업이라, 이번 단계와는 독립적으로 진행된다. 그 다음 단계(`sticker-studio-shell`)에서 실제로 데스크톱에 아이콘이 생기고, 이번에 만든 옵션(`logicalSize: 512x512, backgroundColor: 'transparent'`)을 실제로 사용하는 화면이 등장한다.

## 아직 안 된 것 (솔직하게)

- 이번 단계는 스티커 스튜디오 화면 자체를 전혀 만들지 않았다. 아이콘도, 창(Window)도, 그리기 도구 연결도 없다 — 오직 캔버스 부품이 "다른 크기/배경색도 받아줄 수 있게" 문만 열어둔 것이다.
- 512×512, 투명 배경이라는 실제 숫자는 다음 단계 이후에 코드로 등장한다. 이번 단계에서는 옵션이 "제대로 전달되고 적용되는지"만 300×300 같은 임의의 테스트 값으로 검증했다.
