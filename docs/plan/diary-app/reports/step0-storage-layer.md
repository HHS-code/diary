# Step 0: 저장소 계층 (storage-layer)

## 이 단계를 왜 했나?

다이어리 앱의 모든 데이터(캔버스에 그린 스티커, 텍스트, 사진 위치 등)는 서버 없이 기기 안에만 저장된다. 브라우저가 제공하는 localStorage(로컬스토리지 — 브라우저가 탭을 닫아도 데이터를 기억하는 저장 공간)를 사용한다.

그런데 "어떻게 저장하고 꺼내는가"를 여러 화면 컴포넌트가 각자 직접 다루면, 나중에 저장 방식이 바뀔 때 모든 화면을 고쳐야 한다. 그래서 저장/조회 로직을 한 파일에 모아 두는 것이 이 단계의 목적이다.

## 무엇을 만들었나?

`src/storage/diaryStorage.js` 파일 하나를 만들었다. 이 파일은 Fabric.js(캔버스 라이브러리)도, React(화면 라이브러리)도 모른다. 순수하게 "데이터를 어디서 읽고, 어디에 쓰는가"만 담당한다.

네 가지 기능을 제공한다:

- **전체 읽기**: 브라우저 저장소에서 전체 다이어리 데이터를 꺼낸다. 아무것도 없으면 빈 객체를 돌려준다.
- **전체 쓰기**: 전체 데이터를 저장소에 통째로 저장한다.
- **날짜+탭 단위 읽기**: "2026-07-09 날짜의 다이어리 탭 데이터 주세요"처럼 특정 날짜·탭 데이터만 꺼낸다. 없으면 빈 상태를 돌려준다.
- **날짜+탭 단위 갱신**: 특정 날짜·탭의 캔버스 데이터를 업데이트한 **새 전체 데이터**를 돌려준다. 원본 데이터는 건드리지 않는다.

## 데이터가 어떻게 흐르나?

```
브라우저 localStorage
  키: "diary-app-data"
  값: JSON 문자열
        |
        | loadAllDiaryData()  -- 읽기
        | saveAllDiaryData()  -- 쓰기
        v
  DiaryData 객체 (JavaScript 메모리 안)
  {
    "2026-07-09": {
      "diary": { canvasJSON: { ... } },
      "movie":  { canvasJSON: null  }
    }
  }
        |
        | getDatePageData(data, "2026-07-09", "diary")
        v
  { canvasJSON: { ... } }   <-- 화면 컴포넌트가 캔버스 복원에 사용

        |
        | setDatePageData(data, "2026-07-09", "diary", 새캔버스JSON)
        v
  새로운 DiaryData 객체    <-- saveAllDiaryData()로 다시 저장
```

숫자 예시: 캔버스에 스티커 3개와 텍스트 1개를 올리면, Fabric.js가 JSON으로 변환한 결과가 약 2~5 KB다. 1년치 365일을 모두 채워도 약 1~2 MB 수준으로, localStorage 용량 제한(보통 5 MB)을 초과하기 어렵다.

## 쉽게 말하면

도서관 사서와 비슷하다. 사서(diaryStorage.js)는 책(데이터)을 어디 서가(localStorage)에 꽂는지 알고, "이 날짜 다이어리 꺼내줘"라는 요청에 맞는 책을 찾아 준다. 독자(화면 컴포넌트)는 서가 구조를 몰라도 된다. 사서에게 부탁하기만 하면 된다.

```
docs/plan/diary-app/steps/index.json   (step 0 상태 completed로 갱신)

```

## 제대로 됐는지 어떻게 확인했나?

두 가지 AC(인수 기준 — 합격 여부를 판단하는 조건)를 모두 통과했다:

- `npm run build`: 에러 없이 성공. 빌드 산출물 크기 193 KB.
- `npm run lint`: 에러 없이 통과 (경고도 없음).

브라우저 화면 확인은 이 단계의 범위 밖이다. 아직 어떤 화면도 이 파일을 사용하지 않는다.

```
[Step 3] 이미지 업로드
```

다음 단계(Step 1)가 캔버스 셸인 이유: 저장소는 생겼지만 아직 캔버스 자체가 없다. 데이터를 넣고 꺼낼 그릇(캔버스)을 먼저 화면에 올려야 나머지 기능(스티커, 이미지, 텍스트)을 붙일 수 있다.

## 아직 안 된 것 (솔직하게)

- 영화 리뷰 탭의 `movieInfo`(제목·감독·별점 등)는 이번에 설계하지 않았다. PRD 2순위 범위라 Step 7 이후에 추가할 예정.
- 저장 용량이 5 MB를 초과하면 어떻게 되는지 처리하지 않았다. 현재는 `localStorage.setItem`이 실패해도 조용히 넘어간다. 실제 사용에서 문제가 생기면 그때 처리한다.
- 날짜 키 형식 유효성 검사 없음. `"YYYY-MM-DD"` 형식이 맞는지 확인하지 않는다.
