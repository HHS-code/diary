# Step 1: 진짜 지우개 — 문지르면 그린 획만 지워진다

## 이 단계를 왜 했나?

Step 0에서 연필·브러시·에어브러시로 그릴 수 있게 됐지만, 잘못 그린 선을 고칠 방법이
없었다. 그림판의 핵심은 지우개다 — 문지르면 지나간 자리만 사라지는 그 감각.

여기서 합의된 원칙이 하나 있다: **지우개는 그린 획만 지운다.** 스티커·사진·텍스트 위를
문질러도 아무 일도 일어나지 않는다. 반쯤 지워진 스티커를 옮기는 건 이상한 경험이고,
스티커를 없애고 싶으면 원래 방법(클릭해서 선택 → Delete)이 자연스럽기 때문이다.

또 하나 확인해야 했던 게 있다. 지운 상태가 **저장 후 다시 열어도 유지되는가**.
이게 안 되면 열심히 지워도 새로고침하면 지운 자국이 되살아난다. 이번 단계에서
이 부분을 자동 테스트로 증명했다.

## 무엇을 만들었나?

1. **지우개 도구 추가** — 도구 목록이 선택/연필/브러시/에어브러시에서 +지우개로 늘었다.
   지우개를 고르면 캔버스가 그리기 모드로 바뀌고, 문지른 경로가 지우개 획이 된다.
   굵기 조절도 브러시와 같은 값을 쓴다 (예: 굵기 10이면 10픽셀 폭으로 지워진다).
2. **"그린 획만" 필터** — 그린 획에는 step 0에서 `erasable`(지울 수 있음)이라는 꼬리표를
   붙여 두었다. 지우개는 이 꼬리표가 있는 것만 건드린다. 스티커·사진·텍스트는 꼬리표가
   없어서 지우개가 지나가도 무시된다.
3. **지운 결과를 저장에 태우기** — 지우기가 끝나면 기존 자동 저장(마지막 변경 후 0.5초
   뒤 저장)이 돌도록 신호를 보낸다. 이게 없으면 지운 상태가 저장되지 않는 구멍이 생긴다.
4. **저장 왕복 검증** — "지운 상태 → 저장용 데이터로 변환 → 새 캔버스에 복원" 왕복을
   테스트로 증명했다. 모아보기 화면의 썸네일 생성 경로도 같이 확인했다.

지우개는 직접 만들지 않고 `@erase2d/fabric`이라는 검증된 부품(라이브러리 — 남이 만들어
공개한 코드 묶음)을 썼다. 이 부품의 실제 사용법을 확인해 기록해 둔다: 지우기가 끝나면
부품이 `end`라는 신호를 주는데, 신호 안에 `path`(지우개가 지나간 경로)와 `targets`
(지워질 대상 목록)가 들어 있다. 기본 동작(자동 반영)을 `preventDefault()`로 막고
`commit({ path, targets })`을 직접 호출하면 반영 완료 시점을 잡을 수 있다 — 그 시점에
저장 신호를 보낸다.

## 데이터가 어떻게 흐르나?

```
지우개로 문지른다 (예: 굵기 10으로 획 위를 드래그)
        |
        v
@erase2d/fabric EraserBrush가 "end" 신호 발생
   신호 내용: path = 지운 경로, targets = erasable 꼬리표가 있고
              경로와 겹친 오브젝트만 (스티커는 애초에 목록에 안 들어옴)
        |
        v
usePaintTools의 핸들러:
   targets가 비어 있으면 -> 아무것도 안 함 (저장도 안 함)
   있으면 -> commit() 호출: 각 획에 "지워진 모양"을 새겨 넣음
        |
        v
획 오브젝트에 clipPath(ClippingGroup)가 붙는다
   = "이 획은 이 모양대로만 보여라"라는 가림막.
     지운 경로가 가림막에 구멍처럼 기록된다
        |
        v
canvas.fire('object:modified') -> 기존 자동 저장이 0.5초 뒤 실행
        |
        v
저장 데이터(JSON)에 가림막까지 포함 -> 새로고침/썸네일에서 그대로 복원
```

풀이: 지우개는 획의 점들을 실제로 삭제하는 게 아니라, 획 위에 "여기는 안 보임"이라는
가림막을 덧씌우는 방식이다. 그래서 획 하나를 여러 번 부분부분 지울 수 있다. 숫자 예시:
길이 300픽셀짜리 가로선 한가운데를 굵기 8 지우개로 세로로 문지르면, 선은 그대로 한
오브젝트인데 가운데 8픽셀 구간만 투명해져서 두 토막으로 보인다 — 테스트에서 실제로
x=150 지점 픽셀이 투명(알파 0)이고 x=50 지점은 검정으로 남는 것을 확인했다.

## 쉽게 말하면

스티커와 사진은 코팅된 카드라서 지우개가 미끄러지고, 내가 그린 낙서는 연필 자국이라
지워진다 — 같은 종이 위에 있어도 재질이 달라서 지우개가 구분한다. 그리고 지운 자국까지
포함해서 종이를 통째로 사진 찍어 보관하기 때문에(자동 저장), 내일 다시 펴도 지운 자리는
지워진 채로 있다.

## 만든 파일

- `src/hooks/usePaintTools.js` (수정) — 지우개 도구 추가. 지우개 부품(`@erase2d/fabric`)을
  쓰는 코드는 이 파일 안에만 있다. 나중에 부품을 바꿔도 이 파일만 고치면 되게 하기 위한
  경계다. 이 부품을 불러오는 것만으로 "가림막(ClippingGroup)" 형식이 복원 시스템에
  자동 등록되는 것도 확인했다 — 별도 등록 코드가 필요 없었다.
- `src/hooks/usePaintTools.test.js` (수정) — 지우개 테스트 4개 추가: 도구 선택, 그린 획만
  지워짐(스티커 무영향), 지운 뒤 저장 신호 발생, 헛스윙(아무것도 안 지움)이면 저장 신호 없음.
- `src/hooks/freeDrawingSerialization.test.js` (신규) — 저장 왕복 테스트 5개: 지운 상태
  유지, 지운 자리 픽셀이 실제로 투명한지, 획의 박제 속성(선택 불가 등) 유지, 썸네일 생성
  경로, 에어브러시 산출물 왕복.
- `docs/plan/free-drawing/steps/index.json` (수정) — step 1 완료 기록.

## 제대로 됐는지 어떻게 확인했나?

약속된 검사 3종을 모두 통과했다.

- `npm run lint` (코드 스타일 검사기) — 통과
- `npm run test` — **67개 테스트 전부 통과** (기존 58개 유지 + 이번에 9개 추가)
- `npm run build` (배포용 묶음 만들기) — 성공

핵심 증명 항목:

- 지우개가 획과 스티커를 동시에 가로질러도, 획에만 가림막이 생기고 스티커는 그대로다.
- 지운 획을 저장 데이터로 바꿨다가 새 캔버스에 복원하면: 가림막 유지, 지운 자리 픽셀
  투명, 안 지운 자리 획 유지, 선택 불가 상태 유지.
- 모아보기 썸네일을 만드는 경로(StaticCanvas)로도 지운 획이 에러 없이 그려진다.
- 에어브러시가 뿌린 점들(그룹)도 저장 왕복 후 같은 점 개수로 복원된다.

## 전체 그림에서 지금 어디?

```
[step 0] 그리기 도구(연필/브러시/에어브러시) + 획 박제   -- 완료
[step 1] 지우개 (그린 획만, 저장 왕복 검증)              -- 완료 (이번)
[step 2] 도구 팔레트 UI + 화면 연결                      -- 다음
[step 3] Ctrl+Z / Ctrl+Y 되돌리기                        -- 예정
[step 4] XP 그림판 룩 실측 매칭                          -- 예정 (수동)
```

다음이 UI(step 2)인 이유: 그리기와 지우개라는 엔진은 완성됐지만, 아직 화면에 버튼이
없어서 사용자가 도구를 고를 방법이 없다. step 2에서 XP 그림판식 도구 팔레트를 만들어
다이어리 화면에 연결하면 그때부터 실제로 써볼 수 있다.

## 아직 안 된 것 (솔직하게)

- **실제 마우스로 문지르는 동작은 자동 테스트로 못 돌렸다.** 테스트 환경(jsdom — 브라우저
  없이 브라우저 흉내를 내는 도구)은 마우스 드래그를 재현하기 어려워서, "지우개가 이 경로로
  지나갔다"는 상태를 코드로 만들어 그 이후 전 과정을 검증했다. 포인터를 눌러서 끄는 앞
  구간은 지우개 부품이 담당하는 검증된 영역이지만, step 2에서 화면에 연결한 뒤 브라우저에서
  직접 문질러 보는 수동 확인이 필요하다.
- **아직 화면에서 지우개를 고를 수 없다.** 버튼이 없다 (step 2).
- **지우기를 되돌릴 수 없다.** Ctrl+Z는 step 3.
- 저장에 포함할 속성 목록이 본체 코드와 테스트에 각각 한 벌씩 있다(본체가 목록을 외부
  공개하지 않아서 테스트에 복사). 본체 목록이 바뀌면 테스트 쪽도 같이 고쳐야 한다 —
  테스트 파일에 주석으로 표시해 두었다.
