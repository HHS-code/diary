# Step 2: 그리기 도구 패널 UI (paint-toolbox-ui)

## 이 단계를 왜 했나?

Step 0과 1에서 그리기 엔진은 이미 완성됐다 — 연필/브러시/에어브러시로 그리고, 지우개로 지우는 로직이 코드 안에 다 들어 있었다. 문제는 그걸 켤 스위치가 화면에 없었다는 것. 도구를 고르고, 색을 고르고, 굵기를 고를 버튼이 없으니 사용자 입장에서는 아무것도 달라진 게 없는 상태였다.

이번 단계는 그 스위치를 만드는 일이다. XP 그림판처럼 생긴 도구 패널을 사이드바 맨 위에 붙여서, 엔진과 사용자를 연결했다.

## 무엇을 만들었나?

사이드바 맨 위(스티커 팔레트 위)에 "그리기" 패널이 새로 생겼다. 위에서 아래로 세 구역이다.

1. **도구 버튼 5개** (2열 격자): 선택(화살표), 연필, 브러시, 에어브러시, 지우개. 지금 쓰는 도구는 XP 그림판처럼 눌린(움푹 들어간) 모양으로 표시된다.
2. **굵기 버튼 4개**: 2, 4, 6, 8 픽셀 굵기. 각 버튼 안에 그 굵기만큼의 가로선이 그려져 있어 눈으로 고를 수 있다. 브러시/에어브러시/지우개일 때만 켜지고, 선택 도구(그릴 게 없음)와 연필(항상 1픽셀 고정)일 때는 흐려지며 눌리지 않는다.
3. **색상 팔레트 28색** (2행 × 14열): XP 그림판의 기본 팔레트 색 그대로. 왼쪽에 지금 고른 색을 보여주는 큰 칸이 하나 있다. 색 선택은 어떤 도구에서든 항상 가능하다 — 지우개를 쓰다가 미리 다음 색을 골라둬도 된다.

중요한 설계 원칙 하나: 이 패널은 **순수 UI**다. 그리기 엔진(Fabric.js — 캔버스에 그림을 그려주는 라이브러리)을 직접 만지지 않고, "버튼이 눌리면 알려주기"만 한다. 실제 브러시 교체는 step 0·1에서 만든 훅(React에서 상태와 로직을 묶어 재사용하는 함수 단위)이 담당한다. 이렇게 나눠두면 나중에 버튼 모양을 아무리 바꿔도 그리기 동작이 깨질 일이 없다.

## 데이터가 어떻게 흐르나?

```
사용자 클릭                          화면 반영
    |                                   ^
    v                                   |
[PaintToolbox 패널]  --"브러시 골랐어요"-->  [usePaintTools 훅]
 (버튼만 있는 UI)      onToolChange('brush')   (도구 상태 보관 +
    ^                                          캔버스에 브러시 장착)
    |                                            |
    +---- tool/color/width 현재값 <--------------+
                                                 v
                                          [Fabric.js 캔버스]
                                          (실제로 선이 그려짐)
```

풀어서 말하면: 패널의 브러시 버튼을 누르면 패널이 "brush를 골랐다"고 훅에 전달한다. 훅은 도구 상태를 brush로 바꾸고, 캔버스에 브러시를 장착한다. 바뀐 상태는 다시 패널로 내려와서 브러시 버튼이 눌린 모양이 된다.

숫자 예시: 빨강(#FF0000)을 클릭하고 굵기 6을 클릭한 뒤 캔버스를 드래그하면, 굵기 6짜리 빨간 선이 그려진다. 이어서 연필 버튼을 누르면 굵기 버튼 4개가 전부 흐려지고, 색은 빨강을 유지한 채 1픽셀 빨간 선이 그려진다.

## 쉽게 말하면

지금까지는 성능 좋은 자동차를 만들어놓고 운전석에 핸들과 페달이 없는 상태였다. 이번 단계는 핸들(도구 선택), 기어(굵기), 그리고 색 고르는 다이얼을 운전석에 달아준 것이다. 엔진(그리기 로직)은 손대지 않았고, 조작부만 얹었다.

## 제대로 됐는지 어떻게 확인했나?

약속된 합격 기준 세 가지를 전부 실행했고 전부 통과했다.

- `npm run lint` (코드 스타일 검사기) — 통과, 경고 0.
- `npm run test` (자동 테스트 전체) — 12개 파일 76개 테스트 전부 통과. 기존 67개에 이번에 새로 쓴 9개가 더해졌고, 기존 것 중 깨진 건 없다.
- `npm run build` (배포용 빌드) — 성공.

새 테스트 9개가 확인하는 것: 도구 버튼 5개가 다 있는지, 각 버튼 클릭이 올바른 도구명을 전달하는지, 현재 도구만 눌린 상태로 표시되는지, 색 클릭이 올바른 색상값을 전달하는지, 팔레트가 정확히 28색인지, 굵기 클릭이 숫자를 전달하는지, 연필/선택일 때 굵기 버튼이 잠기는지, 브러시일 때 풀리는지.

테스트는 구현보다 먼저 작성했다(TDD — 실패하는 테스트를 먼저 쓰고 그걸 통과시키는 코드를 짜는 방식). 첫 실행에서 "컴포넌트가 없다"는 실패를 확인한 뒤 구현했다.

## 전체 그림에서 지금 어디?

```
[step 0] 그리기 엔진 (연필/브러시/에어브러시 + 획 박제)   ... 완료
[step 1] 지우개 엔진 (@erase2d) + 저장 호환 검증          ... 완료
[step 2] 도구 패널 UI + 화면 연결                          ... 완료  <-- 지금 여기
[step 3] 되돌리기/다시 실행 (Ctrl+Z / Ctrl+Y)              ... 다음
[step 4] XP 그림판 레퍼런스와 픽셀 실측 매칭               ... 수동 세션
```

다음이 step 3(undo/redo)인 이유: 이제 사용자가 실제로 그리고 지울 수 있게 됐으니, 실수를 되돌릴 수단이 바로 다음으로 필요하다. 그림판 감성의 마지막 필수 조각이기도 하다.

## 아직 안 된 것 (솔직하게)

- **Ctrl+Z / Ctrl+Y가 아직 없다.** 잘못 그으면 지우개로 지우는 것 말고는 방법이 없다. step 3 몫.
- **패널의 픽셀 단위 생김새는 아직 레퍼런스와 다를 수 있다.** 색상값·배치는 기존 XP 스타일 스펙과 그림판 28색을 썼지만, 실제 XP 그림판 스크린샷과의 정밀 비교(색·간격·눌림 효과 실측)는 step 4에서 한다. 계획된 순서다.
- **실제 브라우저에서 마우스로 그리는 손맛은 이 단계에서 자동 테스트로 못 본다.** 자동 테스트는 버튼과 상태 연결까지만 검증한다. 드래그해서 선이 그려지는 것 자체는 step 0·1 테스트와 step 4의 수동 검증이 담당한다.
- **에어브러시 굵기 체감**: 굵기 숫자가 에어브러시에선 분사 범위로 쓰여서, 같은 "6"이라도 브러시와 에어브러시의 두께 느낌이 다르다. 이상하게 느껴지면 step 4에서 조정 대상.
