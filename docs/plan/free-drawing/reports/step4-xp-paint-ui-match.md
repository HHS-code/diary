# Step 4: XP 그림판 UI 실측 매칭

## 이 단계를 왜 했나?

step 2에서 만든 그리기 도구 패널은 기능은 다 되지만, 스타일은 기존 다이어리 버튼 스펙을
재사용한 "1차 근사"였다. 이번 phase의 합의(coplan OQ-9)는 "마음에 들 때까지"가 아니라
"레퍼런스 스크린샷과 실측값이 일치할 때까지"를 종료 조건으로 삼는 것이라, 진짜 XP 그림판
스크린샷에서 픽셀 색을 직접 뽑아 비교하는 마무리 단계가 필요했다.

## 무엇을 만들었나?

reference/ms-paint-on-xp-v0-lilxt1d7tbf11.webp(440×410)를 브라우저 캔버스에 올려
도구 팔레트·색상 팔레트 영역의 픽셀 RGB를 실측했다
(실측: 이미지의 특정 좌표에서 실제 색 숫자값을 읽는 것). 그 결과와 현재 구현의 차이 3가지를
고쳐서 PaintToolbox가 그림판과 같은 규칙으로 그려지게 했다.

| 요소 | 레퍼런스 실측 | 고치기 전 | 고친 후 |
|---|---|---|---|
| 선택된 도구 버튼 | 흰색 배경 + 회색(#9c9e9c) 보더 | 베이지 + 어두운 그림자 | 실측값 적용 |
| 비선택 도구 버튼 | 패널 배경과 같은 플랫(보더 없음) | 그라디언트 + 갈색 보더 | 실측값 적용 |
| 색상 칸 보더 | 좌상 #aeaca9 / 우하 #e5e4e3 (파인 느낌) | 균일한 갈색 보더 | 실측값 적용 |

팔레트 28색과 패널 배경(#ece9d8)은 실측 결과 이미 레퍼런스와 일치해서 손대지 않았다.

## 데이터가 어떻게 흐르나?

```
레퍼런스 webp ──> 브라우저 canvas ──> 좌표별 RGB 추출 (예: 선택 버튼 = #fbfdfa~#ffffff)
                                          │
현재 화면(localhost) ──> getComputedStyle ──> 비교 ──> 차이만 코드 수정 ──> 재실측 확인
```

수정 후 재실측: 선택 버튼 rgb(255,255,255)·보더 rgb(156,158,156), 색상 칸 보더
rgb(174,172,169)/rgb(229,228,227) — 레퍼런스 실측값과 오차 ΔRGB ≤ 4 (webp 손실압축 노이즈 범위 안).

## 쉽게 말하면

옷을 사진 보고 눈대중으로 비슷하게 맞춘 게 아니라, 사진 속 옷의 색을 색채계로 찍어서
같은 물감 번호로 다시 칠한 것이다.

## 만든 파일

- 수정: src/components/PaintToolbox/PaintToolbox.jsx (색·보더 값만 — 레이아웃·아이콘·로직 무수정)

## 제대로 됐는지 어떻게 확인했나?

- npm run lint / npm run test(87개) / npm run build 모두 통과
- 스타일 재실측 ΔRGB ≤ 4
- 보너스로, 테스트 환경(jsdom)에서는 불가능해서 남겨뒀던 브라우저 실동작 검증도 여기서 수행:
  - 브러시로 긋기: 캔버스 픽셀 흰색 → 35,35,35 (획 그려짐)
  - 지우개로 문지르기: 해당 픽셀이 흰색으로 (획만 지워짐)
  - Ctrl+Z: 지운 획 복원 / Ctrl+Y: 다시 지워짐
  - 선택 도구로 획 클릭: 잡히지 않음(오브젝트 툴바 복사 버튼 disabled 유지) — "박제" 확인

## 전체 그림에서 지금 어디?

free-drawing phase의 마지막 step. step 0(그리기 로직) → 1(지우개) → 2(패널 UI) →
3(전체 undo/redo) → **4(실측 매칭, 완료)**. phase 전체 완료.

## 아직 안 된 것 (솔직하게)

- 도형 그리기(직선/사각형/원)는 합의대로 이번 범위 밖 — ROADMAP의 후속 후보(paint-shapes).
- step 3 리포트에 적힌 알려진 한계 그대로: 저장된 다이어리를 여는 시점의 복원 과정이
  히스토리에 낱개 단계로 쌓일 수 있음(동작 오류는 아니고, 첫 Ctrl+Z 몇 번이 "복원 단계"를
  되돌릴 수 있다는 뜻). 실사용에서 거슬리면 후속 논의.
- 레퍼런스가 손실압축 webp라 실측값에 ±4 정도의 노이즈가 있음 — 위 값들은 그 범위 안에서 확정.
