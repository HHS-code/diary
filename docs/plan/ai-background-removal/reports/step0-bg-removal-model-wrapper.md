# Step 0: bg-removal-model-wrapper

## 이 단계를 왜 했나?

이번 phase(ai-background-removal)의 목표는 스티커 스튜디오에 "AI 배경제거" 버튼을 넣는 것이다. 하지만 버튼을 누르면 실제로 배경을 지우는 일을 해줄 "엔진"이 먼저 있어야 한다. 이 step은 그 엔진 부분만 먼저 만들고, 화면에 보이는 버튼이나 UI는 다음 단계로 미뤘다.

미리 확인이 필요했던 이유가 하나 더 있다. 이전에 합의한 설계 문서(architecture.md)는 실제로 라이브러리(다른 사람이 만들어 배포한, 우리 프로젝트에 가져다 쓰는 코드 묶음)를 설치해보기 전에 웹 조사만으로 쓴 것이라 "이 문서의 가정은 미검증"이라고 스스로 밝혀 두었다. 그래서 이 step에서 실제로 설치하고 돌려봐서, 문서의 가정이 맞는지 틀렸는지를 확인하는 일도 같이 했다.

## 무엇을 만들었나?

`removeBackgroundFromImage`라는 함수 하나를 만들었다. 캔버스(그림이 그려진 화면 영역)나 이미지 파일을 넣으면, AI가 배경이라고 판단한 부분을 투명하게 지운 PNG 이미지를 돌려준다.

AI가 배경/사물을 구분하는 데 쓰는 "판단 기준"을 모델이라고 부르는데, 이 모델은 크기가 있는 파일(이번에 쓰는 가벼운 버전은 약 4.36MB)이라 매번 새로 내려받으면 느리다. 그래서 처음 한 번 부른 뒤로는 같은 브라우저 탭 안에서는 다시 내려받지 않고 재사용하도록 만들었다(합의 문서 ADR-4의 결정).

이번 step에서는 이 함수만 만들었고, 화면에 버튼을 붙이는 일은 하지 않았다. 그래서 지금 앱을 열어봐도 겉보기엔 아무것도 달라지지 않는다.

## 데이터가 어떻게 흐르나?

```
[사진/그림이 그려진 캔버스]
        │
        ▼
removeBackgroundFromImage(캔버스 등)
        │
        ├─ 모델을 아직 준비 안 했다면:
        │     └─ 모델 파일(u2netp.onnx, 4.36MB) 다운로드 + 준비   ← 최초 1회만
        │
        ├─ 이미 준비된 모델이 있다면: 바로 재사용             ← 2번째 호출부터
        │
        ▼
   AI가 픽셀 하나하나를 "배경"과 "사물"로 판단(추론)
        │
        ▼
[배경이 투명해진 PNG 이미지] 를 반환
```

실측 숫자(64×64픽셀짜리 단색 테스트 이미지 기준, 개발용 컴퓨터에서):
- 모델 파일 크기: 4,574,861바이트(약 4.36MB)
- 첫 호출: 약 4.5초(모델 준비 + 추론 포함)
- 두 번째 호출(같은 탭): 약 3.2초(모델은 재사용하지만, AI 판단 자체에 시간이 걸림 — 이 부분은 매번 필요한 작업이라 짧아지지 않는다)
- 실제 사용자가 스티커 스튜디오에서 쓸 이미지는 이보다 크므로 처리 시간은 더 걸릴 수 있다. "처리 중" 표시는 다음 step(bg-removal-button)에서 다룬다.

## 쉽게 말하면

사진관에 배경을 오려내주는 로봇을 하나 들여놓았다고 생각하면 된다. 이번 step은 그 로봇을 창고에 설치하고, "사진을 넣으면 오려진 사진이 나온다"는 것까지 직접 시험해서 확인한 단계다. 다만 로봇 앞에 손님이 누를 수 있는 버튼(스위치)은 아직 달지 않았다 — 그건 다음 step의 일이다.

## 만든 파일

- `src/ai/backgroundRemoval.js` — 위에서 설명한 `removeBackgroundFromImage` 함수.
- `src/ai/backgroundRemoval.test.js` — 이 함수가 모델을 한 번만 준비하는지, 진행 상황 콜백(진행률을 알려주는 함수)이 제대로 전달되는지 등을 확인하는 자동 테스트.
- `public/models/u2netp.onnx` — AI 모델 파일 자체(4.36MB). 아래 "확인 방법"에 이 파일이 왜 꼭 필요한지 적었다.
- `package.json`, `package-lock.json` — `@bunnio/rembg-web`(배경 제거 라이브러리, MIT 라이선스)이 새 의존성으로 추가됨.

## 제대로 됐는지 어떻게 확인했나?

AC(완료 기준)를 하나씩 실제로 실행해 확인했다.

1. `npm run lint && npm run build && npm run test` — 모두 통과(테스트 33개 파일, 190개 케이스 전부 통과).

2. `npm run dev`로 실제 개발 서버를 띄우고, 브라우저 자동화 도구(Playwright)로 페이지를 열어 콘솔(브라우저가 오류나 경고를 출력하는 창)을 직접 확인했다. 단순히 파일을 불러오는 것뿐 아니라, 실제로 이미지를 넣어 배경 제거까지 끝까지 실행시켜봤다. 결과: 에러 0건. 경고는 3건 있었는데 전부 "이 컴퓨터에서는 더 빠른 처리 방식(WebGL, 멀티스레드)을 못 써서 느린 방식으로 대체한다"는 성능 관련 안내일 뿐, 기능이 멈추거나 틀린 결과를 내는 문제는 아니었다.

   이 과정에서 문서에 없던 문제를 하나 발견해 고쳤다: 이 라이브러리는 모델 파일을 인터넷 어딘가에서 알아서 받아오는 게 아니라, **우리 앱이 자기 서버(`/models/u2netp.onnx` 경로)에 미리 파일을 갖다 놔야** 동작하는 방식이었다. 파일이 없으면 서버가 엉뚱하게 "찾을 수 없다"는 안내 화면을 대신 돌려주고, AI 라이브러리는 그걸 모델 파일인 줄 알고 읽으려다 실패했다("protobuf parsing failed"라는, 파일 형식이 안 맞다는 에러). 그래서 실제 AI 모델 파일을 공식 배포처(rembg 프로젝트의 GitHub Releases)에서 받아 `public/models/u2netp.onnx`에 넣었다. 이 폴더(`public`)에 있는 파일은 개발 중에도, 실제 배포판을 만들 때도 그대로 서빙되므로 앞으로 별도 조치 없이 계속 쓸 수 있다.

3. 프로덕션 빌드(`npm run build`로 만드는, 실제 배포에 쓰는 최종 결과물)에서도 확인했다. 다만 이번 step은 아직 어떤 화면 코드도 이 함수를 불러 쓰지 않기 때문에, 빌드 결과물에는 이 기능이 아예 포함되지 않는(안 쓰는 코드는 자동으로 빠지는) 상태였다. 그래서 검증을 위해 임시로 화면 진입점(`App.jsx`) 맨 위에 한 줄만 추가해 빌드하고, 브라우저에서 콘솔 에러가 없는지 확인한 뒤 그 한 줄을 다시 지웠다(최종적으로 `App.jsx`는 이 step 시작 전과 똑같다). 이 임시 확인에서도 에러는 0건이었다.

4. 새로 쓴 자동 테스트 3개: (1) 모델 세션을 만들어 이미지를 처리하고 결과를 그대로 돌려주는지, (2) 함수를 두 번 불러도 모델 준비 함수는 한 번만 불리는지(캐싱 확인), (3) 진행 상황 콜백을 넘기면 라이브러리 호출에 그대로 전달되는지 — 모두 통과.

## 전체 그림에서 지금 어디?

```
[step 0: 엔진 만들기] ← 지금 여기 (완료)
        │
        ▼
[step 1: "AI 배경제거" 버튼 연결]
        │  스티커 스튜디오 화면에 버튼을 달고, 누르면 이 step에서 만든
        │  엔진을 호출해 실제로 그림의 배경을 지운다. "처리 중..." 표시도 여기서.
        ▼
[step 2: AI 보정 도구(복원/삭제)]
        │  자동으로 지운 결과가 완벽하지 않을 때, 사람이 영역을 그려
        │  다시 보이게 하거나 더 지우는 도구.
```

이번 step이 먼저인 이유는 명확하다 — 버튼을 누르면 실제로 할 일(엔진)이 없으면 버튼을 만들 수 없기 때문이다.

## 아직 안 된 것 (솔직하게)

- 화면에는 아무 변화가 없다. 스티커 스튜디오를 열어봐도 새 버튼이나 도구가 보이지 않는다 — 전부 다음 step(bg-removal-button)의 일이다.
- 실제 사용자가 스티커 스튜디오에서 쓸 법한, 사람이나 사물이 있는 사진으로는 아직 테스트하지 않았다(이번엔 64×64픽셀 단색 테스트 이미지로만 확인). AI가 배경을 얼마나 정확히 구분하는지는 다음 step에서 실제 그림/사진으로 확인해야 한다.
- 처음 라이브러리를 불러올 때 필요한 부가 파일(AI 연산 엔진 자체, WASM이라는 형식의 파일)의 압축 크기가 약 6.4MB로 꽤 크다는 것을 확인했다. 모델 파일(4.36MB)과 합치면 처음 버튼을 누른 사용자는 10MB 넘게 받아야 할 수 있다는 뜻이다 — 합의 문서(ADR-4)에서 이미 "몇 초 정도의 대기는 감수한다"고 정했지만, 네트워크가 느리면 그보다 오래 걸릴 수 있다는 점은 다음 step에서 "처리 중" 표시를 만들 때 염두에 둘 부분이다.
- 이 라이브러리는 내려받은 모델 파일이 조작되지 않았는지 확인하는 자체 검사 기능이 있는데, 우리가 쓰는 모델(u2netp)에 대해서는 이 검사가 이름 표기 방식 차이(확장자 유무) 때문에 항상 건너뛰어지는 라이브러리 자체의 버그를 발견했다. 이번엔 공식 배포처에서 직접 받은 파일이라 문제가 없었지만, 나중에 모델 파일을 바꾸거나 다른 곳에서 받아올 일이 있다면 이 부분을 다시 살펴봐야 한다.
