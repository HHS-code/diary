# 빌드 리포트 — diary (ai-background-removal)

각 step이 무엇을 했는지 위에서 아래로 읽으면 프로젝트가 자라는 과정이 보인다.

- [완료] **Step 0: bg-removal-model-wrapper** — 공개 함수: removeBackgroundFromImage(source: HTMLCanvasElement | Blob, onProgress?) -> Promise<Blob> (src/ai/backgroundRemoval.js). 실제 API는 architecture.md 가정과 다름: newSession()은 비동기(Promise<BaseSession>)이고, remove()는 onlyMask 없이 호출하면 이미 배경이 투명해진 PNG Blob을 바로 반환한다(마스크 합성은 이 step에서 하지 않음 — 다음 step이 이 Blob을 그대로 FabricImage로 교체하는 경로를 검토해야 함). 세션은 모듈 스코프 Promise로 캐싱(ADR-4), newSession('u2netp', undefined, { onProgress })로 모델 이름 고정. 중대 이슈(실측): 라이브러리는 모델 파일을 자체 다운로드하지 않고 자기 앱의 /models/<name>.onnx를 직접 서빙해야 함 — public/models/u2netp.onnx(실제 rembg 공식 GitHub Releases에서 받은 4.36MB 파일, sha256 309c8469258dda742793dce0ebea8e6dd393174f89934733ecc8b14c76f4ddd8)를 추가해야 실제로 동작함(파일 없으면 Vite dev 서버가 index.html을 대신 반환해 'protobuf parsing failed' 에러 발생). 라이브러리 내장 무결성 해시(MODEL_HASHES)는 조회 키가 'u2netp.onnx'인데 실제 조회는 'u2netp'로 일어나 항상 스킵됨(라이브러리 자체 버그, 이번 실측에서는 유리하게 작용해 별도 조치 불필요). dev/build 양쪽에서 Playwright로 실제 추론까지 성공 확인(진행 상태 콜백 수신, PNG Blob 반환, 콘솔 에러 0건 — WebGL 미지원/WASM 단일스레드 폴백 경고만 존재, 무해함). vite.config.js는 수정 불필요. 다음 step이 읽어야 할 파일: src/ai/backgroundRemoval.js, public/models/u2netp.onnx, docs/plan/ai-background-removal/reports/step0-bg-removal-model-wrapper.md. ([리포트](step0-bg-removal-model-wrapper.md))
- [완료] **Step 1: bg-removal-button** — StickerStudio.jsx에 'AI 배경제거' 버튼 추가(commit 27fe98d). resolveLassoCutoutTarget으로 대상 이미지를 고르고, rasterizeTargetImageBounds(canvas.toCanvasElement + filter, commitLassoCutout과 동일 패턴)로 경계 영역만 rasterize해 originalBeforeAiRemovalRef(useRef)에 보관 — 다음 step(AI 보정의 복원)이 참조할 원본. backgroundRemovalStatus state('idle'|'processing'|'error')로 버튼 disabled 및 라벨('AI 배경제거'/'처리 중...(progress message)'/'실패')을 제어, onProgress 콜백은 info.message를 텍스트로 반영. 성공 시 결과 Blob을 FabricImage.fromURL로 만들어 원래 left/top/assetId를 승계해 캔버스 오브젝트 교체, 실패 시 catch에서 'error' 상태로 잠깐 표시 후 idle로 복귀(SAVED_LABEL_RESET_MS 재사용). assetStorage 저장은 하지 않음(ADR-4, '완성' 버튼만 저장). npm run lint/test 통과 확인(StickerStudio 테스트 12개 통과, 기존 회귀 없음). 참고: 이 step은 headless 세션이 구현을 커밋한 뒤 상태 갱신 전에 중단되어(토큰 소진으로 추정) 사람이 직접 코드/테스트를 검증하고 이 요약을 사후 작성함 — step2(ai-correction-tool)는 이 산출물(특히 originalBeforeAiRemovalRef의 존재)을 그대로 전제로 진행하면 됨. ([리포트](step1-bg-removal-button.md))
- [완료] **Step 2: ai-correction-tool** — 공개 함수: restoreRegion(currentCanvasElement, originalCanvasElement, regionPath, left, top) -> HTMLCanvasElement, eraseRegion(currentCanvasElement, regionPath, left, top) -> HTMLCanvasElement (src/fabric/stickerAiCorrection.js). architecture.md 가정과 달리 left/top 파라미터를 추가했다(불확실성 명시 문구가 허용한 조정) — 두 함수 모두 대상 이미지 경계 영역만 크롭한 캔버스를 받으며(step1의 originalBeforeAiRemovalRef와 동일 크롭 관례), regionPath는 스티커 캔버스 절대좌표로 그려지므로 크롭 원점(left,top)을 알아야 stickerCutout.js와 동일한 util.sendObjectToPlane(regionPath, undefined, util.createTranslateMatrix(left, top)) 패턴으로 정렬 가능하다(regionPath는 previewLassoCutout처럼 제자리 변경됨, 재사용 금지 — 적용 후 버리는 일회용). 마스크 렌더링은 FabricObject.render(ctx)를 캔버스 없이 직접 호출해 만든다(별도 StaticCanvas 불필요) — fill을 흰색으로 강제해 열린 곡선도 lasso와 동일하게 Canvas fill의 암묵적 닫힘 규칙에 맡긴다. usePaintTools.js에 'ai-correction' 도구 추가(PencilBrush, 주황 점선 #ff8000/[2,6] — 올가미 파랑과 구분), path:created 시 그림 획으로 박제하지 않되 캔버스 제거는 하지 않음(StickerStudio.jsx가 직접 구독해 canvas.remove 후 pendingCorrectionRegion state로 보관·적용 후 null 리셋). PaintToolbox.jsx에 'AI 보정' 도구 버튼 추가(MdAutoFixHigh). StickerStudio.jsx: originalBeforeAiRemovalRef 저장 형태를 canvasElement 단독에서 {canvasElement,left,top}으로 변경(step1 코드의 불가피한 최소 수정 — 이 정보 없이는 정렬 불가능, 다른 소비처 없어 안전). '복원' 버튼은 원본 없음 또는 보정 영역 미지정 시 비활성화, '삭제'는 보정 영역 미지정 시에만 비활성화. npm run lint/build/test 전체(34 files·209 tests) 통과 확인. 다음 step이 읽어야 할 파일: src/fabric/stickerAiCorrection.js, src/components/StickerStudio/StickerStudio.jsx(originalBeforeAiRemovalRef 저장 형태·pendingCorrectionRegion 상태), docs/plan/ai-background-removal/reports/step2-ai-correction-tool.md. ([리포트](step2-ai-correction-tool.md))
