# Step 0: 오브젝트 선택 상태 추적

## 이 단계를 왜 했나?

캔버스에 스티커나 이미지를 올려놓고 나면, "지금 뭐가 선택되어 있는지"를 화면(코드) 쪽에서도 알아야 다음 단계(복사·삭제·정렬 버튼)를 붙일 수 있다. 지금까지는 캔버스(그림판) 내부적으로는 어떤 오브젝트가 선택됐는지 알고 있었지만, 그 정보가 캔버스 바깥의 버튼이나 화면 요소로 전달되지 않았다. 이번 단계는 그 "선택 정보를 화면 쪽으로 흘려보내는 통로"만 먼저 만드는 작업이다. 버튼은 아직 없다.

## 무엇을 만들었나?

`useActiveSelection`이라는 이름의 훅(hook, 여러 컴포넌트에서 재사용 가능한 로직 조각)을 새로 만들었다. 이 훅은 캔버스에서 오브젝트를 하나 클릭하면 "지금 이게 선택됐어"라는 정보를, 두 개 이상을 드래그로 묶어 선택하면 "지금 여러 개(N개)가 선택됐어"라는 정보를 실시간으로 알려준다.

그리고 `DiaryCanvas`(캔버스가 그려지는 화면 컴포넌트)가 이 훅을 사용하도록 연결했다. 아직 정식 툴바(버튼 묶음)는 없으므로, 확인용으로 캔버스 오른쪽 근처에 다음과 같은 임시 글자만 나타나게 해두었다.

- 오브젝트 1개 선택 시: `선택됨: text` (text 자리에는 실제 오브젝트 종류가 들어간다. 예: 이미지면 `image`, 이모지 스티커면 `text`)
- 오브젝트 2개 이상 선택 시: `다중 선택: 3개` (숫자는 실제 선택한 개수)

이 임시 글자는 다음 단계에서 실제 버튼 툴바로 교체되어 사라질 예정이다.

## 데이터가 어떻게 흐르나?

```
[사용자가 캔버스에서 오브젝트 클릭/드래그 선택]
              │
              ▼
   Fabric.js(그림판 엔진)가 "선택이 생겼다/바뀌었다/없어졌다" 이벤트 발생
   (selection:created / selection:updated / selection:cleared)
              │
              ▼
   useActiveSelection 훅이 이 이벤트를 듣고 있다가
   "지금 선택된 게 1개인지 2개 이상인지" 판단
              │
              ▼
   1개면 activeObject 값 채움 / 2개 이상이면 activeSelection 값 채움
              │
              ▼
   DiaryCanvas 화면이 이 값을 받아서 임시 텍스트로 표시
```

숫자로 예를 들면: 캔버스에 하트 스티커 3개를 올려놓고 그중 1개만 클릭하면 `activeObject`에 그 하트 정보가 들어가고 화면엔 `선택됨: text`가 뜬다. 이어서 마우스로 3개를 한 번에 드래그해서 묶으면 `activeObject`는 비워지고 `activeSelection`에 3개 정보가 들어가 화면엔 `다중 선택: 3개`가 뜬다. 아무것도 선택하지 않은 빈 캔버스를 클릭하면 둘 다 비워진다.

## 쉽게 말하면

그림판(캔버스)에 CCTV를 하나 설치한 것과 같다. CCTV(useActiveSelection)는 "지금 누가 선택되어 있나"만 계속 지켜보다가, 상황이 바뀔 때마다(선택함/선택 바꿈/선택 풀림) 관제실(DiaryCanvas 화면)에 무전으로 알려준다. 관제실은 아직 그 정보로 아무 버튼도 만들지 않았고, 일단 "지금 이런 상황이야"라고 화면 구석에 메모만 적어두는 단계다. 버튼(복사, 삭제 등)을 붙이는 건 다음 단계다.

## 만든 파일

- `src/hooks/useActiveSelection.js` (신규) — CCTV 역할을 하는 훅. 화면 그림(JSX)은 전혀 그리지 않고, 오직 "지금 뭐가 선택됐는지" 정보만 계산해서 돌려준다.
- `src/components/DiaryCanvas/DiaryCanvas.jsx` (수정) — 위 훅을 불러다 쓰고, 확인용 임시 텍스트를 캔버스 옆에 띄우도록 했다. 또한 캔버스를 감싸는 테두리 상자에 "이 안에서 절대 위치로 뭔가를 띄울 수 있게" 기준점(`position: relative`)을 추가했다. 다음 단계의 버튼 툴바가 캔버스 위 정확한 위치에 뜨려면 이 기준점이 필요하다.

## 제대로 됐는지 어떻게 확인했나?

이번 단계는 실제 브라우저 클릭 없이(headless, 즉 화면 없는 자동 확인) 진행했기 때문에, 아래 두 가지 자동 검사와 코드를 직접 눈으로 검토하는 방식으로 확인했다.

1. `npm run lint` (코드 스타일/실수 검사): 통과.
2. `npm run build` (실제 배포 가능한 형태로 코드를 조립하는 과정, 문법 오류가 있으면 여기서 걸러진다): 통과.
3. 코드 리뷰: `useActiveSelection`이 3가지 이벤트(선택 생김/선택 바뀜/선택 풀림)를 모두 듣고 있는지, 화면이 사라질 때(언마운트) 그 "듣기"를 제대로 끄는지 직접 코드를 읽어 확인했다. 세 이벤트 모두 구독하고 있고, 정리(cleanup) 함수에서 전부 해제하는 것을 확인했다.

한 가지 신경 쓴 부분이 있다. 그림판(Fabric.js 캔버스)은 화면에 나타난 직후 바로 준비되는 게 아니라, 약간의 시간차를 두고 뒤늦게 준비된다(기존 코드의 구조상 그렇다). 그래서 `useActiveSelection`이 화면이 뜨자마자 캔버스를 찾으려고 하면 "아직 없음"으로 실패할 수 있다. 이를 대비해 50밀리초(0.05초)마다 "캔버스 준비됐나?"를 확인하다가, 준비되는 즉시 이벤트 듣기를 시작하도록 처리했다.

## 전체 그림에서 지금 어디?

```
[Step 0: 선택 감지] → Step 1: 툴바 버튼(복사/삭제/레이어) → Step 2: 정렬 툴바 → Step 3: 키보드 단축키
      (완료)              (다음)
```

이번 단계는 "누가 선택됐는지 아는 것"만 다뤘다. 다음 단계(Step 1)에서 이 선택 정보를 받아 실제 복사·삭제·레이어 순서 버튼을 가진 미니 툴바(`ObjectToolbar`)를 만들고, 지금 만든 임시 텍스트를 그 툴바로 교체한다. 순서를 이렇게 나눈 이유는, "선택 감지"와 "버튼 동작"을 한 번에 만들면 문제가 생겼을 때 어느 쪽이 원인인지 구분하기 어렵기 때문이다.

## 아직 안 된 것 (솔직하게)

- 실제 브라우저를 열어서 클릭/드래그로 확인하지는 않았다(자동 검사 환경이라 화면을 띄울 수 없음). 코드상으로는 Fabric.js의 표준 이벤트와 API를 그대로 따랐지만, 실제 클릭 테스트는 다음 단계에서 툴바가 붙은 뒤 사람이 직접 화면으로 확인하는 게 안전하다.
- 복사, 삭제, 정렬 등 실제 "동작"은 이번 단계에 전혀 없다. 지금은 "누가 선택됐는지 아는 것"까지만이다.
- 화면에 보이는 `선택됨: text` 같은 임시 문구는 다음 단계에서 사라질 디버깅용 표시다. 최종 사용자에게 보여줄 목적이 아니다.
