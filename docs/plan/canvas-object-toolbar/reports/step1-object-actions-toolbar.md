# Step 1: 오브젝트 조작 미니 툴바

## 이 단계를 왜 했나?

지난 step(0)에서는 "지금 캔버스에서 뭐가 선택돼 있는지"만 감지해서 화면에 임시로 "선택됨: 텍스트" 같은 글자로 보여줬다. 이번 step은 그 감지 결과를 실제로 쓸모 있게 만드는 단계다 — 오브젝트를 하나 클릭했을 때 그 근처에 "복사 / 삭제 / 맨 앞으로 / 맨 뒤로 / 한 단계 앞으로 / 한 단계 뒤로" 버튼 6개가 뜨는 작은 툴바(플로팅 미니 툴바)를 만들었다.

## 무엇을 만들었나?

두 가지를 만들었다.

1. **`useObjectActions`** — 버튼을 눌렀을 때 실제로 일어나는 일을 처리하는 부분. 화면에 아무것도 그리지 않고, "복사해라", "삭제해라" 같은 명령을 캔버스(그림판)에 전달하는 역할만 한다.
2. **`ObjectToolbar`** — 화면에 버튼 6개를 그리는 부분. 버튼이 눌리면 위 `useObjectActions`가 만든 함수를 그대로 호출할 뿐, 캔버스를 직접 건드리지 않는다.

이렇게 "명령을 내리는 부분"과 "버튼을 그리는 부분"을 파일 단위로 완전히 나눴다. 나중에 버튼 색깔이나 위치만 바꾸고 싶을 때 `ObjectToolbar` 파일만 고치면 되고, 복사/삭제 로직이 실수로 망가질 위험이 없다.

## 데이터가 어떻게 흐르나?

```
[사용자가 스티커 클릭해서 선택]
         |
         v
useActiveSelection (step 0)
  선택된 오브젝트를 감지
         |
         v
DiaryCanvas.jsx
  activeObject가 있으면 <ObjectToolbar> 렌더링
         |
         v
ObjectToolbar.jsx (버튼 6개 그림)
  "복사" 버튼 클릭
         |
         v
useObjectActions.copy(target)
  1) target을 복제한다 (target.clone())
  2) 복제본 위치를 오른쪽 아래로 10px, 10px 옮긴다
  3) 캔버스에 추가한다
  4) 새로 생긴 복제본을 선택 상태로 만든다
         |
         v
[화면에 원본 옆에 살짝 겹친 복사본이 나타남]
```

예를 들어 하트 스티커(❤️)가 캔버스 좌표 `(100, 100)`에 있을 때 "복사" 버튼을 누르면, 복제본은 `(110, 110)`에 생긴다. 즉 원본과 정확히 겹치지 않고 오른쪽 아래로 살짝 밀려서 보이기 때문에, 두 개가 생겼다는 걸 바로 알 수 있다.

툴바의 위치도 자동으로 계산한다. 오브젝트의 화면 좌표(`top`)가 40px보다 작으면(캔버스 맨 위쪽에 가까우면) 툴바가 화면 밖으로 잘릴 수 있으므로, 이 경우엔 오브젝트 위가 아니라 아래에 툴바를 띄운다.

## 쉽게 말하면

워드프로세서나 파워포인트에서 도형 하나를 클릭했을 때 옆에 작은 서식 도구모음이 뜨는 것과 같은 동작이다. 이번 step은 그 "작은 도구모음"과 "그 안의 버튼들이 실제로 하는 일"을 만든 것이다.

## 만든 파일

- `src/hooks/useObjectActions.js` — 복사/삭제/레이어 순서 변경 명령 6개
- `src/components/ObjectToolbar/ObjectToolbar.jsx` — 버튼 6개짜리 미니 툴바 화면
- `src/components/DiaryCanvas/DiaryCanvas.jsx` (수정) — step 0의 임시 표시 문구를 지우고 `ObjectToolbar`를 실제로 띄우도록 연결

## 제대로 됐는지 어떻게 확인했나?

- `npm run lint` (코드 스타일/실수 자동 검사): 통과, 오류 없음
- `npm run build` (실제 배포 가능한 형태로 코드를 묶는 작업): 통과, 정상적으로 결과물 생성됨
- `ObjectToolbar.jsx` 파일에 "그림판을 직접 조작하는 코드(fabric 패키지)를 가져다 쓰지 않는다"는 규칙을 지켰는지 직접 검색(grep)으로 확인 — 실제 `import` 문은 없고, 타입 설명을 위한 주석(JSDoc)에서만 `fabric`이라는 단어가 나온다. 규칙 준수 확인됨.

자동 테스트는 이번 step에서 새로 만들지 않았다. 이유: 이 프로젝트에는 아직 자동 테스트를 돌리는 도구(테스트 러너)가 설치돼 있지 않고, 이번 step에서 지시된 검증 기준(AC)도 lint·build·자가 점검 세 가지만 명시돼 있었다. 새 테스트 도구를 설치하는 것은 이번 step 범위를 벗어나는 일이라 판단해 하지 않았다.

## 전체 그림에서 지금 어디?

```
[완료] Step 0: 선택 상태 감지        (뭘 선택했는지 안다)
[완료] Step 1: 조작 미니 툴바        (선택한 걸 복사/삭제/순서 변경할 수 있다)  <- 지금 여기
[예정] Step 2: 정렬 툴바            (여러 개 선택했을 때 줄 맞추기)
[예정] Step 3: 키보드 단축키        (Delete, Ctrl+C/V)
```

여러 개를 한꺼번에 선택했을 때 나타나는 "정렬 툴바"는 하나만 선택했을 때 쓰는 이번 툴바와는 버튼 구성도, 계산 방식(여러 오브젝트의 위치를 비교해야 함)도 달라서 다음 step으로 분리했다.

## 아직 안 된 것 (솔직하게)

- 여러 개를 동시에 선택했을 때(다중 선택) 뜨는 정렬 툴바는 아직 없다 — 다음 step에서 만든다.
- `Delete` 키로 삭제하거나 `Ctrl+C`/`Ctrl+V`로 복사·붙여넣기 하는 키보드 단축키는 아직 없다 — 그 다음 step에서 만든다.
- 이번 step에서 만든 기능은 자동 테스트로 검증되지 않았다. 실제로 브라우저에서 클릭해서 눈으로 확인하는 절차가 아직 남아 있다면 별도로 확인이 필요하다.
