# Step 3: 키보드 단축키

## 이 단계를 왜 했나?

이전 단계까지는 캔버스 위 오브젝트(스티커·이미지·텍스트)를 다루려면 화면에 뜬 미니 툴바 버튼을 마우스로 눌러야 했다. 파워포인트나 그림판을 쓰던 사람이라면 손이 먼저 `Delete` 키나 `Ctrl+C`/`Ctrl+V`로 가기 마련이다. 이 단계는 마우스 없이도 삭제·복사·붙여넣기가 되도록 키보드 단축키를 연결하는 작업이다.

## 무엇을 만들었나?

`useCanvasKeyboardShortcuts`라는 새 훅(hook, 여러 화면 컴포넌트가 공유해서 쓰는 재사용 로직 단위) 하나를 만들었다. 이 훅은 화면에 아무것도 그리지 않고, 키보드 입력을 감시하다가 세 가지 키 조합에 반응한다.

- `Delete`: 지금 선택된 오브젝트를 지운다. 여러 개를 한꺼번에 선택한 상태(다중 선택)라면 선택된 것 전부를 지운다.
- `Ctrl+C` (맥에서는 `Cmd+C`): 지금 선택된 오브젝트를 "기억"해 둔다. 화면에는 아무 변화가 없다.
- `Ctrl+V` (맥에서는 `Cmd+V`): 방금 기억해 둔 오브젝트를 원본에서 오른쪽 아래로 10픽셀(화면 위 아주 작은 거리 단위) 떨어진 자리에 새로 하나 더 만들어 놓는다.

그리고 한 가지 예외 규칙이 있다. 텍스트 메모를 더블클릭해서 글자를 편집하는 중이라면 이 세 단축키는 전부 무시된다. 편집 중에 `Delete`를 눌러 글자를 지우거나 `Ctrl+C`로 문장을 복사하는 건 원래 브라우저의 기본 동작이어야지, 캔버스가 가로채서 오브젝트를 통째로 지워버리면 안 되기 때문이다.

## 데이터가 어떻게 흐르나?

```
키보드 입력
    │
    ▼
[텍스트 편집 중인가?] --예--> 아무 것도 안 함 (브라우저 기본 동작에 맡김)
    │ 아니오
    ▼
    ┌───────────┬───────────────┬───────────────┐
    │  Delete    │   Ctrl+C      │   Ctrl+V      │
    ▼           ▼               ▼
 선택된 오브젝트  선택된 오브젝트를    "기억해둔 오브젝트"를
 를 캔버스에서    복제해서            다시 복제해서
 제거            "기억" 공간에 보관    +10,+10 위치에 추가
```

예를 들어 스티커 하나를 캔버스의 (100, 100) 위치에 놓고 클릭해 선택한 뒤 `Ctrl+C`를 누르면, 그 스티커의 복제본이 훅 내부의 "기억 공간"에 저장된다. 이어서 `Ctrl+V`를 세 번 누르면 (110, 110), 다시 (110, 110), 또 (110, 110) — 매번 똑같은 위치에 새 스티커가 하나씩 추가된다. 원본 위치를 기준으로 항상 +10, +10만큼 떨어진 자리에 놓기 때문에, 붙여넣기를 반복해도 스티커들이 계단처럼 점점 멀어지지 않는다(이미 놓인 복제본을 기준으로 또 복제하는 게 아니라, 처음 복사했던 원본 기억을 매번 다시 사용하기 때문).

## 쉽게 말하면

이번 단계는 리모컨에 단축 버튼을 추가한 것과 비슷하다. 이전 단계에서 이미 "복사" 버튼, "삭제" 버튼은 화면에 있었다(리모컨 자체는 이미 있었다). 이번에는 그 버튼과 똑같은 동작을 하는 손가락 단축 조합(`Delete`, `Ctrl+C`, `Ctrl+V`)을 추가로 연결한 것뿐이다. 화면 버튼도 그대로 잘 작동하고, 이제 키보드로도 같은 일을 할 수 있게 됐다.

## 만든 파일

- `src/hooks/useCanvasKeyboardShortcuts.js` — 신규. 키보드 단축키 로직 전체.
- `src/components/DiaryCanvas/DiaryCanvas.jsx` — 수정. 이 훅을 호출하는 한 줄 추가.

## 제대로 됐는지 어떻게 확인했나?

- `npm run lint` (코드 스타일과 흔한 실수를 검사하는 도구) 통과.
- `npm run build` (실제 배포용으로 코드를 묶어보는 과정 — 여기서 문법 오류나 잘못된 참조가 있으면 실패한다) 통과.
- 코드 리뷰 자가 점검 세 가지, 전부 확인됨:
  1. 텍스트 편집 중 가드(`isEditing` 체크)가 `Delete`/`Ctrl+C`/`Ctrl+V` 세 동작 모두에 적용되는가 — 세 동작을 나누기 전, 함수 맨 앞에서 한 번만 검사하도록 만들어서 셋 다 자동으로 보호된다.
  2. 키보드 감시가 화면을 벗어날 때(다른 페이지로 이동 등) 제대로 해제되는가 — 등록한 감시를 정리하는 코드를 넣어 확인했다.
  3. 붙여넣기를 여러 번 눌러도 위치가 이상하게 튀지 않는가 — 항상 "처음 복사했던 원본"을 기준으로 +10, +10만 이동하도록 만들어 확인했다.

이 프로젝트에는 자동 테스트 도구(코드가 스스로 동작을 검증하는 장치)가 아직 설치돼 있지 않아서, 이번에도 이전 단계들과 마찬가지로 자동 테스트 파일은 만들지 않았다. 대신 위 세 가지를 코드를 직접 읽으며 눈으로 확인했다.

## 전체 그림에서 지금 어디?

```
[Step 0] 선택 상태 추적       완료
[Step 1] 복사/삭제/레이어 버튼  완료

```

PRD에서 정한 "성공 기준" 다섯 가지 중 마지막 항목("Delete 키로 삭제되고, Ctrl+C/Ctrl+V로 복사·붙여넣기가 된다")까지 이번 단계로 채워졌다. 계획 문서상으로는 이 phase의 마지막 step이다.

## 아직 안 된 것 (솔직하게)

- 이 프로젝트 전체에 자동 테스트 도구(vitest, jest 같은 것)가 설치돼 있지 않다. 그래서 "복사가 실제로 되는지"는 브라우저에서 직접 눌러보기 전까지는 코드 리뷰로만 확인한 상태다. 브라우저로 직접 눌러보는 확인은 이번 단계에서 하지 않았다.
- PRD에 있던 "시스템 클립보드"(다른 프로그램과 복사/붙여넣기를 공유하는 기능)는 이번 범위가 아니라서 구현하지 않았다 — 이 캔버스 안에서만 복사/붙여넣기가 통한다.
- 여러 개를 선택한 상태에서 `Ctrl+C`/`Ctrl+V`를 하면 지금은 "선택 전체"가 아니라 Fabric.js가 반환하는 하나의 선택 묶음(ActiveSelection)을 통째로 복제한다. 여러 개를 동시에 복사·붙여넣기하는 시나리오는 PRD에 별도로 명시되어 있지 않아 별도 검증은 하지 않았다.
