# Step 2: 정렬/등간격 배치 툴바

## 이 단계를 왜 했나?

지난 step(1)에서는 오브젝트 하나를 클릭했을 때 복사/삭제/레이어 순서를 조작하는 미니 툴바를 만들었다. 하지만 스티커를 여러 개 골라서 나란히 줄 맞추는 일(정렬)은 아직 할 수 없었다. 파워포인트에서 도형 여러 개를 선택하고 "왼쪽 맞춤", "가로 간격 동일하게"를 누르는 것과 같은 동작을 이번 step에서 추가했다.

## 무엇을 만들었나?

역시 두 가지로 나눠서 만들었다.

1. **`useAlignment`** — "왼쪽 맞춤을 실행해라", "가로 간격을 동일하게 만들어라" 같은 실제 계산과 위치 이동을 처리하는 부분. 화면을 그리지 않는다.
2. **`AlignmentToolbar`** — 버튼 8개(좌/우/상/하/가운데 정렬 2종 + 가로/세로 등간격)를 그리는 부분. 버튼이 눌리면 `useAlignment`가 만든 함수를 그대로 호출할 뿐 계산은 직접 하지 않는다.

이번에도 step 1과 같은 원칙으로 "계산하는 부분"과 "그리는 부분"을 파일로 완전히 나눴다. 버튼 배치나 색깔만 바꾸고 싶으면 `AlignmentToolbar.jsx`만 고치면 되고, 정렬 계산이 망가질 걱정이 없다.

## 데이터가 어떻게 흐르나?

```
[사용자가 스티커 3개를 Shift+클릭 또는 드래그로 다중 선택]
         |
         v
useActiveSelection (step 0)
  2개 이상 선택되면 activeSelection을 채움
         |
         v
DiaryCanvas.jsx
  activeSelection이 있으면 <AlignmentToolbar> 렌더링
         |
         v
AlignmentToolbar.jsx (버튼 8개 그림)
  "좌 정렬" 버튼 클릭
         |
         v
useAlignment.alignLeft(selection)
  1) 선택된 오브젝트마다 화면 좌표(getBoundingRect)를 구함
  2) 그중 가장 왼쪽 값을 찾음
  3) 나머지 오브젝트를 전부 그 값에 맞춰 옮김
  4) 화면을 다시 그림(renderAll)
         |
         v
[선택된 오브젝트들이 왼쪽 기준으로 한 줄에 맞춰짐]
```

예를 들어 스티커 3개가 각각 캔버스 좌표 `left = 50`, `120`, `80`에 있을 때 "좌 정렬"을 누르면, 가장 작은 값인 `50`을 기준으로 세 개 모두 `left = 50`으로 옮겨진다.

"가로 등간격" 버튼은 조금 다르게 동작한다. 세 개의 스티커가 가로로 `중심 좌표 30, 200, 90`에 흩어져 있다면, 먼저 중심 좌표 기준으로 순서를 정렬해 `30(고정), 90, 200(고정)`으로 본 다음, 양 끝(`30`과 `200`)은 그대로 두고 가운데 스티커만 정확히 중간 지점(`115`)으로 옮긴다. 스티커가 2개 이하면 "가운데에 둘 오브젝트"가 없으므로 아무 일도 하지 않는다 — 에러가 나지 않고 그냥 무시된다.

## 쉽게 말하면

파워포인트나 피그마에서 도형 여러 개를 선택했을 때 나타나는 "정렬" 도구 모음과 똑같은 동작이다. 줄자로 여러 물건의 왼쪽 끝을 재서 한 줄로 맞추거나, 물건 사이 간격을 자로 재서 똑같이 벌려놓는 것을 컴퓨터가 대신 계산해주는 것이다.

## 만든 파일

- `src/hooks/useAlignment.js` — 정렬 6개(좌/우/상/하/가운데 수평/가운데 수직) + 등간격 배치 2개(가로/세로) 명령
- `src/components/ObjectToolbar/AlignmentToolbar.jsx` — 버튼 8개짜리 정렬 툴바 화면
- `src/components/DiaryCanvas/DiaryCanvas.jsx` (수정) — 다중 선택 시 `AlignmentToolbar`를 실제로 띄우도록 연결

## 제대로 됐는지 어떻게 확인했나?

- `npm run lint` (코드 스타일/실수 자동 검사): 통과, 오류 없음
- `npm run build` (실제 배포 가능한 형태로 코드를 묶는 작업): 통과, 정상적으로 결과물 생성됨
- `AlignmentToolbar.jsx`가 "그림판을 직접 조작하는 코드(fabric 패키지)를 가져다 쓰지 않는다"는 규칙을 지켰는지 직접 검색(grep)으로 확인 — 실제 `import` 문은 없고, 타입 설명을 위한 주석(JSDoc)에서만 `fabric`이라는 단어가 나온다. 규칙 준수 확인됨.
- "가로 등간격"/"세로 등간격"이 오브젝트가 2개 이하일 때 안전하게 아무 동작도 안 하는지 코드로 확인 — `objects.length < 3`이면 계산을 시작하기 전에 함수를 즉시 끝내도록(`return`) 만들어져 있어, 배열 접근으로 인한 오류가 날 수 없는 구조다.

자동 테스트는 이번 step에서도 새로 만들지 않았다. 이유는 step 1과 동일 — 이 프로젝트에는 아직 자동 테스트 도구(테스트 러너)가 설치돼 있지 않고, 이번 step의 검증 기준(AC)도 lint·build·자가 점검 세 가지만 명시돼 있었다.

## 전체 그림에서 지금 어디?

```
[완료] Step 0: 선택 상태 감지        (뭘 선택했는지 안다)
[완료] Step 1: 조작 미니 툴바        (하나를 골랐을 때 복사/삭제/순서 변경)
[완료] Step 2: 정렬 툴바            (여러 개를 골랐을 때 줄 맞추기/등간격)  <- 지금 여기
[예정] Step 3: 키보드 단축키        (Delete, Ctrl+C/V)
```

정렬 툴바는 하나만 선택했을 때 쓰는 조작 툴바(step 1)와 필요한 계산이 완전히 달랐다(여러 오브젝트의 위치를 서로 비교해야 함). 그래서 step을 나눴고, 이번 step으로 "여러 개 선택했을 때 할 수 있는 일"이 마무리됐다. 마지막으로 남은 건 마우스 없이 키보드만으로 삭제/복사/붙여넣기를 하는 것이다.

## 아직 안 된 것 (솔직하게)

- `Delete` 키로 삭제하거나 `Ctrl+C`/`Ctrl+V`로 복사·붙여넣기 하는 키보드 단축키는 아직 없다 — 다음 step에서 만든다.
- 이번 step에서 만든 기능은 자동 테스트로 검증되지 않았다. 실제로 브라우저에서 스티커 여러 개를 선택해 정렬 버튼을 눌러보는 절차가 아직 남아 있다면 별도로 확인이 필요하다.
- 회전된 오브젝트를 정렬할 때는 `getBoundingRect()`가 회전 이후의 바깥 테두리 기준 좌표를 반환하므로, 정렬 결과가 "회전되지 않은 상태의 좌표"를 기대한 감각과 다르게 보일 수 있다. 이는 PRD에 명시된 요구사항은 아니라 별도 처리하지 않았다.
