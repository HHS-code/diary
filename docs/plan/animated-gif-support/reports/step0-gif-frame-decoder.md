# Step 0: GIF 프레임 디코더

## 이 단계를 왜 했나?

지금 다이어리 캔버스에 GIF를 올리면 첫 장면에서 멈춘 사진처럼 보인다. 브라우저가 GIF를 화면에 그릴 때 쓰는 방식(캔버스의 `drawImage`)이 원래 "이미지 한 장을 캡처해서 붙여넣기"만 하는 동작이라, 움직이는 GIF도 그 순간의 한 프레임만 붙잡아 버리기 때문이다.

애니메이션을 실제로 재생하려면 GIF 파일 안에 들어 있는 "프레임들"(한 장 한 장의 그림)을 직접 꺼내서, 정해진 시간마다 순서대로 바꿔 그려줘야 한다. 이번 단계는 그 첫 삽질 — GIF 파일 바이너리를 받아서 "프레임 그림들의 목록"과 "각 프레임을 몇 밀리초(1000분의 1초) 동안 보여줄지"를 뽑아내는 부분만 떼어서 만들었다. 캔버스에 실제로 올리고 재생하는 건 다음 단계 몫이다.

## 무엇을 만들었나?

`decodeGifFrames`라는 함수 하나를 만들었다. GIF 파일(정확히는 "Blob" — 브라우저가 다루는 파일 데이터 덩어리)을 넣으면, 두 가지를 돌려준다.

- `frames`: 이 GIF의 각 장면을 그려 넣은 "캔버스"(HTML에서 그림을 그릴 수 있는 도화지 같은 요소) 목록
- `delays`: 각 장면을 몇 밀리초 동안 보여줘야 하는지 적힌 숫자 목록

GIF 파일 자체를 해독하는 복잡한 작업(LZW라는 압축 방식을 푸는 것 등)은 `gifuct-js`라는 외부 라이브러리에 맡겼다. 우리가 새로 만든 부분은 이 라이브러리가 뱉어주는 "반쪽짜리 정보"를 완성된 그림으로 조립하는 로직이다.

왜 반쪽짜리냐면, GIF는 용량을 아끼려고 매 장면 전체를 다시 그리지 않는다. 예를 들어 배경은 그대로 두고 캐릭터 눈만 깜빡이는 GIF라면, 두 번째 장면에는 "눈 부분만" 들어있지 화면 전체 그림이 들어있지 않다. 이 부분을 "패치(patch)"라고 부른다. 그래서 이전 장면 위에 이번 패치를 겹쳐 그려야 완성된 그림이 나온다 — 이 겹쳐 그리기(합성) 로직이 이번 단계의 핵심이다.

또 GIF에는 "이번 장면을 보여준 다음, 다음 장면을 그리기 전에 이 부분을 어떻게 처리할지"를 정하는 규칙(disposal method)이 있다. 예를 들어 "이번 장면이 그렸던 자리를 지우고(투명하게 만들고) 다음 장면을 그려라"는 규칙이 있으면, 그 처리도 해줘야 한다. 이걸 빼먹으면 지워져야 할 자리에 이전 그림 잔상이 계속 남아 애니메이션이 깨져 보인다.

## 데이터가 어떻게 흐르나?

```
GIF 파일(Blob)
   │
   ▼
gifuct-js: 바이너리 해독
   │  프레임마다: { 패치(부분 그림), 위치(left, top, width, height),
   │               표시시간(delay), 처리규칙(disposalType) }
   ▼
decodeGifFrames (이번 단계에서 만든 조립 로직)
   │
   │  도화지 하나(compositeCanvas)를 준비해두고,
   │  프레임을 순서대로 처리하면서:
   │
   │  1) 직전 프레임의 처리규칙을 먼저 적용
   │     (지우기 / 이전 상태로 되돌리기 / 그대로 두기)
   │  2) 이번 프레임의 패치를 도화지 위에 겹쳐 그리기
   │  3) 지금 도화지 상태를 통째로 복사해서 "완성된 프레임"으로 저장
   │
   ▼
{ frames: [완성된 그림1, 완성된 그림2, ...],
  delays: [100, 100, 100, ...] }  (밀리초)
```

구체적인 숫자로 보면 이렇다. 테스트에 쓴 가상의 2x2픽셀짜리 3장면 GIF를 예로 들면:

- 1번째 장면: 4칸 전체를 빨강/초록/파랑/흰색으로 채움. "그대로 두기" 규칙.
- 2번째 장면: 오른쪽 아래 1칸만 빨강으로 바꾸는 패치. "다음 장면 그리기 전에 이 칸을 지워라" 규칙.
- 3번째 장면: 왼쪽 위 1칸만 초록으로 바꾸는 패치.

이걸 그대로 겹쳐 그리기만 하면(처리규칙을 무시하면) 3번째 장면에 2번째 장면이 칠했던 빨강이 그대로 남아있어야 하는데, 실제로는 2번째 장면의 "지워라" 규칙 때문에 3번째 장면에서 그 칸은 투명하게 나와야 한다. 이번 단계에서 만든 로직으로 실제 이렇게 나오는지 픽셀 값(RGBA 4개 숫자)까지 확인했다.

## 쉽게 말하면

플립북(팔락팔락 넘기면 그림이 움직이는 책)을 상상하면 된다. GIF 파일은 그 플립북의 페이지들을 압축해서 담아둔 상자다. 그런데 종이를 아끼려고 어떤 페이지는 "이전 페이지에서 바뀐 부분만" 스티커로 붙여둔 식으로 만들어져 있다. `decodeGifFrames`는 이 상자를 열어서, 스티커를 알맞은 위치에 다 붙이고 "완성된 낱장 그림들"로 다시 만들어주는 작업이다. 그래야 다음 단계에서 이 낱장들을 그냥 순서대로 넘기기만 하면 애니메이션이 된다.

## 만든 파일

- `src/fabric/gifFrameDecoder.js` — `decodeGifFrames` 함수 (신규)
- `src/fabric/gifFrameDecoder.test.js` — 테스트 (신규)
- `package.json` / `package-lock.json` — `gifuct-js` 의존성 추가

## 제대로 됐는지 어떻게 확인했나?

- 2칸 이상 장면이 있는 가상 GIF를 디코딩하면 `frames`와 `delays` 개수가 같고, 각 장면이 실제 캔버스 요소인지 — 통과.
- 위에서 설명한 "지우기 규칙" 시나리오대로 3장면을 만들어 각 칸의 색이 정확히 예상한 값으로 나오는지(픽셀 단위 검증) — 통과. 특히 2번째 장면이 그린 칸이 3번째 장면에서 투명하게(RGBA 값 0,0,0,0) 나오는지까지 확인했다.
- 1장면짜리(정지) GIF를 넣으면 `frames.length`가 1인지 — 통과.
- GIF가 아닌 파일(PNG 파일 시작 부분 바이트)을 넣으면 에러가 나는지 — 통과.
- 프로젝트 전체 검사 명령(`npm run lint`, `npm run build`, `npm run test`) 모두 통과. 기존 테스트 121개 + 이번에 추가한 4개 = 125개 전부 통과, 기존 테스트는 하나도 깨지지 않았다.

테스트용 GIF 파일은 저장소에 실제 파일로 추가하지 않고, 코드 안에 문자로(base64) 직접 박아 넣었다 — 합의된 계획대로다.

## 전체 그림에서 지금 어디?

```
[0] GIF 프레임 디코더        ← 지금 여기, 완료
[1] 애니메이션 GIF 오브젝트    (다음 단계)
[2] 공유 렌더 루프
[3] 등록 경로 통합 (AssetImportPanel 등)
[4] 저장/재로드
```

이번 단계는 "GIF 파일 → 완성된 그림 여러 장 + 각 그림 표시 시간"을 뽑아내는 부품 하나만 만든 것이다. 이 부품은 아직 캔버스에 올라가지도, 화면에서 재생되지도 않는다. 다음 단계(애니메이션 GIF 오브젝트)에서 이 부품이 뽑아낸 그림들을 실제 캔버스 오브젝트(회전하고 크기 조절할 수 있는 물건)에 넣게 된다. 이 순서로 진행하는 이유는, "그림을 뽑아내는 일"과 "캔버스에 물건으로 다루는 일"을 분리해두면 각각 따로 테스트하고 검증하기 쉬워지기 때문이다.

## 아직 안 된 것 (솔직하게)

- 캔버스에 GIF를 올려도 여전히 안 움직인다. 이번 단계는 "재료 준비"만 했다.
- 실제 사람이 만든 진짜 GIF 파일(예: 짤방, 이모티콘)로는 아직 테스트하지 않았다 — 손수 만든 아주 작은(2x2픽셀) 가상 GIF로만 검증했다. 다만 GIF 표준 규격에 맞춰 만들었고, 다음 단계에서 실제 캔버스에 올려보며 실제 GIF 파일로도 확인할 예정이다.
- 화면 크기가 매우 크거나 프레임이 아주 많은 GIF(예: 수백 프레임짜리)를 디코딩할 때 시간이 얼마나 걸리는지는 아직 재보지 않았다. PRD에도 이 최적화는 이번 phase 범위 밖으로 명시돼 있다.
