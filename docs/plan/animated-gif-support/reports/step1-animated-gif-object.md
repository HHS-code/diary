# Step 1: animated-gif-object

## 이 단계를 왜 했나?

지난 단계(step 0)에서 GIF 파일을 뜯어서 "프레임 이미지 여러 장 + 각 프레임을 얼마나 보여줄지(시간)" 목록으로 바꾸는 작업까지는 끝냈다. 하지만 그 프레임 목록을 캔버스(그림판) 위에 올려놓고, 다른 이미지처럼 회전하고 크기를 바꾸고 옮기고 복사하고 지울 수 있는 "물건"으로 다루려면, 캔버스가 이해하는 형태(오브젝트)로 포장하는 작업이 하나 더 필요하다. 이번 단계가 그 포장 작업이다.

## 무엇을 만들었나?

`AnimatedGif`라는 새 캔버스 오브젝트 종류를 만들었다. 캔버스 라이브러리(Fabric.js)가 이미 갖고 있는 "정지 이미지" 오브젝트(`FabricImage`)를 그대로 물려받아 만들었기 때문에, 회전·크기조절·이동·복사·삭제 같은 기존 기능은 손댈 필요 없이 자동으로 따라온다. 여기에 딱 두 가지만 얹었다.

1. **`advanceFrame()`** — "다음 프레임으로 넘겨라"는 명령. 호출할 때마다 지금 보여주는 프레임이 다음 프레임으로 바뀐다. 마지막 프레임 다음엔 다시 첫 프레임으로 돌아간다(순환). 이 함수 자체는 "언제" 호출할지는 정하지 않는다 — 그건 다음 단계(공유 렌더 루프)의 몫이다. 이번 단계는 "누르면 한 칸 넘어간다"는 스위치만 만든 것.
2. **저장 방식** — 캔버스를 저장했다가 다시 불러올 때, GIF의 프레임 이미지 자체(용량이 큰 데이터)는 저장하지 않고, "이 GIF 원본이 어디 있는지"를 가리키는 짧은 참조값(`assetId`)만 저장한다. 다시 불러올 때 이 참조값으로 원본 GIF를 다시 찾아서 프레임을 새로 만든다.

원본을 찾다가 실패하는 경우(예: 원본 파일이 저장소에서 지워진 경우)에도 캔버스 전체가 깨지지 않도록, 실패 시에는 애니메이션 없이 마지막으로 보이던 모습을 그대로 보여주는 일반 정지 이미지로 조용히 대체하는 안전장치도 넣었다.

## 데이터가 어떻게 흐르나?

```
[캔버스에 GIF 오브젝트가 있음]
        |
        |  advanceFrame() 호출 (다음 단계가 주기적으로 부를 예정)
        v
  현재 프레임 index를 +1 (마지막이면 0으로)
        |
        v
  화면에 그릴 이미지를 그 프레임으로 교체
        |
        v
  캔버스가 다시 그려지면 새 프레임이 보임


[캔버스 저장]                          [캔버스 다시 불러오기]
  GIF 프레임 데이터는 저장 안 함              assetId로 원본 GIF를 저장소에서 재조회
  "assetId"라는 짧은 참조값만 저장     -->    원본을 프레임 목록으로 재분해
                                              성공 -> 애니메이션 GIF로 복원
                                              실패(원본 없음/깨짐) -> 정지 이미지로 대체
```

숫자로 예를 들면, 프레임이 3장(빨강 → 초록 → 파랑)인 GIF가 있다고 하자. `advanceFrame()`을 세 번 부르면 순서는 빨강 → 초록 → 파랑 → (다시) 빨강이 된다. 세 번째 호출에서 다시 처음으로 돌아오는 게 "순환"이다.

## 쉽게 말하면

기존 정지 이미지 오브젝트가 "사진 한 장이 붙은 액자"라면, `AnimatedGif`는 "여러 장의 사진을 넣고 페이지를 넘길 수 있는 미니 앨범이 붙은 액자"다. 액자 틀(회전·크기조절·이동 같은 기능)은 완전히 똑같은 걸 그대로 쓰고, 안에 든 내용물만 여러 장으로 바뀐 것뿐이다. "페이지를 넘겨라"라는 버튼(`advanceFrame`)만 이번에 달았고, 그 버튼을 "1초에 몇 번 누를지" 정하는 건 다음 단계다.

## 만든 파일

- `src/fabric/AnimatedGif.js` — `AnimatedGif` 오브젝트 클래스 본체.
- `src/fabric/AnimatedGif.test.js` — 검증용 테스트 7개.

## 제대로 됐는지 어떻게 확인했나?

아래 항목을 자동 테스트로 만들어 전부 통과를 확인했다(`npm run lint && npm run build && npm run test`, 총 132개 테스트 전부 통과, 기존 테스트도 깨지지 않음).

- 페이지 넘기기(`advanceFrame`) 한 번 호출 시 다음 프레임으로 정확히 바뀌는지.
- 프레임 개수만큼 반복 호출하면 다시 처음 프레임으로 돌아오는지(순환 확인).
- 프레임이 1장뿐인 경우 `advanceFrame`을 불러도 아무 일도 안 일어나는지(방어 확인).
- 저장 형식(`toObject`)에 참조값(`assetId`)·프레임별 시간(`frameDelays`)·현재 프레임 번호가 포함되고, 프레임 이미지 데이터 자체는 포함되지 않는지.
- 캔버스 라이브러리가 `AnimatedGif`라는 이름의 오브젝트 종류를 제대로 인식하는지.
- 저장소에 원본이 있을 때, 참조값만으로 애니메이션이 정상 복원되는지.
- 저장소에 원본이 없을 때(삭제된 경우 등), 에러 없이 정지 이미지로 안전하게 대체되는지.

## 전체 그림에서 지금 어디?

```
[완료] step 0: GIF 파일 -> 프레임 목록으로 분해
[완료] step 1: 프레임 목록 -> 캔버스가 다루는 "오브젝트"로 포장  (지금 여기)
[예정] step 2: "1초에 몇 번 페이지를 넘길지" 정하는 공유 타이머
[예정] step 3: 사용자가 GIF를 실제로 캔버스에 올리는 화면 연결
[예정] step 4: 저장/새로고침 후에도 애니메이션이 이어지는지 마무리 점검
```

지금까지는 "오브젝트를 만들 수 있다"와 "저장/복원 형식이 맞다"까지 확인된 상태다. 다음 단계에서 실제로 이 `advanceFrame()`을 주기적으로 눌러주는 타이머를 붙여야 화면에서 눈으로 움직임을 볼 수 있다.

## 아직 안 된 것 (솔직하게)

- 화면에서 실제로 GIF가 움직이는 걸 눈으로 볼 수는 아직 없다. 이번 단계는 "부품"만 만들었고, 그 부품을 주기적으로 작동시키는 타이머(다음 단계)와, 사용자가 GIF 파일을 선택했을 때 이 부품을 실제로 만들어 캔버스에 얹는 연결 작업(그 다음 단계)이 남아있다.
- "원본이 없으면 정지 이미지로 대체"하는 안전장치는 이번 단계에서 새로 정한 규칙이며, 합의 문서(설계 문서)에 명시적으로 적혀있던 내용은 아니다. 캔버스 전체가 깨지는 것보다는 낫다고 판단해 추가했다.
