# Step 4: gif-persistence

## 이 단계를 왜 했나?

step 0~3에서 GIF를 프레임으로 쪼개고, 캔버스 오브젝트로 만들고, 자동으로 다음 프레임을 넘겨주고, 실제로 화면에 올리는 것까지 전부 끝났다. 이제 마지막으로 남은 질문 하나: "다이어리를 저장했다가 새로고침(또는 다른 날 다시 열기)해도 GIF가 계속 움직이는가?"

이건 그냥 "될 것 같다"가 아니라 실제로 저장 → 불러오기를 끝까지 돌려서 눈으로 확인해야 하는 부분이다. 저장할 때는 원본 GIF 파일 전체를 다시 담지 않고 "저장소(IndexedDB)의 몇 번 파일을 보면 된다"는 표지판(참조)만 남기기로 이미 설계돼 있었는데(3단계 이전 문서에서 합의한 방식), 이 표지판만 갖고 정말로 애니메이션을 되살릴 수 있는지, 그리고 되살아난 GIF가 저절로 다시 움직이는지를 이번 단계에서 검증했다.

## 무엇을 만들었나?

이번 단계는 새 기능을 만들었다기보다 "이미 만든 배선이 실제로 맞물려 돌아가는지" 확인하고, 확인 과정에서 발견한 작은 낭비 하나를 고쳤다.

1. **불필요한 이중 조회 제거**: 캔버스를 불러올 때, 저장소 참조(assetId)를 가진 모든 오브젝트에 대해 "저장소에서 파일을 찾아 임시 링크(objectURL)를 만들어주는" 공통 처리 단계가 있다. 그런데 애니메이션 GIF 오브젝트는 이미 스스로 저장소를 찾아가 원본을 다시 디코딩하는 자기만의 복원 로직을 갖고 있어서, 이 공통 처리 단계가 만들어주는 임시 링크는 애초에 쓰이지도 않는다. 그런데도 지금까지는 GIF에 대해서도 이 공통 단계가 똑같이 저장소 조회를 한 번 더 하고, 쓰이지도 않을 임시 링크까지 만들어서 해제도 안 하고 버려두고 있었다. 이번에 "이 오브젝트가 애니메이션 GIF라면 이 공통 단계는 건드리지 말고 그냥 통과시켜라"는 한 줄을 추가해서, 쓸데없는 조회와 버려지는 임시 링크를 없앴다.

2. **저장 → 불러오기 전 과정을 실제로 돌려서 확인**: 3장짜리 진짜 애니메이션 GIF를 캔버스에 올리고, 저장 데이터로 직렬화한 뒤, 완전히 새로운 캔버스 인스턴스에 그 저장 데이터를 불러와서 (1) 애니메이션 GIF로 제대로 복원되는지, (2) 프레임이 다시 3장 다 살아나는지, (3) 자동 재생 타이머(공유 렌더 루프)에 다시 등록돼서 시간이 지나면 저절로 다음 프레임으로 넘어가는지까지 전부 실제로 실행해서 확인하는 테스트를 새로 만들었다.

3. **저장 용량 검증**: 저장 데이터 안에 "디코딩된 프레임 그림 자체"가 통째로 들어있지는 않은지(설계상 절대 들어가면 안 되는 부분) 직접 저장 데이터를 문자열로 바꿔서 눈으로 확인하는 테스트를 추가했다.

## 데이터가 어떻게 흐르나?

```
[저장할 때]
움직이는 GIF 오브젝트
  |
  | 저장소 참조(어느 파일인지), 프레임 간격, 지금 몇 번째 프레임인지만 뽑아냄
  | (디코딩된 프레임 그림 자체는 뽑아내지 않음)
  v
저장 데이터(JSON) 안의 GIF 부분:
  { 파일 참조 ID, [프레임1 간격, 프레임2 간격, 프레임3 간격], 현재 프레임 번호 }
  예: { assetId: "abc-123", frameDelays: [100, 100, 100], currentFrameIndex: 0 }


[새로고침 후 불러올 때]
저장 데이터(JSON) 읽음
  |
  v
"이 오브젝트, 저장소 참조를 갖고 있네?" 하는 공통 처리 단계
  |
  | 타입이 "애니메이션 GIF"면?  -->  건드리지 않고 그냥 통과 (이번에 고친 부분)
  | 타입이 그냥 이미지면?      -->  저장소에서 파일 찾아 임시 링크 채워줌 (기존 그대로)
  v
애니메이션 GIF 자신의 복원 로직 작동
  |
  | 저장소 참조(assetId)로 원본 GIF 파일을 저장소에서 다시 찾음
  | 찾은 파일을 다시 프레임 3장으로 쪼갬 (1단계에서 만든 도구 재사용)
  v
화면에 다시 나타난 GIF 오브젝트 (프레임 3장 다시 갖고 있음)
  |
  v
공유 재생 타이머에 자동으로 등록됨 (오브젝트가 캔버스에 추가되는 순간 자동으로 걸림)
  |
  v
시간이 지나면 저절로 다음 프레임으로 넘어감 -> 계속 움직임
```

예를 들어 원본 GIF가 프레임 3장, 각 프레임 100밀리초(0.1초)짜리라면, 저장 데이터에는 그림 데이터 없이 `frameDelays: [100, 100, 100]`처럼 숫자 3개와 파일 참조 ID 문자열 하나만 남는다. 새로고침 후 이 오브젝트가 캔버스에 다시 올라가는 순간, 100밀리초마다 자동으로 다음 프레임으로 넘어가는 동작이 다시 시작된다 — 사람이 따로 "재생 버튼"을 누를 필요가 없다.

## 쉽게 말하면

영화 필름 전체를 금고(저장 데이터)에 넣어두는 대신, "이 영화는 창고 몇 번 선반에 있다"는 쪽지 하나만 금고에 넣어두는 방식이다. 나중에 금고를 다시 열면, 쪽지를 보고 창고로 가서 필름을 다시 꺼내와 영사기에 걸면 된다. 이번 단계는 실제로 금고를 잠갔다 열어보고, 쪽지가 맞는 창고를 가리키는지, 필름을 제대로 다시 꺼내오는지, 영사기가 알아서 다시 돌아가는지를 처음부터 끝까지 직접 재생해본 것이다. 그리고 그 과정에서, 원래 필요 없는데도 창고를 한 번 더 들락거리던 불필요한 걸음 하나를 없앴다.

## 만든 파일

- `src/hooks/useFabricCanvas.js` — 저장소 참조를 임시 링크로 채워주는 공통 처리 함수에, "애니메이션 GIF 타입은 건드리지 않고 통과시킨다"는 분기 한 줄 추가.
- `src/hooks/useFabricCanvas.test.js` — 아래 3가지를 검증하는 테스트 추가.
  - 애니메이션 GIF 오브젝트는 저장소 참조가 있어도 임시 링크를 채우지 않고 그대로 통과하는지.
  - 애니메이션 GIF를 직렬화한 저장 데이터에 "프레임 그림 자체"가 들어있지 않은지.
  - 저장 데이터를 완전히 새 캔버스에 불러오면 애니메이션 GIF로 복원되고, 자동 재생 타이머에 다시 등록돼 시간이 지나면 실제로 다음 프레임으로 넘어가는지.

## 제대로 됐는지 어떻게 확인했나?

`npm run lint && npm run build && npm run test`를 실행해 전부 통과를 확인했다(테스트 25개 파일, 총 147개 전부 통과 — 기존 144개 + 이번에 추가한 3개, 기존 테스트도 깨지지 않음).

새로 추가한 테스트로 아래를 직접 실행해 확인했다.

- 3장짜리 진짜 애니메이션 GIF를 캔버스에 올리고 저장 데이터로 바꾼 뒤, 그 데이터를 문자열로 펼쳐서 "프레임" 관련 키가 전혀 없는 것을 확인(용량을 부풀리지 않는다는 설계 약속이 실제로 지켜지는지).
- 그 저장 데이터를 완전히 새로운 캔버스에 불러온 뒤, 복원된 오브젝트가 진짜 "애니메이션 GIF" 종류인지, 프레임이 다시 3장인지 확인.
- 시간을 흘려보내는 것을 흉내 내는 방식으로 자동 재생 타이머의 한 틱을 강제로 실행시켜, 복원된 GIF가 실제로 다음 프레임으로 넘어가고 화면이 다시 그려지는지 확인 — 이게 "공유 렌더 루프에 등록되어 있다"는 걸 가장 확실하게 보여주는 방법이라 판단했다.
- 저장소 참조를 가진 애니메이션 GIF 오브젝트를 공통 처리 함수에 넣었을 때, 원본 오브젝트가 전혀 바뀌지 않고 그대로 통과하는지 확인.

## 전체 그림에서 지금 어디?

```
[완료] step 0: GIF 파일 -> 프레임 목록으로 분해
[완료] step 1: 프레임 목록 -> 캔버스가 다루는 "오브젝트"로 포장
[완료] step 2: "1초에 몇 번 페이지를 넘길지" 정하는 공유 타이머
[완료] step 3: 사용자가 GIF를 실제로 캔버스에 올리는 화면 연결
[완료] step 4: 저장/새로고침 후에도 애니메이션이 이어지는지 확인  (지금 여기, 마지막 단계)
```

이걸로 애니메이션 GIF 지원 작업(animated-gif-support)의 계획된 단계가 모두 끝났다. GIF를 캔버스에 올리고, 회전·크기조절·이동 같은 기존 조작을 하고, 저장했다가 새로고침해도 계속 움직이는 것까지 전부 확인됐다.

## 아직 안 된 것 (솔직하게)

- 이번에도 테스트는 손수 만든 아주 작은 2x2픽셀짜리 GIF로 검증했다. 실제 사용자가 올릴 법한 큼직한 진짜 GIF 파일로 브라우저에서 직접 눈으로 저장→새로고침→재생까지 확인하는 절차는 아직 안 했다.
- 처음 합의 문서(PRD)에 있던 "GIF를 5개 이상 동시에 올려도 다른 조작이 끊기지 않는다"는 기준은 설계(공유 타이머 하나로 전부 처리)상 구조적으로 성립하도록 만들어졌고 step 2에서 그 구조 자체는 테스트했지만, 이번 단계에서 실제로 5개 이상을 동시에 올려놓고 체감 성능을 눈으로 확인하지는 않았다.
- 프레임 간격이 매우 짧은(예: 20밀리초 미만) GIF의 경우, 화면 재생 빈도(보통 1초에 60번)보다 프레임이 더 촘촘하면 정확히 원본과 같은 속도로는 못 넘어갈 수 있다는 점은 처음 설계 단계(ADR-4)에서부터 알려진 한계이고, 이번 단계에서 새로 발생한 문제는 아니다.
