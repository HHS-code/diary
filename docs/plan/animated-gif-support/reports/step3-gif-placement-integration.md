# Step 3: gif-placement-integration

## 이 단계를 왜 했나?

여태까지 만든 부품(GIF를 프레임으로 쪼개는 도구, 그 프레임들을 담는 액자, 액자의 페이지를 자동으로 넘겨주는 감독관)은 전부 준비돼 있었지만, 사용자가 실제로 GIF 파일을 화면에서 선택했을 때 이 부품들을 엮어서 캔버스에 올려주는 연결부가 없었다. 파일을 등록하는 창구는 두 곳이다 — 에셋 등록 패널(폴더 째로 여러 파일을 미리 등록해두는 곳)과 사이드바의 "이미지 추가" 버튼(파일 하나를 바로 캔버스에 올리는 곳). 이 두 창구가 각자 다른 방식으로 GIF인지 아닌지를 판단하게 두면, 나중에 한쪽만 고치고 다른 쪽을 깜빡하는 실수가 생기기 쉽다. 그래서 "이 파일이 애니메이션 GIF인지 판단해서 알맞은 오브젝트를 만드는 일"을 한 곳에 모아두고, 두 창구가 똑같이 그 한 곳을 부르도록 만드는 게 이번 단계의 핵심이다.

추가로, 사이드바의 "이미지 추가" 버튼은 지금까지 파일을 저장소(IndexedDB)에 남기지 않고 파일 내용을 통째로 문자열로 바꿔서(base64) 캔버스 저장 데이터에 직접 박아 넣는 예전 방식을 쓰고 있었다. 이 방식으로는 GIF를 다시 불러올 때 "원본 GIF 파일이 어디 있는지" 알 방법이 없다 — 이미 캔버스 첫 프레임만 문자열로 박제된 상태이기 때문이다. 그래서 이 버튼도 저장소를 거치는 방식으로 바꿔야 했다. 이건 GIF만이 아니라 이 버튼으로 올리는 모든 이미지에 적용되는 변화다.

## 무엇을 만들었나?

1. **GIF 판별 공용 창구** (`createImageOrAnimatedGifObject`): 파일 하나를 넣으면 "이건 그냥 이미지야" 또는 "이건 움직이는 GIF야"를 판단해서 알맞은 오브젝트를 만들어 돌려주는 함수 하나를 새로 만들었다. 이미지 확대·축소나 캔버스에 올리는 작업은 이 함수가 하지 않는다 — 오직 "무슨 오브젝트를 만들지"만 결정한다.

2. **에셋 등록 패널 연결** (`canvasAssetPlacement.js`): 미리 등록해둔 이미지를 캔버스에 배치하는 기존 코드가, 이제 이 공용 창구를 거쳐 GIF면 움직이는 오브젝트로, 아니면 예전처럼 정지 이미지로 올린다.

3. **사이드바 버튼 전환** (`ImageUploadButton.jsx`): 파일을 문자열로 바꿔 박제하던 방식을 걷어내고, 파일을 저장소에 먼저 저장한 뒤 그 저장소 참조를 들고 같은 공용 창구를 부르도록 바꿨다. GIF든 아니든 이제 이 버튼으로 올리는 모든 이미지가 저장소 참조를 갖게 된다.

## 데이터가 어떻게 흐르나?

```
[사용자가 GIF 파일을 선택]
         |
         v
+---------------------------------------------+
|  createImageOrAnimatedGifObject              |
|  (판별은 여기 딱 한 곳에서만)                 |
+---------------------------------------------+
   |                        |
   | 파일 형식이 GIF가       | 파일 형식이 GIF고
   | 아니거나, GIF지만       | 프레임이 2장 이상
   | 프레임이 1장뿐          |
   v                        v
[정지 이미지로 반환]     [움직이는 GIF 오브젝트로 반환]
   |                        |
   +-----------+------------+
               v
   [호출한 쪽이 크기 조절 + 캔버스 중앙에 배치]
               |
               v
   [캔버스에 추가 -> 공유 타이머(step2)에 자동 등록]
```

두 창구(에셋 등록 패널 경로, 사이드바 버튼 경로)는 이 판별 창구 하나를 똑같이 부른다. 예를 들어 3장짜리 애니메이션 GIF를 올리면 어느 경로로 올리든 똑같이 "움직이는 GIF 오브젝트"가 나오고, 프레임이 1장뿐인 GIF(움직이지 않는 GIF)를 올리면 어느 경로든 똑같이 "정지 이미지"가 나온다.

사이드바 버튼은 추가로 이런 변화가 있다.

```
[예전] 파일 선택 -> 파일을 통째로 문자열로 변환 -> 캔버스 저장 데이터에 직접 박음
[지금] 파일 선택 -> 저장소(IndexedDB)에 저장 -> 저장소 참조(ID)만 캔버스 저장 데이터에 남김
```

## 쉽게 말하면

우체국 접수 창구가 두 곳(에셋 등록 패널, 사이드바 버튼)인데, 예전엔 각 창구 직원이 "이게 움직이는 편지인지 아닌지" 각자 눈대중으로 판단했다. 이번엔 그 판단을 도맡아 하는 검수 담당자를 한 명 새로 뽑아서, 두 창구 모두 소포를 이 담당자에게 먼저 보여주고 판정을 받은 뒤에야 처리하도록 바꿨다. 담당자 한 명만 있으니 판정 기준이 두 창구에서 어긋날 일이 없다.

## 만든 파일

- `src/fabric/placeImageOrGif.js` — GIF 판별 공용 창구(`createImageOrAnimatedGifObject`) 본체.
- `src/hooks/canvasAssetPlacement.js` — 에셋 등록 패널 경로가 이 공용 창구를 쓰도록 수정.
- `src/hooks/canvasAssetPlacement.test.js` — GIF(움직임)/정지 GIF(1프레임)/일반 이미지(PNG) 3가지 경우 검증 케이스 추가.
- `src/components/ImageUploadButton/ImageUploadButton.jsx` — 문자열 박제 방식을 걷어내고 저장소 경로 + 공용 창구로 전환.
- `src/components/ImageUploadButton/ImageUploadButton.test.jsx` — 신규. 저장소 저장 여부, 크기 조절, GIF/정지 이미지 분기를 검증.

## 제대로 됐는지 어떻게 확인했나?

`npm run lint && npm run build && npm run test`를 실행해 전부 통과를 확인했다(테스트 25개 파일, 총 144개 전부 통과, 기존 테스트도 깨지지 않음).

새로 추가한 테스트로 아래를 확인했다.

- 에셋 등록 패널 경로에서, 프레임 3장짜리 GIF를 올리면 움직이는 GIF 오브젝트가 캔버스에 추가되는지.
- 같은 경로에서, 프레임 1장뿐인 GIF와 일반 PNG를 올리면 둘 다 (움직이지 않는) 정지 이미지 오브젝트로 추가되는지.
- 사이드바 버튼에서 파일을 선택하면 저장소에 실제로 저장되고, 저장된 파일명·형식이 맞는지, 캔버스에 추가된 오브젝트가 저장소 참조(ID)를 갖는지.
- 사이드바 버튼에서도 캔버스보다 큰 이미지가 비율을 유지한 채 축소되는 기존 동작이 그대로인지(회귀 확인).
- 사이드바 버튼에서 GIF/일반 이미지를 각각 올렸을 때 오브젝트 종류가 올바르게 갈리는지.

## 전체 그림에서 지금 어디?

```
[완료] step 0: GIF 파일 -> 프레임 목록으로 분해
[완료] step 1: 프레임 목록 -> 캔버스가 다루는 "오브젝트"로 포장
[완료] step 2: "1초에 몇 번 페이지를 넘길지" 정하는 공유 타이머
[완료] step 3: 사용자가 GIF를 실제로 캔버스에 올리는 화면 연결  (지금 여기)
[예정] step 4: 저장/새로고침 후에도 애니메이션이 이어지는지 마무리 점검
```

이제 GIF 파일을 에셋 등록 패널이나 사이드바 버튼 어느 쪽으로 올리든, 실제로 화면에서 움직이는 걸 볼 수 있다. 다음 단계는 "캔버스를 저장했다가 새로고침한 뒤에도 애니메이션이 그대로 이어지는지"를 마무리로 점검하는 일이다 — 저장/불러오기 배선 자체는 이전 단계들에서 이미 만들어 뒀지만(오브젝트가 저장소 참조만 들고 있다가, 불러올 때 원본 GIF를 다시 찾아 프레임을 재구성하는 방식), 아직 이 단계에서 실제로 만든 GIF 오브젝트로 그 흐름 전체를 끝까지 확인하지는 않았다.

## 아직 안 된 것 (솔직하게)

- 이 단계에서는 GIF를 캔버스에 "올리는" 부분까지만 실제로 확인했다. 캔버스를 저장했다가 새로고침해도 애니메이션이 이어지는지는 다음 단계(step 4)에서 확인한다.
- 에셋 등록 패널 경로(`canvasAssetPlacement.js`)와 사이드바 버튼(`ImageUploadButton.jsx`) 두 곳 모두 "캔버스보다 크면 비율 유지한 채 축소"하는 계산 코드가 거의 똑같이 중복돼 있다. 설계 문서에서도 이 중복을 이번 단계에서 반드시 정리할 필요는 없다고 명시해서, 손대지 않고 그대로 뒀다.
- 브라우저에서 실제 GIF 파일로 직접 눈으로 확인하지는 않았다(테스트는 손수 만든 아주 작은 2x2픽셀 GIF로 검증). 실제 GIF 파일로 동작을 확인하는 건 아직 남아 있다.
