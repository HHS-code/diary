# 빌드 리포트 — diary (animated-gif-support)

각 step이 무엇을 했는지 위에서 아래로 읽으면 프로젝트가 자라는 과정이 보인다.

- [완료] **Step 0: gif-frame-decoder** — 공개 시그니처: decodeGifFrames(blob: Blob) -> Promise<{ frames: HTMLCanvasElement[], delays: number[] }> (src/fabric/gifFrameDecoder.js). gifuct-js의 parseGIF/decompressFrames(buildImagePatches=true)로 얻은 프레임별 patch(dims, disposalType)를 캔버스 하나(compositeCanvas)에 순서대로 누적해 매 프레임을 '완전 합성된' 새 canvas로 clone해 반환 — disposalType 2(restore-to-background)는 clearRect, 3(restore-to-previous)는 직전 스냅샷 복원, 나머지는 그대로 둔다. delay는 gifuct-js가 이미 ms 단위로 변환해 주므로 그대로 쓰되 MIN_FRAME_DELAY_MS=20으로 하한을 둔다. header.signature !== 'GIF'거나 프레임이 0개면 Error를 던진다(호출부 책임 없음, 조용히 무시 안 함). 테스트(gifFrameDecoder.test.js)는 실제 파일을 추가하지 않고 손수 만든 base64 GIF 바이너리(disposal method 2종 포함)를 인라인해 픽셀 단위로 합성 결과를 검증한다. 다음 step(animated-gif-object)은 이 함수를 호출해 frames/delays를 FabricImage 상속 오브젝트에 넣으면 된다 — Fabric은 이 파일에서 import하지 않았다. ([리포트](step0-gif-frame-decoder.md))
- [완료] **Step 1: animated-gif-object** — 공개 시그니처: class AnimatedGif extends FabricImage (src/fabric/AnimatedGif.js), static type='AnimatedGif', static customProperties=['assetId','frameDelays','currentFrameIndex']. constructor(element, options)는 options.frames(HTMLCanvasElement[], 기본 [element])/frameDelays/currentFrameIndex(기본 0)를 세팅. advanceFrame()은 currentFrameIndex를 순환시키고 this._element를 frames[idx]로 교체 — frames.length<=1이면 no-op. _render/render/toObject는 오버라이드하지 않음(customProperties 메커니즘에 위임). static async fromObject(object, options)는 assetId로 getAsset() 조회 후 decodeGifFrames(record.blob)로 재디코딩해 new AnimatedGif(frames[0], {...object, frames, frameDelays: delays})를 반환하고, getAsset이 null이거나 디코딩 실패 시 FabricImage.fromObject(object에서 assetId/frameDelays/currentFrameIndex 제외한 나머지, options)로 폴백(에러를 던지지 않음). 파일 하단에서 classRegistry.setClass(AnimatedGif) 호출. 주의: FabricImage.toObject를 오버라이드하지 않으므로 직렬화 시 상속받은 src 필드(getSrc() 결과, 현재 프레임의 dataURL)가 함께 포함된다 — 이 값이 폴백 경로에서 FabricImage.fromObject가 이미지를 복원하는 데 쓰인다. 다음 step(shared-render-loop)은 advanceFrame()을 호출해 프레임을 넘기고 canvas.renderAll()을 호출하는 requestAnimationFrame 루프를 만들면 된다 — 이 파일은 스케줄링을 하지 않는다. ([리포트](step1-animated-gif-object.md))
- [완료] **Step 2: shared-render-loop** — 공개 시그니처: createSharedGifRenderLoop(fabricCanvas) -> { register(obj), unregister(obj), start(), stop() } (src/fabric/sharedGifRenderLoop.js). 내부 Set+WeakMap으로 등록 오브젝트별 마지막 프레임전환 시각을 추적, 매 rAF 틱마다 obj.frameDelays[obj.currentFrameIndex]가 지난 오브젝트만 advanceFrame() 호출하고, 이번 틱에 하나라도 갱신됐을 때만 fabricCanvas.renderAll()을 최대 1회 호출(ADR-4). start()는 rafId가 이미 있으면 no-op, stop()은 cancelAnimationFrame 후 rafId를 null로 리셋. useFabricCanvas.js는 fabricCanvas 생성 직후 이 루프를 만들어 start()하고, object:added/object:removed에 별도 리스너를 달아 target instanceof AnimatedGif일 때만 register/unregister하며(fabric의 loadFromJSON도 내부적으로 canvas.add()를 거쳐 object:added가 발생하므로 재로드된 AnimatedGif도 이 리스너 하나로 자동 등록됨 — 별도 처리 불필요, 소스 확인함), effect cleanup에서 gifRenderLoop.stop()을 fabricCanvas.dispose() 이전에 동기 호출. 다음 step(gif-placement-integration)은 canvasAssetPlacement.js/ImageUploadButton에서 AnimatedGif 인스턴스를 만들어 canvas.add()하기만 하면 이 루프에 자동 register된다 — 이 step은 스케줄링 배선만 하고 GIF 판별/배치 로직은 건드리지 않았다. ([리포트](step2-shared-render-loop.md))
- [완료] **Step 3: gif-placement-integration** — 공개 시그니처: createImageOrAnimatedGifObject({ blob, assetId, mimeType }) -> Promise<FabricImage | AnimatedGif> (src/fabric/placeImageOrGif.js) — 이게 GIF 판별/분기의 유일한 위치다. mimeType(없으면 blob.type)이 image/gif가 아니거나 decodeGifFrames 결과 frames.length===1이면 FabricImage.fromURL(objectURL) 정지 이미지를, frames.length>=2면 new AnimatedGif(frames[0], { assetId, frames, frameDelays })를 반환한다. 스케일 다운/중앙배치/canvas.add()는 안 함(호출부 책임). canvasAssetPlacement.js의 addImageAssetToCanvas와 ImageUploadButton.jsx의 addImageToCanvas가 둘 다 이 함수 하나만 호출하도록 바꿈. ImageUploadButton.jsx는 ADR-5대로 FileReader.readAsDataURL을 걷어내고 saveAsset({type:'image',...,blob:file})로 IndexedDB에 저장 후 반환된 id를 assetId로 써서 이 함수를 호출하는 방식으로 전환(기존처럼 input reset은 saveAsset을 await하지 않고 동기로 즉시 수행 — 같은 파일 재선택 허용 동작 유지). 신규/수정 테스트 파일: src/hooks/canvasAssetPlacement.test.js(GIF/정적GIF/PNG 3케이스 추가), src/components/ImageUploadButton/ImageUploadButton.test.jsx(신규, saveAsset 저장·스케일다운·GIF/PNG 분기 검증). npm run lint && npm run build && npm run test(144 passed) 전부 통과 확인함. 다음 step(gif-persistence)은 useFabricCanvas.js의 resolveObjectAssetReference에 AnimatedGif(type==='AnimatedGif') 통과 분기를 추가하고 저장/재로드 흐름을 검증하면 된다 — 이 step은 그 부분을 건드리지 않았다. ([리포트](step3-gif-placement-integration.md))
