# Step 2: shared-render-loop

## 이 단계를 왜 했나?

지난 단계(step 1)에서 GIF를 "여러 장의 사진이 든 미니 앨범이 붙은 액자"(`AnimatedGif` 오브젝트)로 만들었고, 그 액자에는 "페이지를 한 장 넘겨라"(`advanceFrame()`)라는 버튼도 달아뒀다. 하지만 그 버튼을 "언제, 얼마나 자주" 누를지 정하는 부분은 아직 없었다 — 버튼만 있고 누르는 사람이 없는 상태였다.

이번 단계는 그 버튼을 자동으로 눌러주는 "공유 타이머"를 만드는 작업이다. 여기서 "공유"가 중요한 이유가 있다. 만약 GIF마다 각자 자기 타이머를 하나씩 갖고 있으면, GIF를 10개 올렸을 때 타이머도 10개가 되고, 화면을 다시 그리는 작업(브라우저 용어로 렌더링)도 GIF 개수만큼 늘어난다. 그러면 GIF가 늘어날수록 다른 오브젝트를 드래그하거나 회전시키는 조작이 버벅거리게 된다. 그래서 캔버스 하나당 타이머를 딱 하나만 두고, 그 타이머가 "지금 넘어갈 때가 된 GIF들"만 골라서 넘긴 다음, 화면 다시 그리기도 한 번만 하도록 만들었다.

## 무엇을 만들었나?

`sharedGifRenderLoop`라는 공유 타이머를 만들었다. 캔버스가 하나 생길 때 이 타이머도 하나 같이 만들어지고, 캔버스가 사라질 때 같이 정리된다.

동작 방식은 이렇다.

1. 캔버스에 애니메이션 GIF가 추가되면, 이 타이머의 "관리 목록"에 등록된다.
2. 이 타이머는 브라우저가 화면을 새로 그릴 때마다(보통 1초에 60번, "틱"이라고 부르자) 한 번씩 관리 목록을 훑는다.
3. 각 GIF마다 "마지막으로 페이지를 넘긴 시각"을 기억해두고 있다가, "그 GIF의 원래 프레임 표시 시간"이 지났으면 그때만 `advanceFrame()`을 눌러 페이지를 넘긴다. 아직 시간이 안 된 GIF는 이번 틱에서는 건드리지 않는다.
4. 이번 틱에서 페이지가 한 장이라도 넘어갔다면, 화면을 딱 한 번만 다시 그린다(GIF가 몇 개 동시에 넘어갔든 상관없이 1번).
5. GIF가 캔버스에서 삭제되면 관리 목록에서도 빠진다 — 더 이상 페이지를 넘기지 않는다.
6. 캔버스 자체가 사라질 때(예: 다이어리 화면을 나갈 때) 타이머도 완전히 멈춘다. 안 멈추면 이미 사라진 캔버스를 계속 다시 그리려는 시도가 남아서 메모리 누수(쓰지도 않는데 계속 자리를 차지하는 문제)로 이어질 수 있다.

## 데이터가 어떻게 흐르나?

```
[캔버스 생성]
     |
     v
[공유 타이머 시작] --- 캔버스당 딱 1개
     |
     |  (매 틱, 보통 1초에 60번)
     v
+--------------------------------------------+
| 관리 목록의 GIF들을 하나씩 확인             |
|   "마지막으로 넘긴 뒤 지난 시간"            |
|      >= "이 GIF의 이번 프레임 표시 시간"?   |
|      예 -> 페이지 넘김(advanceFrame)        |
|      아니오 -> 이번 틱은 건너뜀             |
+--------------------------------------------+
     |
     v
이번 틱에 한 장이라도 넘어갔으면
     |
     v
화면 다시 그리기 (딱 1번, GIF 개수와 무관)


[GIF를 캔버스에 추가] -> 관리 목록에 등록
[GIF를 캔버스에서 삭제] -> 관리 목록에서 제거
[캔버스 자체가 사라짐] -> 타이머 완전히 정지
```

숫자로 예를 들어보면 이렇다. GIF A는 프레임을 100밀리초(0.1초)마다 넘기고, GIF B는 200밀리초(0.2초)마다 넘긴다고 하자. 타이머는 1초에 60번(약 16.7밀리초마다) 도는데, 매번 두 GIF를 다 확인은 하지만 실제로 넘어가는 건 다르다 — A는 대략 6번째 틱마다(약 100ms), B는 대략 12번째 틱마다(약 200ms) 넘어간다. 두 GIF가 같은 틱에서 동시에 넘어가는 순간(대략 200ms마다 한 번씩)에도 화면 다시 그리기는 1번만 일어난다.

GIF를 5개, 10개로 늘려도 이 "화면 다시 그리기는 틱당 최대 1번"이라는 규칙은 그대로다. 그래서 GIF 개수가 늘어나도 다른 오브젝트를 조작할 때 버벅임이 눈에 띄게 늘지 않는다.

## 쉽게 말하면

GIF마다 개인 비서를 붙여서 "각자 알아서 몇 초에 한 번씩 페이지 넘기고, 넘길 때마다 전체 그림을 다시 그려주세요"라고 시키면, 비서가 10명이면 그림을 다시 그리는 요청도 10번씩 겹쳐서 들어온다. 이번 단계는 그 대신 감독관 한 명을 두고, "매초 60번 순찰 돌면서, 넘길 때 된 GIF들만 모아서 한꺼번에 넘기고, 그림도 한 번만 다시 그려주세요"라고 바꾼 것이다. 감독관은 캔버스 하나에 딱 한 명뿐이다.

## 만든 파일

- `src/fabric/sharedGifRenderLoop.js` — 공유 타이머(감독관) 본체.
- `src/fabric/sharedGifRenderLoop.test.js` — 검증용 테스트 4개.
- `src/hooks/useFabricCanvas.js` — 캔버스가 생성/삭제될 때 이 타이머도 같이 시작/정지되도록, 그리고 GIF가 추가/삭제될 때 관리 목록에 자동으로 등록/제거되도록 연결(수정).
- `src/hooks/useFabricCanvas.test.js` — 캔버스가 사라질 때 타이머도 같이 멈추는지 확인하는 회귀 테스트 1개 추가.

## 제대로 됐는지 어떻게 확인했나?

`npm run lint && npm run build && npm run test`를 실행해 전부 통과를 확인했다(테스트 24개 파일, 총 137개 전부 통과, 기존 테스트도 깨지지 않음).

새로 추가한 테스트로 아래를 확인했다.

- 등록된 GIF가 "표시 시간"이 지난 뒤 틱에서만 페이지가 넘어가고, 그때 화면 다시 그리기도 호출되는지.
- GIF 여러 개가 같은 틱에서 동시에 페이지를 넘겨야 하는 상황에서도, 화면 다시 그리기는 그 틱에 딱 1번만 일어나는지.
- 관리 목록에서 뺀(삭제된) GIF는 그 뒤로 더 이상 페이지가 안 넘어가는지.
- 타이머를 멈추면 실제로 브라우저의 "다음 틱 예약"이 취소되는지.
- (회귀 테스트) 캔버스가 화면에서 사라질 때 타이머도 같이 멈춰서, 이미 예약돼 있던 다음 틱이 실행되더라도 사라진 캔버스를 다시 그리려는 시도가 일어나지 않는지.

## 전체 그림에서 지금 어디?

```
[완료] step 0: GIF 파일 -> 프레임 목록으로 분해
[완료] step 1: 프레임 목록 -> 캔버스가 다루는 "오브젝트"로 포장
[완료] step 2: "1초에 몇 번 페이지를 넘길지" 정하는 공유 타이머  (지금 여기)
[예정] step 3: 사용자가 GIF를 실제로 캔버스에 올리는 화면 연결
[예정] step 4: 저장/새로고침 후에도 애니메이션이 이어지는지 마무리 점검
```

이제 "GIF 오브젝트를 캔버스에 올리기만 하면 자동으로 움직인다"는 배선까지는 갖춰졌다. 다만 아직 사용자가 GIF 파일을 실제로 선택했을 때 이 `AnimatedGif` 오브젝트를 만들어서 캔버스에 올려주는 연결부(다음 단계)가 없어서, 지금은 화면에서 눈으로 확인할 방법이 없다.

## 아직 안 된 것 (솔직하게)

- 여전히 화면에서 실제로 GIF가 움직이는 걸 눈으로 볼 수는 없다. 이번 단계는 "타이머 배선"까지만 했고, 사용자가 GIF 파일을 사이드바나 파일 등록 화면에서 선택했을 때 이 부품들을 실제로 엮어서 캔버스에 올려주는 작업(다음 단계)이 남아있다.
- 이번 단계 테스트 중 하나는 실제 브라우저 대신 테스트 환경(자동 검증용 가짜 브라우저)에서 캔버스를 없앨 때, 캔버스 그림 라이브러리(Fabric.js)가 화면 요소를 다루는 방식과 화면을 그리는 도구(React)가 요소를 정리하는 방식이 살짝 어긋나 나는 부수적인 오류 하나를 무시하도록 처리했다. 이 오류는 이번에 만든 타이머 정지 로직과는 무관하고, 실제 브라우저 사용 환경에서 문제를 일으킨다는 근거는 없다 — 다만 근본 원인 자체를 고치는 작업은 이번 단계 범위 밖이라 다루지 않았다.
