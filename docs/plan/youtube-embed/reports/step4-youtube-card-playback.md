# Step 4: youtube-card-playback

## 이 단계를 왜 했나?

지난 단계까지 유튜브 카드는 "화면에 나타나고, 저장해도 사라지지 않는" 상태까지 만들어졌다. 하지만 카드를 눌러도 아무 일도 일어나지 않았다 — 여전히 클릭하면 선택만 될 뿐, 정지된 썸네일 그림일 뿐이었다. 이번 단계는 카드를 클릭하면 그 자리에서 실제로 영상이 재생되도록 만드는, 이 기능의 핵심 목표다.

## 무엇을 만들었나?

캔버스(그림 그리는 영역)는 원래 `<canvas>`라는 HTML 태그 하나로 되어 있고, 그 안에 그려지는 모든 것(사진, 스티커, 글씨, 유튜브 카드까지)은 전부 "그림"이다 — 실제로 클릭할 수 있는 버튼이나 재생되는 영상이 아니라, 픽셀로 그려진 그림에 불과하다. 그런데 유튜브 영상은 `<iframe>`이라는, 그림이 아니라 진짜로 살아있는 별도의 작은 웹페이지로만 재생할 수 있다. `<iframe>`은 `<canvas>` 그림 "안"에 넣을 수 없다 — 그림은 그림일 뿐이니까.

그래서 이번 단계가 한 일은: 카드를 클릭하면, 카드가 그려진 화면 좌표를 정확히 계산해서 그 자리에 딱 맞는 크기로 `<iframe>`(진짜 유튜브 재생 화면)을 캔버스 "위에" 겹쳐 놓는 것이다. 종이 위에 그려진 텔레비전 그림 위에, 딱 맞는 크기로 자른 투명 필름에 진짜 화면을 비춰서 얹어놓는 것과 비슷하다. 카드가 옮겨지거나 화면이 확대/축소되면 그 필름도 같이 움직이고 커지고 작아진다.

구체적으로 만든 것 두 가지:

1. **"지금 뭐가 재생 중인지, 어디에 놓아야 하는지" 계산하는 로직** (`useYoutubeCardPlayback`) — 카드를 클릭하면 "이 카드가 재생 중"이라고 기록하고, 카드가 아닌 빈 곳이나 다른 물건을 클릭하면 재생을 멈춘다. 재생 중인 동안에는 화면이 다시 그려질 때마다(카드를 옮기거나 확대할 때마다) "지금 화면 어디에, 얼마 크기로 있어야 하는지" 좌표를 다시 계산한다. 또한 재생 중에는 카드를 돌리거나 크기를 늘리는 조작이 안 되게 잠그고(이동만 허용), 재생을 멈추면 다시 원래대로 풀어준다.

2. **실제 iframe을 그려주는 화면 조각** (`YoutubeCardOverlay`) — 위에서 계산한 좌표·크기를 받아서 그 위치에 정확히 유튜브 재생 화면을 띄운다. 재생 중이 아니면 이 조각은 아예 존재하지 않는다(화면에서 완전히 사라진다) — "정지시켰다"가 아니라 "지워버렸다"는 뜻이라, 확실하게 소리도 영상도 멈춘다.

### 조사하며 확인한 것 (설계 문서와 실제가 다른 부분)

설계 문서(architecture.md)는 "캔버스 확대/축소(zoom)뿐 아니라 화면을 손으로 밀어서 보는(pan) 기능도 있다"고 가정했는데, 실제 코드를 뒤져보니 이 앱에는 **화면을 미는 기능(pan)이 아예 없었다**. 확대/축소(zoom)만 있다. 그래서 화면 밀기에 대한 동기화 코드는 만들지 않았다 — 없는 기능을 위한 코드를 미리 만들어두는 건 이 프로젝트의 원칙(요청한 것만 만든다)에 어긋나기 때문이다.

또 하나, 카드의 "화면 위 위치"를 계산하는 fabric 라이브러리 함수(`getBoundingRect`)가 확대/축소 배율이 반영되지 않은 값을 돌려준다는 것도 실제 코드를 읽어보고서야 확인했다(라이브러리 버전에 따라 다를 수 있다고 설계 문서도 미리 경고해뒀던 부분). 그래서 이 값에 확대/축소 배율을 직접 곱해서 최종 화면 좌표를 계산하도록 만들었다.

### 예상 밖에 발견하고 고친 문제

이번 단계를 검증하던 중 뜻밖의 문제를 하나 발견했다: **재생 중이 아니어도, 유튜브 카드가 캔버스에 하나라도 있으면 "PNG로 저장하기" 버튼과 "모아보기" 갤러리 썸네일 만들기 기능이 전부 에러를 내며 멈춰버렸다.** 원인은 이번 단계와 직접 관련은 없고, [1]단계에서 유튜브 썸네일 그림을 가져올 때 "이 그림은 다른 사이트(유튜브)에서 가져온 거니까, 나중에 이 그림이 포함된 캔버스를 사진 파일로 저장해도 된다"는 허가를 브라우저에 미리 구하지 않았던 게 원인이었다. 브라우저는 이런 허가 없이 가져온 외부 이미지가 캔버스에 섞이면, 보안을 위해 그 캔버스를 통째로 "오염됨" 상태로 표시하고 사진으로 저장하는 기능을 막아버린다.

다행히 유튜브 썸네일 서버(`img.youtube.com`)는 이 허가를 이미 흔쾌히 내주도록 설정되어 있었다(직접 확인함). 그래서 썸네일을 가져올 때 "허가받고 가져오기" 옵션 한 줄만 추가해 해결했다. 이 문제는 재생 여부와 무관하게 카드가 있기만 하면 항상 발생하는 문제였고, 이번 단계의 "PNG로 저장해도 재생 상태와 무관하게 항상 썸네일이 찍히는지" 확인 항목을 검증하려다가 우연히 발견했다. 원인과 고친 범위가 명확하고(딱 한 줄, 외부에서 이미 허용한 옵션을 켠 것뿐), 고치지 않으면 이번 단계의 확인 항목 자체를 통과시킬 수 없었기 때문에 바로 수정했다.

## 데이터가 어떻게 흐르나?

```
사용자가 카드를 클릭
       |
       v
"이 카드가 재생 중"으로 기록 + 카드 회전/크기조절 잠금
       |
       v
카드의 화면 위 위치 계산
  캔버스 안에서의 좌표 (예: 카드가 캔버스 왼쪽에서 200, 위에서 100만큼 떨어짐)
       x 확대/축소 배율 (예: 화면이 0.4배로 축소돼 있으면 x0.4)
       + 캔버스 자체가 화면 어디에 떠 있는지 (예: 브라우저 창 기준 왼쪽 위 (440, 146))
       = 최종 화면 좌표 (예: 왼쪽 504px, 위 174px, 폭 96px, 높이 72px)
       |
       v
그 좌표·크기에 딱 맞춰 실제 유튜브 재생 화면(iframe)을 얹음
       |
       | ... 사용자가 카드를 마우스로 60px 오른쪽, 40px 아래로 드래그 ...
       v
화면이 다시 그려질 때마다 좌표를 다시 계산 (자동)
       왼쪽 504 -> 564, 위 174 -> 214 (정확히 드래그한 만큼 함께 이동, 실측 확인)
       |
       | ... 사용자가 캔버스 빈 공간을 클릭 ...
       v
재생 중 기록을 지움 + 회전/크기조절 잠금 해제
       |
       v
iframe을 화면에서 완전히 없앰 (영상·소리 확실히 멈춤)
       |
       v
카드는 다시 정지된 썸네일 + 재생버튼 모습으로 돌아감
```

실제로 재생 중에 "PNG로 저장하기"를 눌러 받은 파일을 열어봤을 때, 재생 화면(iframe)이 아니라 정지된 썸네일(재생 버튼 포함)이 그대로 찍혀 있었다 — 이건 iframe이 애초에 `<canvas>` 태그 "밖"에 떠 있는 별개의 물건이라, 캔버스를 사진으로 뜨는 기능이 원래부터 iframe을 볼 수조차 없기 때문이다(따로 "재생 중이면 숨겨라" 같은 코드를 쓴 게 아니라, 구조적으로 애초에 안 보인다).

## 쉽게 말하면

카드는 항상 "그림"이다. 재생 버튼이 그려진 정지 사진일 뿐이다. 클릭하면, 그 사진과 정확히 같은 자리·같은 크기로 오려낸 투명 필름을 하나 그 위에 얹고, 그 필름에 진짜 유튜브 화면을 비춰준다. 필름은 사진이 움직이거나 커지면 같이 따라 움직이지만, 사진 자체(캔버스 내부)에는 전혀 손대지 않는다. 그래서 사진을 통째로 찍어서(PNG로 저장) 인화해도, 필름은 애초에 사진의 일부가 아니었으니 인화지에는 찍히지 않고 항상 원래 그림(정지 썸네일)만 나온다. 다른 곳을 클릭하면 필름을 아예 걷어버려서 영상도 소리도 바로 멈춘다.

## 만든 파일

- `src/hooks/useYoutubeCardPlayback.js` (신규) — 클릭으로 재생 시작/종료를 판단하고, 재생 중 카드의 화면 좌표·크기를 계속 계산해주는 로직
- `src/hooks/useYoutubeCardPlayback.test.js` (신규) — 클릭 시 재생 시작/종료, 회전·크기조절 잠금/해제, 다른 카드로 전환, 확대/축소에 따른 좌표 변화, 이동에 따른 좌표 재계산까지 검증하는 테스트 7건
- `src/components/YoutubeCardOverlay/YoutubeCardOverlay.jsx` (신규) — 실제 iframe을 그리는 화면 조각(재생 중 아니면 아예 없음)
- `src/components/DiaryCanvas/DiaryCanvas.jsx` (수정) — 위 두 가지를 캔버스 화면에 연결
- `src/fabric/YoutubeCard.js` (수정) — 썸네일 이미지를 가져올 때 "허가받고 가져오기" 옵션 추가(PNG 저장/갤러리 썸네일이 깨지는 버그 수정)

## 제대로 됐는지 어떻게 확인했나?

자동 검사(코드 스타일 검사 lint + 전체 테스트 241개)가 모두 통과했다. 이번 단계에서 새로 확인한 항목(7개 테스트)은: 카드를 클릭하면 재생 상태가 세팅되는지, 빈 공간을 클릭하면 해제되는지, 재생 중 카드의 회전·크기조절이 잠기고 종료 시 원래대로 돌아오는지, 재생 중이 아닌 다른 물건을 클릭해도 종료되는지, 다른 유튜브 카드를 클릭하면 재생 대상이 그쪽으로 바뀌는지, 화면이 다시 그려질 때마다 좌표가 재계산되는지, 캔버스를 확대(zoom)했을 때 재생 화면의 크기·위치도 배율만큼 함께 커지는지(이 앱에는 확대/축소를 누르는 버튼이 따로 없어서, 테스트 코드 안에서 직접 배율을 2배로 바꿔보고 확인했다).

실제 브라우저(Playwright)로도 직접 확인했다:

1. 다이어리를 열고 유튜브 링크를 붙여넣어 카드를 만든 뒤 클릭 → 카드와 정확히 같은 위치·크기에 재생 화면(iframe)이 나타남을 확인(좌표: 카드 위치 왼쪽 504px/위 174px/폭 96px/높이 72px, 재생 화면도 동일).
2. 재생 중 카드를 마우스로 (오른쪽 60px, 아래 40px) 드래그 → 재생 화면도 정확히 같은 거리만큼 함께 이동함을 확인.
3. 이 앱에는 확대/축소 버튼이 없어서, 위 항목은 자동 테스트로 대신 확인(위 참고).
4. 캔버스 빈 공간을 클릭 → 재생 화면이 완전히 사라지고, 카드가 다시 정지 썸네일+재생버튼 모습으로 돌아옴을 스크린샷으로 확인.
5. 재생 중에 "PNG 내보내기"를 눌러 실제로 받은 파일을 열어봄 → 재생 화면이 아니라 정지된 썸네일(재생버튼 포함)이 찍혀 있음을 확인. (첫 시도에서는 앞서 설명한 "허가받지 않은 이미지" 문제로 저장 자체가 실패했고, 수정 후 재시도해 성공을 확인했다.)

## 전체 그림에서 지금 어디?

```
[0] 유튜브 링크 판별
[1] 카드 오브젝트 만들기 (썸네일 + 재생버튼 모양)
[2] 붙여넣기와 연결하기 (Ctrl+V 하면 카드가 캔버스에 생김)
[3] 저장/불러오기 (파일로 저장해도 카드가 남아있음)
[4] 실제 재생 (카드 클릭하면 영상이 뜸)  <- 지금 여기, 마지막 단계
```

PRD에서 계획한 5단계(0~4)가 모두 끝났다. 유튜브 링크 붙여넣기 → 카드 생성 → 저장/불러오기 → 클릭 재생까지, 처음에 정한 목표(성공 기준)를 전부 하나씩 실제로 눌러보고 확인했다.

## 아직 안 된 것 (솔직하게)

- 재생 중인 iframe에 마우스 커서를 올리면 유튜브 자체 재생 컨트롤(일시정지, 소리 조절 등)이 그 안에서 동작한다 — 이건 유튜브 기본 플레이어 화면 자체이므로 이 앱이 따로 만든 게 아니고, PRD에서도 "유튜브 기본 플레이어 동작 그대로"라고 범위를 정해뒀다.
- 재생 중에 카드를 캔버스 밖(회색 작업대 영역)으로 드래그해서 일부가 잘리듯 걸치면, 재생 화면도 그 경계에서 잘리지 않고 원래 크기 그대로 삐져나올 수 있다 — 이번 단계에서 별도로 테스트하거나 다듬지는 않았다.
- 이 컴퓨터(샌드박스 환경)에서는 인터넷 접속이 막혀 있어, 실제 유튜브 영상 화면 대신 "동영상을 재생할 수 없음" 안내 화면만 떴다. 하지만 이번 검증에서 확인하려던 건 "정확한 위치에 정확한 크기로 뜨는지, 이동/삭제와 함께 잘 따라가는지, PNG에는 안 찍히는지"였고 이런 것들은 실제 유튜브 화면이 뜨는지와 무관하게 그대로 확인 가능했다. 인터넷이 되는 환경에서 실제 영상 재생까지는 별도로 재확인이 필요할 수 있다.
