# Step 3: youtube-card-persistence

## 이 단계를 왜 했나?

지난 단계([2] youtube-card-placement)에서 유튜브 링크를 붙여넣으면 캔버스 한가운데 카드가 나타나게 만들었다. 하지만 그건 화면에 "떠 있는" 상태일 뿐, 페이지를 새로고침하거나 다른 날짜로 갔다가 돌아오면 카드가 사라지는지, 저장된 파일을 다시 불러왔을 때 카드가 그대로 복원되는지는 아직 확인되지 않은 상태였다.

이 앱은 이미 "이미지"와 "움직이는 GIF" 오브젝트에 대해 저장/불러오기를 지원하고 있었다. 이번 단계는 그 기존 저장/불러오기 통로에 유튜브 카드가 잘못 걸리지 않고 무사히 통과하도록, 딱 한 군데 예외 처리를 추가하는 작업이다.

## 무엇을 만들었나?

이 앱이 캔버스를 저장할 때는 모든 오브젝트를 JSON(글자로 된 데이터)으로 바꾼다. 불러올 때는 그 반대로 JSON을 다시 화면의 오브젝트로 되돌린다. 이 되돌리는 과정에는 "에셋 치환"이라는 중간 단계가 하나 있다: 컴퓨터에서 올린 이미지 파일은 원본 데이터를 브라우저 저장소(IndexedDB, 앱 안에 파일을 저장해두는 공간)에 따로 보관해두고, JSON에는 "그 파일을 찾을 수 있는 꼬리표(assetId)"만 적어둔다. 불러올 때 이 꼬리표를 보고 저장소에서 실제 파일을 다시 찾아와 화면에 붙이는 게 "에셋 치환"이다.

그런데 유튜브 카드는 애초에 컴퓨터에 저장된 파일이 아니라 유튜브 서버에 있는 썸네일 이미지를 그대로 가져다 쓰는 카드다. 그래서 이 카드는 "꼬리표(assetId)"가 아니라 "영상 번호(videoId)"만 가지고 있으면 된다 — 불러올 때 영상 번호로 유튜브 썸네일 주소를 다시 조립하면 그만이다([1]단계에서 이미 만든 방식).

문제는, 기존 "에셋 치환" 로직이 어떤 오브젝트든 "꼬리표(assetId) 필드가 있으면 무조건 저장소에서 찾아 바꿔치기"하도록 되어 있었다는 점이다. 유튜브 카드에 우연히라도 assetId와 비슷한 필드가 섞이면, 이 로직이 잘못 끼어들어 카드가 기대하는 필드를 망가뜨릴 수 있었다. 그래서 이번 단계에서 한 일은:

1. **에셋 치환 로직에 "유튜브 카드는 건드리지 마라" 예외 추가**: 기존에도 "움직이는 GIF(AnimatedGif)는 이 치환 과정을 건너뛴다"는 예외가 이미 있었다(GIF도 자기만의 복원 방식이 따로 있어서). 그 옆에 "유튜브 카드(YoutubeCard)도 마찬가지로 건너뛴다"는 한 줄을 추가했다.

2. **"영상 번호가 저장 데이터에 실제로 포함되는지" 검증**: [1]단계에서 이미 "카드를 저장하면 영상 번호(videoId)도 같이 저장되게 하는 설정"을 심어뒀는데, 이번 단계에서 그게 실제로 동작하는지 테스트로 직접 확인했다. 별도 코드 수정 없이 이미 되어 있었다는 걸 검증한 것이다.

3. **저장 → 불러오기 왕복이 실제로 되는지 확인**: 카드를 만들고, 저장 형태(JSON)로 바꾼 뒤, 그 JSON을 다시 새 캔버스에 불러왔을 때 원래와 동일한 영상 번호를 가진 카드로 되살아나는지 테스트로 확인했다.

## 데이터가 어떻게 흐르나?

```
캔버스에 유튜브 카드가 있음 (videoId="dQw4w9WgXcQ")
       |
       v
저장(자동 저장 또는 "JSON 내보내기" 버튼)
       |
       +-- 카드를 글자 데이터(JSON)로 변환
       |     { type: "YoutubeCard", videoId: "dQw4w9WgXcQ", left: 560, top: 320, ... }
       |     (재생 상태·iframe 관련 정보는 애초에 여기 담기지 않음)
       v
브라우저 저장소(localStorage)에 저장됨
       |
       | ... 새로고침 또는 파일 다시 불러오기 ...
       v
불러오기 시작
       |
       v
"에셋 치환" 단계 — 오브젝트마다 꼬리표(assetId)가 있는지 확인
       |
       +-- type이 "YoutubeCard"? --- 예 --> 그냥 통과시킴 (이번 단계에서 추가한 예외)
       |                                      (실수로 assetId가 섞여 있어도 무시)
       +-- 아니면 --> 기존처럼 저장소에서 파일 다시 찾아 붙임
       v
카드 복원 (videoId="dQw4w9WgXcQ"로 썸네일 주소 재조립, [1]단계 로직)
       v
화면에 원래와 똑같은 썸네일+재생버튼 카드가 다시 나타남
```

실제 확인한 숫자: 브라우저 저장소에 저장된 데이터를 열어보면 카드 오브젝트가 `{"type":"YoutubeCard","videoId":"dQw4w9WgXcQ","left":560,"top":320,"width":480,"height":360, ...}` 형태로 들어 있었다. `assetId`나 재생 상태 같은 필드는 전혀 없었다 — 딱 필요한 것만 저장된 것이다.

## 쉽게 말하면

이 앱의 불러오기 로직은 원래 "이 오브젝트에 창고 보관표(assetId)가 붙어 있으면, 창고에 가서 실제 물건을 찾아다 놓는다"는 규칙으로 동작한다. 그런데 유튜브 카드는 창고에 보관된 물건이 아니라 "영상 번호만 있으면 언제든 유튜브에서 새로 받아올 수 있는 카드"다. 그래서 이번 단계는 창고지기(에셋 치환 로직)에게 "유튜브 카드는 네가 신경 쓸 대상이 아니니 그냥 지나쳐도 된다"고 알려준 것뿐이다. 카드 자체를 만드는 방법이나 저장하는 방법은 이미 이전 단계에서 다 만들어져 있었고, 이번엔 그게 기존 시스템과 부딪히지 않고 무사히 통과하는지 확인하고 다리 하나를 놓은 셈이다.

## 만든 파일

- `src/hooks/useFabricCanvas.js` (수정) — 에셋 치환 로직에 "YoutubeCard는 건너뛴다" 예외 한 줄 추가(및 이유를 설명하는 주석)
- `src/hooks/useFabricCanvas.test.js` (수정) — 검증 코드 3건 추가(저장 시 영상 번호 포함 확인 / 치환 건너뛰기 확인 / 저장-불러오기 왕복 확인)

## 제대로 됐는지 어떻게 확인했나?

자동 검사(코드 스타일 검사 lint + 테스트, 총 234개 테스트)가 모두 통과했다. 이번 단계에서 새로 확인한 항목은:

- 유튜브 카드를 캔버스에 추가하고 저장 형태(JSON)로 바꾸면, 그 안에 영상 번호(videoId)가 정확히 들어있는지.
- 저장 형태(JSON)에 "유튜브 카드 + 창고 보관표(assetId)"가 같이 들어있는 오브젝트를 넣었을 때, 창고 치환 로직이 이 오브젝트를 건드리지 않고 원본 그대로 통과시키는지(움직이는 GIF와 동일하게 예외 처리되는지).
- 저장 형태(JSON)를 새 캔버스에 불러오면, 원래와 동일한 영상 번호를 가진 유튜브 카드로 정확히 복원되는지.
- (금지 사항 검증) 기존 저장 설정 목록(`EXTRA_SERIALIZED_PROPS`)에 영상 번호 필드를 억지로 추가하지 않아도, 카드 자체의 설정만으로 이미 저장이 잘 되는지 — 별도 수정 없이 통과함을 확인.

추가로 실제 코드를 잠깐 원래대로 되돌려서(예외 처리를 뺀 상태로) 새로 만든 검증 코드가 정말로 실패하는지도 확인했다 — 실패하는 걸 직접 본 뒤에 다시 수정 사항을 되살렸다. 이렇게 하면 "테스트가 통과하는 이유가 우연이 아니라 실제로 이 수정이 필요했기 때문"이라는 걸 확실히 할 수 있다.

마지막으로 개발 서버를 직접 띄워 브라우저로 실제 동작을 확인했다: 다이어리 화면을 열고 유튜브 링크(`https://www.youtube.com/watch?v=dQw4w9WgXcQ`)를 붙여넣어 카드를 만든 뒤, 페이지를 완전히 새로고침하고 같은 날짜로 다시 들어가 봤다. 카드가 똑같은 위치에 똑같은 썸네일(리키 애슬리 영상)과 재생 버튼 모습으로 다시 나타났다. 브라우저에 저장된 실제 데이터도 열어봤는데, `type: "YoutubeCard"`, `videoId: "dQw4w9WgXcQ"`가 정확히 들어있는 걸 확인했다.

## 전체 그림에서 지금 어디?

```
[0] 유튜브 링크 판별
[1] 카드 오브젝트 만들기 (썸네일 + 재생버튼 모양)
[2] 붙여넣기와 연결하기 (Ctrl+V 하면 카드가 캔버스에 생김)
[3] 저장/불러오기 (파일로 저장해도 카드가 남아있음)  <- 지금 여기
[4] 실제 재생 (카드 클릭하면 영상이 뜸)
```

다음 단계인 [4]는 카드를 클릭했을 때 캔버스 위에 실제 영상 재생 화면(iframe)을 겹쳐 보여주는 작업이다. 카드가 "화면에 나타나는 것"과 "저장해도 사라지지 않는 것"이 먼저 확실해야, 그 위에 "클릭하면 재생되는" 기능을 올려도 안전하기 때문에 이 순서로 진행했다. 재생 기능 자체는 이번 단계에서 전혀 손대지 않았다 — PRD에서도 저장되는 건 영상 번호뿐이고 재생 상태는 저장하지 않기로 합의했기 때문에, 저장/불러오기와 재생은 서로 완전히 독립적인 작업이다.

## 아직 안 된 것 (솔직하게)

- 카드를 클릭해도 영상이 재생되지 않는다 — 지금도 여전히 눌러도 선택만 될 뿐이다(재생은 [4]의 몫).
- 유튜브 서버가 응답하지 않거나 인터넷 연결이 끊긴 상태에서 불러오면 썸네일이 깨진 이미지로 보일 수 있다 — 이건 PRD에서 "온라인 전제"로 이미 범위 밖(out-of-scope)으로 정한 부분이라 이번 단계에서도 손대지 않았다.
- "JSON 내보내기" 버튼으로 파일을 내려받았다가 "JSON 불러오기" 버튼으로 다시 올리는 경로는 이번에 직접 클릭해서 확인하지는 않았다 — 다만 이 버튼들이 쓰는 저장/불러오기 함수가 이번에 테스트하고 실제로 새로고침 확인까지 마친 것과 동일한 통로(`useFabricCanvas`의 저장/불러오기 로직)를 그대로 쓰기 때문에, 동작 원리상 문제 없이 통과할 것으로 본다.
