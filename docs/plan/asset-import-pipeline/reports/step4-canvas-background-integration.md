# Step 4: canvas-background-integration

## 이 단계를 왜 했나?

Step 0에서 새 창고(IndexedDB)를 만들었고, Step 1에서 예전에 저장돼 있던 배경 사진들을 그 창고로 옮겼다. 하지만 지금 사용자가 "배경 이미지" 버튼을 눌러 새 사진을 고르면, 그 버튼은 여전히 예전 방식(base64 긴 문자열)으로 사진을 저장하고 있었다. 창고는 새로 만들었는데 정작 "사진을 새로 넣는 입구" 하나가 옛날 방식 그대로였던 셈이다.

이번 step은 그 입구 하나(배경 이미지 설정 버튼)를 새 창고로 연결하는 작업이다. 사용자가 보기엔 버튼을 누르고 사진을 고르는 동작이 똑같지만, 뒤에서 저장되는 방식이 base64 문자열에서 원본 파일(Blob) 저장으로 바뀐다.

## 무엇을 만들었나?

`src/hooks/useCanvasBackground.js`의 `setImage(file)` 함수를 바꿨다.

이전 방식:
1. 사용자가 고른 파일을 브라우저 내장 기능(FileReader)으로 읽어 base64라는 아주 긴 문자열로 변환.
2. 그 문자열을 캔버스(그림판) 위 사진으로 바로 붙임.

바뀐 방식:
1. 사용자가 고른 파일을 그대로(변환 없이) 새 창고(assetStorage)에 저장하고, 그 사진의 "보관 번호"(assetId)를 받는다.
2. 그 파일로 임시 링크(objectURL)를 만든다.
3. 그 임시 링크로 캔버스 위에 사진을 붙인다. 이때 사진 오브젝트에 "이건 배경이다"라는 기존 표시(`isBackground: true`)와 더불어 "보관 번호는 몇 번이다"(`assetId`)라는 표시를 함께 남긴다.

또한 캔버스를 저장할 때(`toJSON`, 자동저장 포함) 이 "보관 번호" 표시가 함께 저장되도록 `src/hooks/useFabricCanvas.js`의 직렬화 대상 목록에 `assetId`를 추가했다. 캔버스를 불러올 때 보관 번호로 사진을 다시 꺼내오는 로직은 Step 1에서 이미 만들어 놓았으므로, 이번 step에서는 그 기존 로직이 배경 사진에도 그대로 잘 맞물려 동작하는지만 테스트로 확인했다(새로 만들지 않음).

배경색 설정, 배경 고정, 배경 초기화 기능은 사진과 무관한 기능이라 전혀 손대지 않았다.

## 데이터가 어떻게 흐르나?

```
[사용자가 "배경 이미지" 버튼으로 사진 파일을 고름]
   │
   ▼
그 파일을 그대로(base64 변환 없이) 새 창고(assetStorage)에 저장
   │
   ▼
저장된 파일의 "보관 번호"(assetId) 발급받음
   │
   ▼
그 파일로 임시 링크(objectURL) 생성
   │
   ▼
임시 링크로 캔버스에 사진 오브젝트 추가
   │  표시: isBackground = true (기존과 동일)
   │  표시: assetId = 보관번호 (이번에 새로 추가)
   │
   ▼
[500ms 후 자동저장] 캔버스 전체를 문서(JSON)로 직렬화
   │  이때 isBackground뿐 아니라 assetId도 함께 저장됨
   ▼
[다음에 앱을 열거나 새로고침]
   │
   ▼
저장된 문서를 불러오는 중 assetId를 가진 오브젝트 발견
   │  (Step 1에서 만든 로직, 이번 step은 재사용만 함)
   ▼
새 창고에서 그 보관번호로 원본 파일을 다시 꺼내 임시 링크 재생성
   │
   ▼
그 링크로 캔버스에 배경 사진 복원
```

숫자로 예를 들면: 사용자가 2MB짜리 배경 사진을 등록하면, 예전 방식은 그 사진을 약 2.7MB(1.33배)짜리 base64 문자열로 부풀려 다이어리 저장 공간(`localStorage`, 브라우저당 5~10MB 한도)에 통째로 박아 넣었다. 새 방식은 2MB 원본을 그대로 별도 창고(IndexedDB, 사실상 한도가 수백MB~GB급)에 저장하고, 다이어리 저장 공간에는 `assetId: "3f2a1c..."` 같은 수십 글자짜리 번호 하나만 남는다.

## 쉽게 말하면

예전에는 사진을 보관할 때 사진을 통째로 복사해서 편지지(다이어리 데이터) 안에 접어 넣는 식이었다. 편지지 한 장에 들어가는 용량은 정해져 있어서, 사진을 몇 장만 넣어도 편지지가 꽉 찼다.

이번 step은 그 대신 사진을 큰 창고(IndexedDB)에 원본 그대로 넣어두고, 편지지에는 "창고 3번 칸에 있음"이라는 쪽지만 적는 방식으로 바꿨다. 편지지는 이제 사진 크기와 상관없이 항상 가볍고, 창고는 훨씬 넉넉해서 사진을 많이 넣어도 문제가 없다.

## 만든 파일

- `src/hooks/useCanvasBackground.js` (수정) — `setImage`가 새 창고를 거치도록 교체
- `src/hooks/useFabricCanvas.js` (수정) — 저장 시 `assetId`도 함께 저장되도록 목록에 추가
- `src/components/CanvasBackgroundControl/CanvasBackgroundControl.jsx` (수정) — 타입 설명(주석)만 갱신, 동작 변경 없음
- `src/hooks/useCanvasBackground.test.js` (수정) — 새 저장 방식과 왕복(저장 후 재로드) 검증 테스트 2개 추가

## 제대로 됐는지 어떻게 확인했나?

자동 테스트를 2개 새로 추가해서 기존 7개(총 9개)를 전부 통과시켰다.

1. `setImage(file)`를 호출한 뒤, 캔버스에 추가된 오브젝트가 `isBackground: true`와 문자열 형태의 `assetId`를 둘 다 갖고 있는지, 그 `assetId`로 새 창고에서 원본 파일 이름·내용을 그대로 되찾을 수 있는지 확인.
2. 배경 사진을 등록한 캔버스를 문서(JSON)로 저장한 뒤, 그 문서를 다시 새 캔버스에 불러왔을 때 `isBackground`와 `assetId`가 그대로 복원되고, 사진 링크가 정상적으로 재생성되는지(왕복 테스트) 확인.

`npm run lint`, `npm run build`, `npm run test`(기존 108개 + 새로 만든 2개 = 총 110개)를 모두 실행해서 전부 통과하는 것을 확인했다.

테스트를 만드는 과정에서 이 프로젝트의 테스트 환경(jsdom, 실제 브라우저가 아닌 흉내용 환경) 자체가 "사진을 실제로 화면에 그리는 것"을 지원하지 않는다는 걸 발견했다. 사진을 캔버스에 추가하고 화면을 다시 그리려고 하면 "이미지 로딩이 안 끝났다"는 오류가 났는데, 이는 이번 코드의 버그가 아니라 테스트 도구가 실제 이미지 디코딩 기능(별도 네이티브 패키지)을 갖추고 있지 않아서였다. 그래서 이 테스트 파일 안에서만 "사진이 다 로드된 것처럼" 흉내 내고 "실제로 화면에 그리는" 부분만 건너뛰도록 처리했다. 실제 서비스 코드(브라우저에서 동작하는 진짜 코드)는 전혀 건드리지 않았다 — 테스트 도구의 한계를 우회한 것뿐이다.

## 전체 그림에서 지금 어디?

```
Step 0: 창고 만들기 (완료)
        │
        ▼
Step 1: 기존 데이터 이사시키기 (완료)
        │
        ▼
Step 2: 파일 등록 화면 만들기 (완료, 아직 화면에 연결은 안 됨)
        │
        ▼
Step 3: 업로드한 폰트를 실제로 쓸 수 있게 등록 (완료)
        │
        ▼
[Step 4: 배경 이미지 설정을 새 창고에 연결] ← 지금 여기, 완료
        │
        ▼
Step 5: Ctrl+V로 이미지 붙여넣기
```

이 step을 4번째로 한 이유는, 새 창고(Step 0)와 기존 데이터 이사(Step 1)가 먼저 끝나 있어야 "새로 등록하는 사진"도 안전하게 같은 방식으로 저장할 수 있기 때문이다. 이제 배경 이미지는 "예전 것(이사됨)"이든 "새로 등록한 것"이든 전부 같은 창고, 같은 보관번호 방식으로 다뤄진다.

## 아직 안 된 것 (솔직하게)

- 배경 이미지 등록 버튼은 여전히 "파일 하나 선택" 방식 그대로다. Step 2에서 만든 "파일 여러 개 선택/폴더 선택/드래그앤드롭" 공용 입력 화면(`AssetImportPanel`)은 이번 step에서 배경 이미지 설정에 연결하지 않았다 — PRD에서 이번 step에 요구한 범위는 "저장 방식 전환"이었고, 입력 경로를 다양하게 넓히는 것은 별도 통합 작업이다.
- 클립보드에 사진을 복사해두고 Ctrl+V로 붙여넣는 기능은 아직 없다 (Step 5 몫).
- 사진에 필터를 걸거나 편집하는 기능은 이번 phase 범위 밖이라 손대지 않았다.
