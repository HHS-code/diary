# 빌드 리포트 — diary (asset-import-pipeline)

각 step이 무엇을 했는지 위에서 아래로 읽으면 프로젝트가 자라는 과정이 보인다.

- [완료] **Step 0: asset-storage** — src/storage/assetStorage.js: saveAsset({type,filename,mimeType,blob,fontFamily?}) -> Promise<string id>, getAsset(id) -> Promise<AssetRecord|null>, listAssets(type) -> Promise<AssetRecord[]> (createdAt asc), deleteAsset(id) -> Promise<void>, createAssetObjectURL(blob) -> string. IndexedDB "diary-assets"/store "assets"/keyPath "id", 외부 라이브러리(idb) 없이 네이티브 indexedDB 직접 사용. 트랜잭션마다 connection을 열고 oncomplete/onerror에서 반드시 db.close() — 안 닫으면 다음 deleteDatabase/open이 블로킹됨(테스트에서 실제로 걸림). 테스트는 devDependency fake-indexeddb(6.2.5)로 jsdom에 IndexedDB polyfill; jsdom Blob과 Node 내장 structuredClone 간 비호환으로 테스트에서만 node:buffer의 Blob을 사용(프로덕션 코드/실브라우저는 영향 없음). 다음 step(asset-migration)은 이 5개 함수만 import해서 쓰면 됨. ([리포트](step0-asset-storage.md))
- [완료] **Step 1: asset-migration** — src/storage/assetMigration.js: runAssetMigration() -> Promise<void>. localStorage['diary-asset-migration-done']==='true'면 즉시 반환(멱등). diaryStorage.loadAllDiaryData()의 모든 dateKey×{diary,movie}.canvasJSON.objects를 순회, src가 'data:'로 시작하는 오브젝트만 fetch(dataUrl)->blob->assetStorage.saveAsset({type:'image',...})로 이관 후 src 제거+assetId 추가, saveAllDiaryData로 재저장. 오브젝트 단위 실패는 원본 유지하고 계속 진행(부분 성공, 롤백 없음). src/hooks/useFabricCanvas.js에 resolveCanvasAssetReferences(canvasJSON) -> Promise<object> 추가 export — loadFromJSON 직전에 objects[].assetId를 assetStorage.getAsset+createAssetObjectURL로 src에 역참조 채움(assetId 없거나 조회 실패 시 원본 그대로 통과). App.jsx 최상위 useEffect(fire-and-forget)에서 runAssetMigration() 호출. 테스트에서 fetch/Blob이 만드는 jsdom Blob은 fake-indexeddb structuredClone과 비호환이라(step0과 동일 이슈) assetMigration.test.js는 vi.stubGlobal('fetch', ...)로 node:buffer Blob을 반환하도록 테스트에서만 우회 — 프로덕션 fetch 호출 자체는 스펙 그대로. ([리포트](step1-asset-migration.md))
- [완료] **Step 2: asset-import-panel** — src/hooks/useAssetLibrary.js: useAssetLibrary() -> { images, fonts, registerImage(file)->Promise<id>, registerFont(file)->Promise<id>, refresh() }, assetStorage.listAssets/saveAsset 위에 상태 래핑. classifyAssetType(filename) -> 'image'|'font'|null 공용 export(확장자: jpg/jpeg/png/gif/webp → image, ttf/otf/woff/woff2 → font). registerFont는 파일명(확장자 제외)을 fontFamily로 쓰고 fonts 목록과 이름 충돌 시 crypto.randomUUID() 앞 4자리 suffix를 붙임. src/components/AssetImportPanel/AssetImportPanel.jsx: props { library } (훅을 내부에서 호출하지 않고 주입받음) — 파일 선택 input, 폴더 선택 input(webkitdirectory), 드래그앤드롭 3경로 모두 classifyAssetType으로 분류 후 library.registerImage/registerFont 호출, 미지원 확장자는 조용히 skip. 등록된 이미지/폰트 파일명만 나열(삭제/편집 UI 없음, PRD out-of-scope). 폴더 드롭 시 webkitGetAsEntry 재귀 탐색은 구현하지 않음(ADR 합의). 스타일은 StickerPalette 톤(XP 스타일 패널)을 그대로 재사용. npm run lint/build/test 전체 통과 확인(신규 테스트 6개 포함 109개) — test 스위트를 여러 번 돌리는 동안 useCanvasHistory.test.js/useCanvasBackground.test.js가 각각 다른 실행에서 5000ms 타임아웃으로 1개씩 실패했으나 단독 실행 시 통과하고 이번 step이 건드리지 않은 파일이라 시스템 부하로 인한 기존 플레이키니스로 판단, 이번 step 코드와 무관. 다음 step(font-registry)은 useAssetLibrary의 fonts 배열과 fontFamily 필드, AssetImportPanel을 그대로 재사용하면 됨. ([리포트](step2-asset-import-panel.md))
- [완료] **Step 3: font-registry** — src/hooks/useFontRegistry.js: useFontRegistry(fontAssets?) -> { customFonts: {label,value}[] }. 인자 생략 시 마운트 시 1회 assetStorage.listAssets('font')로 자체 조회, 인자(예: useAssetLibrary().fonts)를 넘기면 그 배열 참조가 바뀔 때마다 재로드 — 두 훅이 상태를 공유하려는 미래 통합 지점을 열어둠(현재는 아무 부모도 이렇게 연결하지 않음, TextMemoButton은 인자 없이 자체 호출). 각 폰트 레코드는 createAssetObjectURL(blob)+new FontFace(fontFamily, url)+.load()+document.fonts.add()로 등록하되 document.fonts를 순회해 동일 family면 재등록 skip. FontFace.load() 실패(손상된/디코딩 불가 폰트)는 assetMigration.js와 동일한 부분 성공 패턴으로 해당 폰트만 건너뛰고 나머지는 계속 등록(원래 구현엔 이 예외처리가 없어 폰트 하나만 깨져도 전체 customFonts가 안 나타나는 버그가 있었음 — 브라우저 실측으로 발견해 수정). src/components/TextMemoButton/TextMemoButton.jsx: 기존 FONTS 상수 유지, [...FONTS, ...useFontRegistry().customFonts]를 드롭다운에 렌더링(그룹 분리 없음, ADR-3). 부수 효과로 src/components/buttonIcons.test.jsx가 TextMemoButton을 마운트하면서 간접적으로 IndexedDB를 타게 돼 'fake-indexeddb/auto' import를 추가해야 했음(기존 다른 테스트 파일들과 동일 컨벤션). npm run lint/build/test 전체 통과(108개, useCanvasHistory.test.js가 다른 회차에서 시스템 부하로 1회 타임아웃 — 단독 실행 시 통과, 이번 step과 무관한 기존 플레이키니스). AssetImportPanel은 아직 DiaryCanvas/앱 어디에도 마운트되지 않은 상태(이전 step도 동일) — 실사용 등록 경로는 다음 step들(4: canvas-background-integration, 5: clipboard-paste-priority)이나 이후 화면 통합에서 연결될 것으로 보임. ([리포트](step3-font-registry.md))
- [완료] **Step 4: canvas-background-integration** — src/hooks/useCanvasBackground.js: setImage(file)를 async로 교체 — FileReader/base64 대신 saveAsset({type:'image',...,blob:file})로 IndexedDB에 저장 후 createAssetObjectURL(file)로 objectURL을 만들어 FabricImage.fromURL에 넘김. 생성된 배경 오브젝트에 기존 isBackground:true에 더해 assetId를 함께 set. src/hooks/useFabricCanvas.js의 EXTRA_SERIALIZED_PROPS에 'assetId' 추가해 toObject()/toJSON() 직렬화에 포함되도록 함 — 로드 시 역참조(resolveCanvasAssetReferences)는 step1에서 이미 구현돼 있어 재사용만 함, 별도 분기 추가 없음. setColor/lockBackground/clearBackground는 미변경. 테스트(useCanvasBackground.test.js)에서 fake-indexeddb/auto + node:buffer 기반 FakeImageFile(File 대신, jsdom File/Blob이 structuredClone과 비호환이라 step0과 동일 이유로 우회) 사용. jsdom은 <img>.src를 설정해도 실제 로드를 안 해 FabricImage.fromURL이 영원히 대기하고, node-canvas 네이티브 바인딩이 없어 실제 drawImage도 'Image given has not completed loading'로 실패하므로, 이 테스트 파일 안에서만 HTMLImageElement.prototype.src setter를 즉시 onload 발화하도록, CanvasRenderingContext2D.prototype.drawImage를 no-op으로 스텁(다른 테스트 파일에는 영향 없음, jsdom 환경은 파일별로 새로 생성됨) — 이 우회 없이는 이미지 배경을 다루는 어떤 fabric 테스트도 이 프로젝트에서 불가능하다는 것을 발견. npm run lint/build/test 전체 통과(110개, 신규 2개 포함). AssetImportPanel/useAssetLibrary(step2)는 여전히 이 화면에 마운트되지 않음 — CanvasBackgroundControl은 기존 <input type=file single> 그대로이고, 다중/폴더/드래그드롭 경로 연결은 이 step 범위 밖(PRD 성공기준의 배경 이미지 파이프라인 전환만 담당). ([리포트](step4-canvas-background-integration.md))
- [완료] **Step 5: clipboard-paste-priority** — src/hooks/useCanvasKeyboardShortcuts.js: 시그니처를 useCanvasKeyboardShortcuts(fabricCanvasRef, { registerImage } = {})로 확장. Ctrl+V 시 navigator.clipboard.read()로 clipboard item을 확인해 image/* 타입이 있으면 File로 감싸 registerImage(file)만 호출(캔버스 삽입 없음, architecture 섹션6 경계 유지), 이미지가 없거나 clipboard API 미지원/권한거부로 예외 발생 시 조용히 기존 pasteFromClipboard(오브젝트 clone)로 폴백. registerImage 미전달(undefined) 시에도 기존 동작 그대로 동작 — 하위호환 유지. src/components/DiaryCanvas/DiaryCanvas.jsx: useAssetLibrary()를 호출해 registerImage를 useCanvasKeyboardShortcuts에 전달하도록 연결(다른 통합은 하지 않음). npm run lint/build/test 전체 통과(113개, 신규 3개 포함, 이번 회차엔 플레이키니스 없었음). AssetImportPanel 자체는 여전히 화면 어디에도 마운트되지 않은 상태 — PRD 성공기준 중 '파일선택/폴더선택/드래그앤드롭' 세 경로의 UI 연결은 이번 phase의 다른 어떤 step에도 배정되지 않았음(step0~5 전부 완료 시점 기준, UI 마운트는 후속 논의 필요). ([리포트](step5-clipboard-paste-priority.md))
