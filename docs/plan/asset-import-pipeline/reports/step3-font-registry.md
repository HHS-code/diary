# Step 3: font-registry

## 이 단계를 왜 했나?

Step 2에서 만든 "접수대"(`AssetImportPanel`)는 사용자가 폰트 파일(.ttf, .otf 등)을 창고(IndexedDB)에 저장하는 데까지만 했다. 하지만 창고에 파일이 저장돼 있다고 브라우저가 그 폰트를 화면에 바로 써주는 건 아니다 — 파일을 실제로 읽어서 "이 이름의 폰트가 있다"고 브라우저에 등록하는 과정이 따로 필요하다. 이번 단계는 그 등록 과정을 만들고, 지금까지 구글 폰트 8종만 고를 수 있던 텍스트 메모의 폰트 목록에 업로드한 폰트도 함께 뜨도록 연결했다.

## 무엇을 만들었나?

**`useFontRegistry` (폰트 통역사)** — 창고에 저장된 폰트 파일들을 하나씩 꺼내 브라우저가 알아듣는 형태로 "통역"해주는 부품이다. 창고의 폰트 레코드에는 파일 원본(Blob)과 이름(`fontFamily`, 예: `MyFont`)이 들어있는데, 이 부품이 그 파일을 브라우저의 `FontFace`라는 표준 기능에 건네주고 브라우저 내부 폰트 목록(`document.fonts`)에 등록한다. 등록이 끝나면 텍스트 메모 화면이 그대로 쓸 수 있는 `{이름, 값}` 형태의 목록(`customFonts`)을 돌려준다.

같은 폰트를 두 번 등록하지 않도록, 등록 전에 "이미 브라우저에 같은 이름이 있나?"를 먼저 확인한다. 그리고 파일이 손상됐거나 폰트 형식이 아닌 걸 잘못 올린 경우(브라우저가 못 읽는 경우) 그 파일 하나만 건너뛰고 나머지 정상 폰트는 그대로 목록에 올라간다 — Step 1(마이그레이션)에서 이미 쓴 "하나 실패해도 나머지는 계속 진행" 방식을 그대로 따랐다.

**`TextMemoButton` 수정** — 기존에 하드코딩돼 있던 구글 폰트 8종 목록은 그대로 두고, 그 뒤에 `useFontRegistry`가 돌려준 업로드 폰트 목록을 이어붙였다. "기본 폰트"/"내 폰트"처럼 구역을 나누지 않고 하나의 드롭다운에 전부 나열한다(이미 합의된 방식).

## 데이터가 어떻게 흐르나?

```
앱이 텍스트 메모 화면을 열 때
        │
        ▼
useFontRegistry가 창고(assetStorage)에서 폰트 레코드 전부를 가져옴
        │
        ▼
  레코드마다: 이미 브라우저에 같은 이름 폰트가 있나?
        │
   있음 ─┤                          없음
        │                            │
   등록 건너뜀                  파일(Blob) → FontFace로 변환 시도
        │                            │
        │                     성공 ──┤── 실패(손상된 파일 등)
        │                       │                │
        │                브라우저에 등록      이 폰트만 건너뜀
        │                       │                │
        └───────────────────────┴────────────────┘
                        │
                        ▼
          customFonts = [{이름, 값}, ...] 완성
                        │
                        ▼
       TextMemoButton 드롭다운 = 구글 폰트 8종 + customFonts
```

예를 들어 사용자가 `MyFont.ttf`(정상)와 `Broken.ttf`(어쩌다 손상된 파일)를 둘 다 창고에 올려둔 상태로 텍스트 메모 화면을 열면: 드롭다운에는 기존 8종 뒤에 `MyFont` 하나만 추가로 나타나고, `Broken`은 조용히 빠진다. 화면이 멈추거나 나머지 8종까지 같이 사라지는 일은 없다.

## 쉽게 말하면

창고에 폰트 파일을 넣어둔 것과, 그 폰트로 실제로 글씨를 쓸 수 있는 것은 다르다 — 마치 종이에 인쇄된 손글씨 샘플을 창고에 보관해 둔 것과, 그 손글씨체를 워드프로세서의 "글꼴" 목록에 실제로 추가해 선택할 수 있게 만드는 것의 차이와 같다. `useFontRegistry`는 창고에서 샘플을 꺼내 워드프로세서(브라우저)의 글꼴 목록에 등록해주는 담당자다. 샘플이 찢어져 있으면(손상된 파일) 그 샘플만 빼고 등록을 계속 진행한다.

## 만든 파일

- `src/hooks/useFontRegistry.js` — 폰트 레코드를 `FontFace`로 등록하고 `customFonts` 목록을 돌려주는 훅
- `src/hooks/useFontRegistry.test.jsx` — 등록/중복 방지/손상된 폰트 건너뛰기/외부 목록 주입 동작을 확인하는 자동 테스트
- `src/components/TextMemoButton/TextMemoButton.jsx` (수정) — 드롭다운에 `customFonts`를 이어붙임
- `src/components/TextMemoButton/TextMemoButton.test.jsx` — 커스텀 폰트가 드롭다운에 기존 8종과 함께 나타나는지 확인하는 자동 테스트
- `src/components/buttonIcons.test.jsx` (수정) — `TextMemoButton`이 이제 창고(IndexedDB)를 들여다보게 되면서, 이 테스트 파일에도 창고를 흉내 내는 준비물(`fake-indexeddb/auto`)을 추가해야 기존 테스트가 계속 정상 동작함

## 제대로 됐는지 어떻게 확인했나?

자동 테스트를 새로 5개 작성해서 전부 통과시켰다 (`useFontRegistry.test.jsx` 4개, `TextMemoButton.test.jsx` 1개): 등록된 폰트가 브라우저 폰트 목록에 추가되고 `customFonts`에 올바른 형태로 나타나는지, 이미 등록된 이름은 중복 등록하지 않는지, 폰트 하나가 등록에 실패해도 나머지는 정상 등록되는지, 외부에서 목록을 주입하면 자체 조회 없이 그 목록을 쓰는지, 드롭다운에 기존 8종+커스텀 폰트가 모두 나타나는지.

`npm run lint`, `npm run build`, `npm run test`를 모두 통과시켰다(총 108개, 전부 통과). 테스트 작성 중 실제로 버그 하나를 발견해 고쳤다 — 처음 구현에서는 폰트 하나의 등록이 실패하면(파일 손상 등) 예외가 그대로 튀어 올라가 그 전에 정상적으로 처리된 다른 폰트들까지 전부 목록에서 사라지는 문제가 있었다. 이건 브라우저를 직접 띄워 가짜 폰트 파일을 창고에 넣고 확인하는 과정에서 실제로 재현해 확인했고, 실패한 폰트만 건너뛰도록 고쳐 재검증했다.

또한 브라우저에서 직접 확인하는 과정에서, 기존 테스트 파일(`buttonIcons.test.jsx`)이 `TextMemoButton`을 화면에 띄우다가 창고 접근 오류로 조용히 깨지는 것을 발견해 함께 고쳤다(이번 step의 변경이 만든 부작용이라 함께 수정하는 게 맞다고 판단).

여러 번 전체 테스트를 돌리는 동안 `useCanvasHistory.test.js`가 한 번 5초 제한 시간 초과로 실패했으나, 따로 떼어 실행하면 바로 통과했다 — 이전 step들 보고서에도 기록된, 여러 무거운 테스트가 동시에 도는 상황에서 컴퓨터가 잠깐 느려져 생기는 우연한 현상이며 이번 step 코드와 무관하다.

## 전체 그림에서 지금 어디?

```
Step 0: 창고 만들기 (완료)
        │
        ▼
Step 1: 기존 데이터 이사시키기 (완료)
        │
        ▼
Step 2: 파일 등록 화면 만들기 (완료)
        │
        ▼
[Step 3: 업로드한 폰트를 실제로 쓸 수 있게 등록] ← 지금 여기, 완료
        │
        ▼
Step 4: 배경 이미지 설정을 새 창고에 연결
        │
        ▼
Step 5: Ctrl+V로 이미지 붙여넣기
```

폰트 등록을 배경 이미지 연결(Step 4)보다 먼저 한 이유는 특별히 없다 — 애초에 계획된 순서였고, 폰트 쪽이 이미지 쪽보다 다루는 대상(폰트 파일 하나, 관련 로직도 `useAssetLibrary`에 이미 있던 `fonts` 목록 재사용)이 더 단순해 먼저 끝내기 좋았다.

## 아직 안 된 것 (솔직하게)

- `AssetImportPanel`(접수대 화면)은 여전히 앱 화면 어디에도 붙어 있지 않다(Step 2 때부터 이어진 상태). 그래서 지금 실제로 사용자가 폰트 파일을 창고에 넣을 방법은 아직 화면상으로는 없고, 이번 step의 검증도 창고에 직접 데이터를 넣어 확인하는 방식으로 했다. 접수대를 실제 화면에 연결하는 건 이후 step들의 몫으로 보인다.
- `useFontRegistry`와 `useAssetLibrary`가 폰트 목록 배열을 실시간으로 공유하도록 설계상 여지는 만들어 뒀지만(같은 부모가 두 훅을 함께 써서 연결하는 방식), 지금은 그 부모가 없어서 실제로 연결돼 있지는 않다. 현재는 텍스트 메모 화면을 열 때(또는 새로고침할 때)마다 창고를 다시 조회하는 방식으로 동작한다 — 화면을 안 닫고 다른 곳에서 폰트를 새로 등록해도 이미 열려 있는 드롭다운이 즉시 갱신되지는 않는다.
- 폰트 서브셋팅이나 미리 불러오기 같은 최적화는 하지 않았다(이번 phase 범위 밖으로 미리 정해둠).
