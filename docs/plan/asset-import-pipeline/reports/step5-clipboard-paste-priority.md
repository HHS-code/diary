# Step 5: clipboard-paste-priority

## 이 단계를 왜 했나?

이 다이어리 편집 화면에는 원래부터 Ctrl+V 단축키가 있었다. 오브젝트(스티커, 텍스트 등)를 하나 복사(Ctrl+C)해두고 Ctrl+V를 누르면, 그 오브젝트가 살짝 옆으로 옮겨진 채 복제되어 하나 더 생기는 기능이다.

이번 phase는 여기에 새로운 기능을 얹으려 한다: 예를 들어 웹에서 사진을 우클릭해 "이미지 복사"를 하거나 캡처 도구로 화면을 캡처하면, 그 사진이 시스템 클립보드(운영체제가 관리하는 복사/붙여넣기 임시 저장소)에 들어간다. 이걸 다이어리 화면에서 Ctrl+V로 바로 붙여넣을 수 있게 하는 것이 목표다.

문제는 Ctrl+V라는 같은 키 조합을 두 기능이 나눠 써야 한다는 점이다. 클립보드에 사진이 있으면 "사진 등록"으로, 없으면 기존 "오브젝트 복제"로 동작해야 사용자가 헷갈리지 않는다. 이 단계는 그 우선순위 판단 로직을 만드는 작업이다.

## 무엇을 만들었나?

`src/hooks/useCanvasKeyboardShortcuts.js`의 Ctrl+V 처리 부분을 다음 순서로 바꿨다.

1. Ctrl+V를 누르면, 먼저 브라우저에게 "지금 클립보드에 사진 파일이 들어있어?"라고 물어본다(`navigator.clipboard.read()`라는 브라우저 기능 사용).
2. 사진이 있다면: 그 사진을 Step 2에서 만들어둔 "에셋 등록" 창구(`registerImage`)로 넘겨서 새 창고(IndexedDB)에 저장한다. 이때 캔버스에 사진을 자동으로 붙여넣지는 않는다 — "등록"까지만 하고, 실제로 화면에 붙이는 건 사용자가 배경 설정 등 다른 버튼을 눌러 직접 고르도록 남겨둔다(이 phase의 문서 합의에 정해진 경계).
3. 사진이 없다면: 예전부터 있던 오브젝트 복제 기능을 그대로 실행한다.
4. 애초에 브라우저가 클립보드 조회 기능을 지원하지 않거나, 사용자가 "클립보드 접근 허용"을 거부해서 오류가 나면, 그 오류를 사용자에게 보여주지 않고 조용히 3번(기존 오브젝트 복제)으로 넘어간다.

텍스트를 편집하는 중에 Ctrl+V를 누르는 경우(글자를 붙여넣으려는 의도)는 기존과 동일하게 아무 동작도 하지 않는다 — 이 부분은 건드리지 않았다.

이 기능이 동작하려면 "사진 등록" 창구(`registerImage`)를 사용할 수 있어야 하는데, 지금까지는 어느 화면도 그 창구를 실제로 열어두지 않았다(Step 2~4 보고서에 계속 언급됐던 부분). 그래서 `src/components/DiaryCanvas/DiaryCanvas.jsx`(편집 화면 전체를 담당하는 컴포넌트)에서 그 창구를 열어(`useAssetLibrary()`) Ctrl+V 처리 로직에 연결했다.

## 데이터가 어떻게 흐르나?

```
[사용자가 Ctrl+V를 누름]
   │
   ▼
텍스트를 편집 중인가? ──예──▶ 아무 동작 안 함 (기존과 동일)
   │ 아니오
   ▼
브라우저에 "클립보드에 사진 있어?" 확인 요청
   │
   ├─ 확인 자체가 실패함(미지원/권한거부) ──▶ [기존 오브젝트 복제로 폴백]
   │
   ├─ 사진이 없음 ──▶ [기존 오브젝트 복제로 폴백]
   │
   └─ 사진이 있음
        │
        ▼
     그 사진을 File로 감싸서 registerImage(file) 호출
        │
        ▼
     새 창고(IndexedDB)에 사진 저장, 보관번호(assetId) 발급
        │
        ▼
     (캔버스에는 아무것도 추가하지 않음 — 등록만 하고 끝)
```

숫자로 예를 들면: 사용자가 웹 브라우저에서 사진 한 장을 "이미지 복사"한 뒤 다이어리 화면 아무 곳에나 커서를 두고 Ctrl+V를 누르면, 새 창고에 사진 1장이 등록되고 보관번호(예: `"9c1a-..."` 같은 문자열) 하나가 발급된다. 반대로 캔버스 위 스티커를 Ctrl+C로 복사해둔 상태에서 Ctrl+V를 누르면(클립보드에 사진이 없으므로) 그 스티커가 오른쪽 아래로 10px씩 밀려난 채 복제되는, 예전과 완전히 동일한 동작이 일어난다.

## 쉽게 말하면

Ctrl+V는 원래 "복사해둔 스티커를 하나 더 만드는" 버튼이었다. 이번 step은 그 버튼에 "먼저 주머니(클립보드) 속에 사진이 있는지 확인하고, 있으면 그 사진을 창고에 넣어두고, 없으면 원래 하던 대로 스티커를 복제하라"는 규칙을 추가한 것이다. 주머니를 확인하다가 손이 묶여 있거나(권한 거부) 주머니 자체가 없는 사람(미지원 브라우저)이면, 그냥 원래 하던 동작을 한다.

## 만든 파일

- `src/hooks/useCanvasKeyboardShortcuts.js` (수정) — Ctrl+V 처리에 클립보드 사진 우선 확인 로직 추가, `registerImage`를 인자로 받도록 시그니처 확장
- `src/components/DiaryCanvas/DiaryCanvas.jsx` (수정) — `useAssetLibrary()`를 열어 `registerImage`를 위 훅에 연결
- `src/hooks/useCanvasKeyboardShortcuts.test.jsx` (수정) — 클립보드에 사진이 있을 때/없을 때/조회 자체가 실패할 때 세 가지 상황을 검증하는 테스트 3개 추가

## 제대로 됐는지 어떻게 확인했나?

자동 테스트를 3개 새로 추가해서 기존 4개(총 7개, 이 파일 기준)를 전부 통과시켰다.

1. 클립보드에 사진이 있는 상황을 흉내 내고 Ctrl+V를 누르면, "사진 등록" 함수가 정확히 1번 호출되고, 기존 오브젝트 복제는 일어나지 않는지(캔버스 오브젝트 개수가 그대로인지) 확인.
2. 클립보드에 사진이 없는 상황(빈 목록)을 흉내 내면, 기존과 동일하게 오브젝트가 복제되는지(개수가 1개→2개로 늘어나는지) 확인 — 기존 동작이 망가지지 않았다는 회귀 확인.
3. 클립보드 조회 자체가 오류를 던지는 상황(권한 거부를 흉내 냄)을 만들면, 오류가 사용자에게 노출되지 않고 조용히 기존 오브젝트 복제로 넘어가는지 확인.

`npm run lint`, `npm run build`, `npm run test`(기존 110개 + 새로 만든 3개 = 총 113개)를 모두 실행해서 전부 통과하는 것을 확인했다. 이번 회차에는 이전 step 보고서들에서 언급됐던 "시스템 부하로 인한 타임아웃" 같은 우연한 실패도 없었다.

## 전체 그림에서 지금 어디?

```
Step 0: 창고 만들기 (완료)
        │
        ▼
Step 1: 기존 데이터 이사시키기 (완료)
        │
        ▼
Step 2: 파일 등록 화면 만들기 (완료, 화면엔 아직 안 붙음)
        │
        ▼
Step 3: 업로드한 폰트를 실제로 쓸 수 있게 등록 (완료)
        │
        ▼
Step 4: 배경 이미지 설정을 새 창고에 연결 (완료)
        │
        ▼
[Step 5: Ctrl+V로 클립보드 사진을 새 창고에 등록] ← 지금 여기, 완료
```

이번 phase(에셋 입력 파이프라인)로 계획된 6개 step(0~5)이 전부 끝났다. 이 step을 마지막에 배치한 이유는, "사진을 새 창고에 등록하는 창구"(`registerImage`, Step 2 산출물)가 먼저 만들어져 있어야 Ctrl+V가 그 창구로 사진을 흘려보낼 수 있었기 때문이다.

## 아직 안 된 것 (솔직하게)

- Ctrl+V로 등록한 사진은 창고에 저장만 될 뿐, 화면(캔버스)에 자동으로 나타나지 않는다. 사용자가 그 사진을 실제로 쓰려면 "배경 이미지 설정" 같은 다른 버튼을 눌러 직접 골라야 한다. 이건 실수로 빠뜨린 게 아니라, 이번 phase의 설계 문서에서 "등록까지만 하고, 캔버스에 넣는 건 각 기능(배경 설정 등)의 몫으로 남긴다"고 미리 정해둔 경계다.
- Step 2에서 만든 "파일 여러 개 선택 / 폴더 선택 / 드래그앤드롭" 공용 입력 화면(`AssetImportPanel`)은 이번 phase의 어느 step에서도 실제 화면에 연결되지 않았다. 지금까지 실제로 동작하는 입력 경로는 배경 이미지의 "파일 하나 선택"(Step 4)과 이번 Ctrl+V(Step 5) 두 가지뿐이다. 파일 여러 개를 한 번에 올리거나 폴더째로 올리거나 드래그앤드롭으로 올리는 기능은 코드로는 존재하지만(`AssetImportPanel.jsx`), 사용자가 실제로 그 화면을 볼 방법이 아직 없다 — 이건 이번 phase의 계획된 6개 step 어디에도 "화면에 붙이기" 작업이 배정돼 있지 않았던 것으로 보이며, 후속 논의가 필요하다.
