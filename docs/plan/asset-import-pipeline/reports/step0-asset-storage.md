# Step 0: asset-storage

## 이 단계를 왜 했나?

지금까지 배경 이미지는 파일을 골라서 넣으면 그 이미지를 통째로 긴 문자열(base64)로 바꿔서, 다이어리 데이터 전체가 들어 있는 브라우저 저장공간 한 칸(localStorage)에 같이 쑤셔 넣는 방식이었다. 그런데 이 저장공간은 브라우저마다 보통 5~10MB로 한도가 정해져 있고, 문자열로 바꾸는 과정에서 원본보다 약 1.33배 커진다. 이번 phase는 이미지와 폰트 파일을 여러 개 등록하는 게 목표라서, 이 방식으로는 몇 개만 넣어도 한도에 부딪힌다.

그래서 이번 step에서는 "파일을 저장하는 창고" 자체를 새로 짓는다. 이후 step(마이그레이션, 파일 등록 UI, 폰트 적용, 배경 이미지 연결)이 전부 이 창고를 갖다 쓰게 된다. 이번 step은 그 창고와 창고 문(입고/조회/목록/폐기) 네 가지 동작만 만드는 것이고, 아직 어디에도 연결하지 않는다.

## 무엇을 만들었나?

`src/storage/assetStorage.js` 파일 하나를 새로 만들었다. 브라우저가 제공하는 IndexedDB라는 저장 기술을 사용한다. IndexedDB는 localStorage와 달리 한도가 사실상 수백MB~GB 단위라, 이미지·폰트 파일을 원본 그대로(문자열로 바꾸지 않고) 저장할 수 있다.

이 파일이 제공하는 기능은 다섯 가지다.

- 저장하기: 이미지나 폰트 파일 하나를 넣으면 고유 번호(id)를 발급해서 저장하고, 그 번호를 돌려준다.
- 조회하기: 번호로 파일 하나를 다시 꺼낸다. 없으면 "없음"을 알려준다.
- 목록 보기: "이미지만" 또는 "폰트만" 골라서 등록된 순서대로 전부 꺼낸다.
- 지우기: 번호로 파일 하나를 창고에서 없앤다. (이번 phase의 화면에는 "지우기 버튼"이 없지만, 창고 기능 자체는 나중을 위해 미리 만들어 둔다.)
- 임시 링크 만들기: 저장된 파일 데이터를 화면에 실제로 보여주려면 브라우저용 임시 링크가 하나 필요한데, 그걸 만들어 주는 작은 도우미 함수.

## 데이터가 어떻게 흐르나?

```
[사용자가 고른 이미지/폰트 파일]
        │
        ▼
  saveAsset({ type, filename, mimeType, blob })
        │  (내부에서 고유 번호 발급)
        ▼
  IndexedDB "diary-assets" 창고
  ┌─────────────────────────────────────┐
  │ id: "a1b2..."                        │
  │ type: "image"                        │
  │ filename: "고양이.png"                │
  │ mimeType: "image/png"                │
  │ blob: <원본 파일 그대로>              │
  │ createdAt: 1752800000000             │
  └─────────────────────────────────────┘
        │
        ▼ (나중에 화면에서 다시 필요할 때)
  getAsset("a1b2...")  →  파일 데이터 다시 꺼냄
  createAssetObjectURL(blob)  →  <img src="..."> 에 넣을 임시 링크
```

숫자로 예를 들면, 5MB짜리 사진 하나를 기존 방식(base64)으로 저장하면 문자열 크기가 약 6.65MB로 불어나고, localStorage 한도(5~10MB)를 혼자서 거의 다 써버린다. 새 방식(IndexedDB)으로 저장하면 5MB 그대로 저장되고, 이런 사진을 수십 장 넣어도 문제없다.

## 쉽게 말하면

기존 방식은 사진을 편지 봉투(localStorage, 용량 작음) 안에 구겨 넣는 것과 비슷했다. 이번에 만든 건 사진을 그대로 넣을 수 있는 큰 서랍장(IndexedDB)을 새로 들여놓은 것이다. 서랍마다 이름표(고유 번호)를 붙여서, 나중에 이름표만 알면 바로 꺼낼 수 있게 했다. 아직은 서랍장만 놓았을 뿐, 방(캔버스 배경, 폰트 목록 등)에 실제로 연결하는 건 다음 단계들이다.

## 만든 파일

- `src/storage/assetStorage.js` — 창고 본체 (저장/조회/목록/삭제/임시링크 함수)
- `src/storage/assetStorage.test.js` — 위 기능이 실제로 되는지 확인하는 자동 테스트
- `package.json` — 테스트 환경에서 IndexedDB를 흉내 내주는 `fake-indexeddb` 패키지 추가 (프로덕션 코드에는 영향 없음, 테스트할 때만 쓰임)

## 제대로 됐는지 어떻게 확인했나?

자동 테스트 4개를 새로 작성해서 전부 통과시켰다.

1. 이미지 파일을 저장한 뒤 번호로 다시 꺼내면, 저장했던 파일 내용·이름·형식이 그대로 나오는지.
2. 없는 번호로 조회하면 "없음"이 나오는지.
3. "이미지만" 목록을 보면 이미지만, "폰트만" 목록을 보면 폰트만 나오는지 (섞이지 않는지).
4. 지운 파일을 다시 조회하면 "없음"이 나오는지.

그리고 이번 phase 전체 완료 기준인 `npm run lint`(코드 스타일 검사), `npm run build`(실제 배포 빌드), `npm run test`(전체 자동 테스트, 새로 만든 4개 포함 91개)를 모두 실행해서 전부 통과하는 것을 확인했다.

테스트를 만드는 과정에서 한 가지 함정을 발견해서 고쳤다: 테스트 환경(jsdom)에서 파일 데이터를 저장했다가 바로 조회하면, 창고 연결을 계속 열어둔 채로 놔뒀을 때 다음 테스트가 멈춰버리는 문제가 있었다. 저장/조회 작업이 끝나면 반드시 연결을 닫도록 고쳐서 해결했다.

## 전체 그림에서 지금 어디?

```
[Step 0: 창고 만들기] ← 지금 여기, 완료
        │
        ▼
Step 1: 기존 데이터 이사시키기 (localStorage → 새 창고)
        │
        ▼
Step 2: 파일 등록 화면 만들기 (드래그앤드롭, 폴더 선택 등)
        │
        ▼
Step 3: 업로드한 폰트를 실제로 쓸 수 있게 등록
        │
        ▼
Step 4: 배경 이미지 설정을 새 창고에 연결
        │
        ▼
Step 5: Ctrl+V로 이미지 붙여넣기
```

창고부터 먼저 만든 이유는, 다음 단계(기존 데이터 이사, 파일 등록 화면, 배경 이미지 연결 등)가 전부 "어딘가에 저장한다"는 동작을 필요로 하기 때문이다. 창고 없이 이사나 화면부터 만들면 나중에 다시 뜯어고쳐야 한다.

## 아직 안 된 것 (솔직하게)

- 기존에 localStorage에 있던 사진 데이터는 아직 이 새 창고로 옮겨지지 않았다 (다음 step 몫).
- 사용자가 파일을 실제로 선택해서 등록할 수 있는 화면이 아직 없다 (다음다음 step 몫).
- 지금 만든 "지우기" 기능은 창고 안에는 있지만, 화면에서 누를 수 있는 버튼은 이번 phase 범위 밖이라 만들지 않았다.
- 이미지 용량을 줄이거나(압축), 파일이 너무 클 때 경고하는 기능은 이번 phase 범위에 들어있지 않아 만들지 않았다.
