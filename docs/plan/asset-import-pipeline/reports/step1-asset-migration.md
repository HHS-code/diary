# Step 1: asset-migration

## 이 단계를 왜 했나?

Step 0에서 이미지·폰트를 원본 그대로 저장하는 새 창고(IndexedDB)를 만들었다. 하지만 사용자가 지금까지 실제로 써온 다이어리에는 이미 기존 방식(base64 문자열)으로 저장된 배경 이미지가 들어 있다. 새 창고를 만들었다고 해서 예전 방식으로 저장된 데이터가 저절로 옮겨지는 건 아니다. 아무 조치 없이 새 창고만 쓰기 시작하면, 사용자 입장에서는 "어제까지 있던 배경 사진이 갑자기 사라졌다"처럼 보인다.

그래서 이번 step에서는 앱을 열 때 자동으로 기존 데이터를 새 창고로 옮기는 "이사" 작업을 만들었다. 사용자는 아무것도 하지 않아도 되고, 이사가 끝나면 다시는 반복되지 않는다.

## 무엇을 만들었나?

`src/storage/assetMigration.js`를 새로 만들었다. 이 파일은 앱이 켜질 때 한 번, 딱 이 일만 한다.

1. "이미 이사 끝냈다"는 표시(완료 플래그)가 있는지 먼저 확인한다. 있으면 아무것도 안 하고 끝낸다.
2. 없으면 기존 다이어리 데이터 전체를 읽어서, 모든 날짜의 "다이어리"·"영화" 탭 안에 있는 그림 데이터(캔버스 JSON)를 하나씩 살펴본다.
3. 그 안에서 "사진처럼 보이는데 실제로는 긴 문자열(base64)로 박혀 있는 오브젝트"를 찾는다.
4. 찾은 사진 하나마다: 문자열을 실제 파일(Blob)로 되돌려서 새 창고(assetStorage)에 저장하고, 새 번호(id)를 받는다. 원래 있던 긴 문자열 자리는 지우고, 그 번호만 남겨둔다.
5. 사진 하나를 옮기다가 실패해도 그 사진만 건너뛰고 나머지는 계속 진행한다. 전체를 되돌리지(롤백하지) 않는다 — 한 장이 안 되더라도 나머지 데이터가 화면에서 통째로 사라지면 안 되기 때문이다.
6. 다 끝나면 바뀐 데이터 전체를 다시 저장하고, "이사 끝냈다" 표시를 남긴다.

그리고 캔버스를 화면에 그리는 쪽(`src/hooks/useFabricCanvas.js`)도 손봤다. 이제 그림 데이터를 불러올 때, 오브젝트에 번호(assetId)만 있고 실제 사진 데이터(src)가 없으면 새 창고에서 그 번호로 사진을 다시 꺼내 화면에 그릴 임시 링크를 만들어 채워 넣는다. 번호가 없는 오브젝트(글자, 도형 등)는 그대로 지나간다. 이 동작은 이번 마이그레이션 여부와 상관없이 항상 작동한다 — 앞으로(step 4에서) 새로 등록하는 배경 이미지도 똑같이 "번호만 저장, 필요할 때 다시 꺼내기" 방식을 쓰기 때문이다.

마지막으로 `src/App.jsx`에 이 이사 작업을 앱이 켜질 때 자동으로 한 번 실행하도록 연결했다. 이사 작업은 화면 그리기를 기다리게 하지 않는다 — 이사가 끝나기 전에도 기존 화면은 예전 방식(base64) 그대로 정상적으로 보인다.

## 데이터가 어떻게 흐르나?

```
[앱 켜짐]
   │
   ▼
'이사 끝냈다' 표시 있음? ──있음──▶ 아무것도 안 함
   │ 없음
   ▼
기존 다이어리 데이터 전체를 읽음
   │
   ▼
각 날짜 × {다이어리, 영화} 탭의 그림 데이터를 순회
   │
   ▼
"data:..."로 시작하는 사진 오브젝트 발견
   │
   ├─ 변환 성공 ──▶ 새 창고(assetStorage)에 저장 → 번호(assetId) 발급
   │                  오브젝트: src 제거, assetId 추가
   │
   └─ 변환 실패 ──▶ 오브젝트를 원본 그대로 둠, 다음 오브젝트로 계속
   │
   ▼
바뀐 데이터 전체 재저장 + '이사 끝냈다' 표시 남김


[화면에 그림을 다시 그릴 때] (이사 여부와 무관하게 항상)
   │
   ▼
오브젝트에 assetId 있음? ──없음──▶ 그대로 그림
   │ 있음
   ▼
새 창고에서 번호로 사진을 다시 꺼내 임시 링크 생성 → 그 링크로 그림
```

숫자로 예를 들면, 2026년 7월 18일 다이어리 탭에 배경 사진 1장이 base64로 박혀 있었다면, 이사 후에는 그 자리에 `assetId: "3f2a..."` 같은 짧은 번호 하나만 남는다. 사진 원본은 새 창고 안에 그대로 있고, 화면을 다시 그릴 때만 그 번호로 불러와 붙인다. 다이어리 데이터 안에 남는 문자열 길이가 (사진 크기에 비례하던 base64) 수백KB~수MB에서 번호 하나(수십 글자)로 줄어드는 셈이다.

## 쉽게 말하면

Step 0에서 큰 서랍장(IndexedDB)을 들여놓았다면, 이번 step은 예전 상자(localStorage) 안에 뒤섞여 있던 사진들을 꺼내서 서랍장에 정리해 넣고, 원래 상자 자리에는 "몇 번 서랍에 있음"이라고 적힌 쪽지만 남기는 이사 작업이다. 이사는 앱을 여는 순간 자동으로, 딱 한 번만 일어난다. 이사 중에 사진 한 장이 상해서 못 옮기면 그 한 장만 원래 자리에 그대로 두고 나머지 이사는 계속한다 — 방 전체를 다시 어지르지 않는다.

## 만든 파일

- `src/storage/assetMigration.js` — 이사 작업 본체 (`runAssetMigration`)
- `src/storage/assetMigration.test.js` — 이사가 실제로 되는지, 두 번 실행해도 안전한지 확인하는 자동 테스트
- `src/hooks/useFabricCanvas.js` (수정) — 그림 데이터를 불러올 때 번호(assetId)를 실제 사진으로 되돌리는 로직 추가
- `src/hooks/useFabricCanvas.test.js` — 위 되돌리기 로직에 대한 자동 테스트
- `src/App.jsx` (수정) — 앱이 켜질 때 이사 작업을 자동 실행

## 제대로 됐는지 어떻게 확인했나?

자동 테스트를 새로 6개 작성해서 전부 통과시켰다.

이사 작업(`assetMigration.test.js`) 2개:
1. base64 사진이 든 가짜 다이어리 데이터로 이사를 실행하면, 이사 후 데이터에서 그 오브젝트가 사진 문자열(src) 없이 번호(assetId)만 갖고 있는지, 나머지 위치·크기 정보는 그대로인지, 새 창고에서 그 번호로 원본과 동일한 내용을 다시 꺼낼 수 있는지 확인.
2. "이사 끝냈다" 표시가 이미 있는 상태에서 다시 실행하면 데이터가 전혀 바뀌지 않는지(같은 이사가 두 번 일어나지 않는지) 확인.

되돌리기 로직(`useFabricCanvas.test.js`) 4개: 번호가 있으면 임시 링크가 채워지는지, 번호가 없는 오브젝트는 그대로인지, 번호가 있어도 창고에 해당 사진이 없으면 원본 그대로 통과하는지, 오브젝트 목록 자체가 없는 그림 데이터는 그대로 반환되는지.

이번 phase 전체 완료 기준인 `npm run lint`, `npm run build`, `npm run test`(기존 91개 + 새로 만든 6개 = 총 97개)를 모두 실행해서 전부 통과하는 것을 확인했다.

테스트를 만드는 과정에서 step 0과 같은 함정을 다시 만났다: 테스트 환경에서는 `fetch`가 만드는 사진 데이터(Blob)가 새 창고를 오갈 때 내용이 깨지는 문제가 있었다(브라우저 실제 환경에서는 발생하지 않는, 테스트 도구 조합 특유의 문제). 실제 서비스 코드는 명세대로 `fetch`를 그대로 쓰고, 테스트 코드에서만 이 문제를 우회하는 방식으로 해결했다.

## 전체 그림에서 지금 어디?

```
Step 0: 창고 만들기 (완료)
        │
        ▼
[Step 1: 기존 데이터 이사시키기] ← 지금 여기, 완료
        │
        ▼
Step 2: 파일 등록 화면 만들기 (드래그앤드롭, 폴더 선택 등)
        │
        ▼
Step 3: 업로드한 폰트를 실제로 쓸 수 있게 등록
        │
        ▼
Step 4: 배경 이미지 설정을 새 창고에 연결
        │
        ▼
Step 5: Ctrl+V로 이미지 붙여넣기
```

이사를 창고 다음에 바로 한 이유는, 다음 단계들(파일 등록 화면, 배경 이미지 연결)이 새 창고를 전제로 동작을 만들기 때문이다. 이사를 나중으로 미루면 그 사이에 만들어질 화면들이 "새 데이터"와 "옛날 데이터"를 둘 다 신경 써야 해서 복잡해진다. 지금 미리 정리해두면, 앞으로는 항상 "번호로 저장하고 번호로 불러오기"라는 한 가지 방식만 신경 쓰면 된다.

## 아직 안 된 것 (솔직하게)

- 이번 step은 "옮기기"까지다. 사용자가 직접 파일을 골라 새로 등록하는 화면은 아직 없다 (step 2 몫).
- 커스텀 폰트를 실제로 목록에 띄우고 적용하는 부분은 아직 없다 (step 3 몫).
- 배경 이미지를 설정하는 버튼(`useCanvasBackground`)은 아직 예전 방식(base64)으로 새 사진을 저장한다 — 이번 step은 "옛날 데이터를 옮기는 것"만 했고, "새로 추가하는 사진을 새 창고에 바로 저장"하는 건 다음 step(4) 몫이다.
- 이사가 실패했을 때(예: 사진 하나가 깨져 있을 때) 사용자에게 "몇 개는 못 옮겼다"고 알려주는 화면은 만들지 않았다. 조용히 건너뛰고 나머지만 옮긴다.
