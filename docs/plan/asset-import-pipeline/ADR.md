# ADR — 에셋 입력 파이프라인 주요 결정

## ADR-1. 저장소를 localStorage(base64)에서 IndexedDB(Blob)로 전환

**맥락**: 기존 이미지 저장은 `FileReader.readAsDataURL()`로 base64 문자열을 만들어 Fabric JSON에 박고, 그 JSON 전체를 `localStorage`(`diary-app-data` 단일 키)에 저장했다. base64는 원본보다 약 1.33배 커지고, `localStorage`는 브라우저당 5~10MB 한도가 있다. 이번 phase는 이미지+폰트 파일을 다수 등록하는 것이 목적이라 이 한도에 곧바로 부딪힌다.

**선택지**:
- (A) 그대로 유지하고 용량 초과 시 경고만 표시
- (B) IndexedDB로 전환 (Blob 그대로 저장, 용량 한도가 사실상 수백MB~GB 단위)
- (C) 서버/클라우드 저장소 도입

**결정**: (B) IndexedDB로 전환.

**결과(트레이드오프)**: 서버 없음·클라이언트 전용이라는 프로젝트 전제([[project_...]] 로드맵 참고)를 유지하면서 용량 문제를 근본적으로 해결한다. 대신 기존 base64 데이터를 그대로 못 쓰므로 마이그레이션(ADR-2)이 필요해지고, 이미지/폰트를 다루는 코드가 전부 비동기(Promise 기반 IndexedDB API)를 거치도록 바뀌어 구현 복잡도가 늘어난다.

## ADR-2. 기존 localStorage 데이터 자동 마이그레이션

**맥락**: 사용자가 실제로 `localStorage`에 보존이 필요한 다이어리 데이터(배경 이미지 포함)를 갖고 있다고 확인함. 저장소를 IndexedDB로 바꾸면 새 코드는 기존 base64 데이터를 읽지 못해 사용자 입장에서 데이터가 사라진 것처럼 보인다.

**선택지**:
- (A) 마이그레이션 없음 — 사용자가 수동으로 백업/재입력
- (B) 앱 시작 시 자동 마이그레이션 — base64 이미지를 감지해 Blob으로 변환 후 IndexedDB에 저장, Fabric JSON은 참조 id로 교체
- (C) 두 저장소를 동시에 계속 지원 (레거시 호환 레이어 영구 유지)

**결정**: (B) 앱 시작 시 1회성 자동 마이그레이션.

**결과(트레이드오프)**: 사용자가 아무 조치 없이 기존 데이터를 그대로 사용할 수 있다. 대신 마이그레이션 로직 자체가 이번 phase에 추가 작업량으로 들어가고, 완료 플래그 관리(중복 실행 방지)가 필요하다. (C)처럼 영구 호환 레이어를 두지 않으므로, 마이그레이션 이후에는 base64 저장 경로를 코드에서 완전히 제거한다 — 레거시 분기를 남겨두지 않는다.

## ADR-3. 폰트 목록을 기존 Google Fonts와 통합

**맥락**: 현재 `TextMemoButton`은 Google Fonts CDN 8종을 하드코딩한 드롭다운을 갖고 있다. 커스텀 폰트를 추가하면 이 목록과의 관계를 정해야 한다.

**선택지**:
- (A) 기존 8종 + 업로드 폰트를 하나의 드롭다운에 합침
- (B) "기본 폰트" / "내 폰트" 두 섹션(또는 별도 UI)으로 분리

**결정**: (A) 하나의 드롭다운에 합침.

**결과(트레이드오프)**: 사용자 입장에서 폰트 선택 흐름이 단순해진다. 대신 폰트 목록이 상태(고정 배열 + IndexedDB에서 로드된 동적 목록)를 합쳐 관리해야 하므로, `TextMemoButton`이 정적 상수 배열만 참조하던 구조에서 IndexedDB 조회를 포함하는 구조로 바뀐다.

## ADR-4. 드래그앤드롭 반응 범위를 에셋 패널로 한정

**맥락**: 드래그앤드롭을 캔버스 전체 또는 화면 전체에 적용할지, 이번 phase가 만드는 에셋 입력 컴포넌트 영역에만 적용할지 정해야 한다. 캔버스는 이미 Fabric의 자체 드래그 동작(오브젝트 이동)을 갖고 있어 충돌 가능성이 있다.

**선택지**:
- (A) 에셋 관리 패널/버튼 영역에만 드롭 반응
- (B) 캔버스(에디터 화면) 전체에 드롭 반응

**결정**: (A) 에셋 패널 영역에만 한정.

**결과(트레이드오프)**: 캔버스의 기존 오브젝트 드래그 동작과 충돌할 여지가 없어 구현이 단순하고 안전하다. 대신 사용자가 캔버스에 직접 이미지를 드롭하는 (아마 더 직관적일 수 있는) 경로는 이번 phase 범위에 포함하지 않는다 — 필요해지면 이후 별도 논의.

## ADR-5. 클립보드 붙여넣기 우선순위: 이미지 파일 우선, 없으면 오브젝트 붙여넣기로 폴백

**맥락**: `useCanvasKeyboardShortcuts`가 이미 Ctrl+V를 캔버스 오브젝트 복제(clipboardRef 기반)로 쓰고 있다. 이번 phase가 추가하는 "클립보드 이미지 파일 붙여넣기"와 단축키가 겹친다.

**선택지**:
- (A) 캔버스에 포커스가 있어도, 시스템 클립보드에 이미지 파일이 있으면 이 파이프라인이 우선 처리하고 없으면 기존 오브젝트 붙여넣기로 폴백
- (B) 에셋 패널에 포커스가 있을 때만 이미지 붙여넣기 활성화 (캔버스 포커스 시엔 기존 동작만)

**결정**: (A) 이미지 파일 우선, 폴백 방식.

**결과(트레이드오프)**: 사용자가 포커스 위치를 신경 쓰지 않고 어디서든 Ctrl+V로 이미지를 붙여넣을 수 있어 사용성이 좋다. 대신 매 Ctrl+V 입력마다 클립보드 내용(이미지 유무)을 먼저 비동기로 확인해야 하므로, 기존 동기적이던 붙여넣기 핸들러가 비동기 분기를 갖게 된다.
