# PRD — 에셋 입력 파이프라인 (asset-import-pipeline)

## 배경 / 왜

지금까지 배경 이미지·업로드 이미지는 전부 `<input type="file">` 클릭 → `FileReader.readAsDataURL()` → base64 문자열을 Fabric JSON 안에 그대로 넣어 `localStorage`(`diary-app-data` 단일 키)에 저장하는 방식이었다. 폰트는 Google Fonts CDN 8종 고정이라 사용자가 파일을 추가할 방법이 아예 없다.

이 방식은 두 가지 한계가 있다:
1. base64는 원본보다 약 1.33배 커지고, `localStorage`는 브라우저당 5~10MB 한도가 있어 이미지를 몇 개만 추가해도 용량 한도에 부딪힐 수 있다.
2. 파일 선택 방식이 "버튼 클릭 → 파일 하나 선택"뿐이라, 여러 파일을 한 번에 등록하거나(폴더째로), 드래그앤드롭·클립보드 붙여넣기로 빠르게 추가하는 경로가 없다.

로드맵상 다음 phase인 `sticker-studio`(스티커 전용 에디터)가 동일한 "에셋 등록" 입력 방식을 재사용할 예정이므로, 이번에 개별 기능이 아니라 **재사용 가능한 공용 입력 파이프라인**으로 만든다.

## 목표

- 배경 이미지와 커스텀 폰트 파일을 여러 경로(파일 선택/폴더 일괄/드래그앤드롭/클립보드 붙여넣기)로 등록할 수 있는 공용 입력 컴포넌트·저장 계층을 만든다.
- 저장소를 `localStorage` base64 방식에서 `IndexedDB` 기반으로 전환해 용량 한계를 근본적으로 해결한다.
- 기존에 `localStorage`에 저장돼 있던 사용자의 실제 다이어리 데이터(배경 이미지 포함)를 데이터 손실 없이 `IndexedDB`로 자동 이관한다.
- 이번 phase에서 만든 공용 입력 파이프라인을 캔버스 배경 설정과 텍스트 메모 폰트 선택 두 곳에 실제로 연결해, 지금 당장 사용 가능한 상태로 만든다.

## 대상 사용자

기존과 동일. 혼자 쓰는 개인 다이어리 사용자.

## In-scope

### 1. IndexedDB 저장 계층 신설

- 이미지·폰트 파일을 Blob 형태로 저장하는 새 IndexedDB 기반 스토리지 모듈을 만든다 (base64 인코딩 오버헤드 없이 원본 그대로 저장).
- 각 에셋은 고유 id, 타입(`image` | `font`), 원본 파일명, MIME 타입/확장자, Blob 데이터를 갖는다.
- 폰트 에셋은 추가로 폰트 패밀리명(파일에서 유추하거나 사용자 입력)을 갖는다.

### 2. 앱 시작 시 자동 마이그레이션

- 앱 로드 시 기존 `localStorage`(`diary-app-data`)에 데이터가 있는지 확인한다.
- 있다면: Fabric JSON 안에 박혀 있는 base64 이미지(`isBackground: true` 오브젝트 포함, 그 외 일반 이미지 오브젝트도 포함)를 찾아 Blob으로 변환해 IndexedDB에 저장하고, Fabric JSON의 해당 필드는 IndexedDB 참조(id)로 교체한다.
- 마이그레이션이 끝나면 완료 플래그를 남겨 다음 실행부터는 재실행하지 않는다.
- 마이그레이션 대상이 없거나 이미 완료된 경우 아무 동작도 하지 않는다.

### 3. 공용 에셋 입력 컴포넌트

- 이미지·폰트 파일을 등록하는 공용 UI 컴포넌트(패널 또는 버튼 영역)를 만든다. `sticker-studio` 등 이후 phase가 그대로 재사용할 수 있는 형태로 설계한다.
- 지원하는 입력 경로:
  - 파일 선택(`<input type="file" multiple>`)
  - 폴더 선택(디렉토리 입력) — 하위 폴더까지 재귀적으로 읽고, 확장자로 이미지/폰트 파일을 자동 구분해 각각 등록. 그 외 확장자는 무시.
  - 드래그앤드롭 — 이 컴포넌트(에셋 관리 패널/버튼) 영역에 드롭했을 때만 반응. 캔버스나 화면 전체 드롭에는 반응하지 않는다.
  - 클립보드 붙여넣기(Ctrl+V) — 이미지 파일 한정. 캔버스에 포커스가 있는 상태에서도 클립보드에 이미지 파일이 있으면 이 파이프라인이 우선 처리하고, 없으면 기존 Fabric 오브젝트 붙여넣기(`useCanvasKeyboardShortcuts`)로 폴백한다.
- 허용 형식:
  - 이미지: jpg, png, gif, webp. 용량 제한 없음.
  - 폰트: ttf, otf, woff, woff2.

### 4. 폰트 등록 및 적용

- 업로드된 폰트 파일은 `FontFace` API로 동적 로드해 문서에 등록한다.
- `TextMemoButton`의 폰트 드롭다운에 기존 8종 Google Fonts와 업로드된 커스텀 폰트가 하나의 목록으로 합쳐져 표시된다.

### 5. 기존 UI 통합

- `useCanvasBackground`의 이미지 배경 설정 경로를 새 공용 입력 컴포넌트/IndexedDB 저장으로 교체한다. 배경 이미지를 Fabric 오브젝트로 추가하는 기존 동작(`isBackground` 태그, lockBackground 등)은 유지하되, 데이터 출처만 base64 대신 IndexedDB 참조로 바뀐다.
- `TextMemoButton`의 폰트 목록을 새 폰트 저장 계층과 연결한다.

## Out-of-scope (명시)

- `sticker-studio` 자체(스티커 전용 에디터 UI, 스티커 편집 기능) — 이번엔 그 phase가 재사용할 공용 입력 파이프라인만 만든다.
- 이미지 압축/리사이즈, 폰트 서브셋팅 등 최적화 처리 — 원본 그대로 저장.
- 에셋 삭제/관리(목록에서 지우기, 용량 확인 UI) — 이번 phase는 "등록"까지만.
- 폴더 선택 시 이미지/폰트 외 파일 형식에 대한 안내/에러 메시지 UI — 조용히 무시.
- `youtube-embed`, `desktop-custom`, `movie-review-standalone` 등 로드맵의 다른 phase 범위.
- `ui-polish` 범위의 스타일 정비.

## 성공 기준

- 이미지/폰트 파일을 파일 선택, 폴더 선택, 드래그앤드롭, 클립보드 붙여넣기(이미지) 네 가지 경로 모두로 등록할 수 있다.
- 등록된 에셋은 IndexedDB에 Blob 형태로 저장되며, `localStorage` 용량 한도와 무관하게 여러 개(수십 개 이상)의 이미지를 추가해도 문제없이 동작한다.
- 앱을 새로고침해도 기존에 `localStorage`에 저장돼 있던 다이어리(배경 이미지 포함)가 그대로 보이며, 데이터가 유실되지 않는다.
- 배경 이미지 설정 시 새 파이프라인을 거쳐 IndexedDB에 저장되고, 캔버스에 정상적으로 표시·저장·재로드된다.
- 업로드한 폰트 파일이 텍스트 메모 폰트 드롭다운에 나타나고, 선택해 적용하면 캔버스에 해당 폰트로 렌더링된다.
- 캔버스에 포커스가 있는 상태에서 Ctrl+V를 눌렀을 때, 클립보드에 이미지가 있으면 이미지가 추가되고, 없으면 기존 오브젝트 붙여넣기가 정상 동작한다.
- 위 변경 전체에 걸쳐 기존 캔버스 저장/불러오기, JSON/PNG export, 오토세이브, 오브젝트 복사/삭제 기능이 이전과 동일하게 동작한다.

## 진행 방식

1. 문서(PRD/ADR/architecture) 합의 → 실행 승인
2. IndexedDB 저장 계층 → 마이그레이션 → 공용 에셋 입력 컴포넌트 → 폰트 등록/적용 → 배경 이미지 통합 → 완료 시 사용자에게 결과 설명
