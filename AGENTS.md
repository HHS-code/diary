# CLAUDE.md

## 핵심 원칙

- 처음 읽는 사람도 한눈에 이해할 수 있는 코드.
- 이름은 의도를 드러낸다. 제어 흐름은 단순하게. 부작용은 함수명에서 드러나게.
- 영리한 코드보다 단순하고 명시적인 코드. 가독성·유지보수성·테스트 용이성 우선.

## 작업 태도

- **추측 금지**: 가정은 명시하고, 불확실하면 묻는다. 해석이 여럿이면 고르지 말고 제시한다.
- **단순함 우선**: 요청한 것만. 추측성 기능·단발성 추상화·불필요한 설정·불가능한 케이스의 에러 처리 금지. 200줄이 50줄로 되면 다시 쓴다.
- **수술적 변경**: 요청과 직접 연결되는 줄만 건드린다. 주변 코드·포맷·주석 "개선" 금지, 멀쩡한 것 리팩터 금지, 기존 스타일에 맞춘다. 내 변경이 만든 미사용 import/변수만 제거하고, 기존 dead code는 발견 시 삭제하지 말고 언급만.
- **목표 주도**: 모호한 과제를 검증 가능한 목표로 바꾼다 ("버그 수정" → "재현 테스트 작성 후 통과"). 다단계는 간단한 계획+검증 기준을 먼저 밝힌다.
- **문서 우선**: DOCS폴더안에 있는 PRD/TRD를 최우선으로 코드를 맞춘다. 문서 변경은 요청 시에만.

## Python 코딩 규칙 (`capstone/ontology_rule_matching/**/*.py`)

### 네이밍

- PEP 8: 함수·변수·모듈·패키지 `snake_case`, 클래스 `PascalCase`, 상수 `UPPER_SNAKE_CASE`.
- 함수명은 구현이 아니라 의도. `동사_목적어` (`create_user`, `calculate_order_total`). `process`·`handle`·`do_task` 금지.
- 불리언은 질문형: `is_`, `has_`, `can_`, `should_`.
- 순수/변환: `calculate_`·`format_`·`parse_`·`normalize_`·`build_` / 부작용: `save_`·`update_`·`delete_`·`send_`·`publish_`·`fetch_`·`load_`.
- 조회 의도 구분: `get_`(존재 기대), `find_`(없을 수 있음), `fetch_`(외부 API), `load_`(파일·DB·설정).

### 함수 스타일

- 작은 순수 함수 선호. 핵심 로직은 부작용 없이. I/O·DB·네트워크·시간·랜덤·환경 접근은 바깥 레이어로.
- 인자 in-place 변경은 함수명에 명시(`_in_place`). 기본은 새 값 반환.
- 복잡한 `map`/`filter`/`lambda`보다 컴프리헨션. 읽기 어려우면 중간 변수. 계산처럼 보이는 함수에 숨은 부작용 금지.

### 제어 흐름

- 들여쓰기 depth ≤ 2 (제어 구조만 카운트). 넘으면 함수 추출·가드 절·early return.
- 삼항 연산자 금지: `x if cond else y` → `if`/`else`.

### 타입 힌트

- public 함수는 타입 힌트·반환 타입 필수(`-> None` 포함). `None` 가능 시 `User | None`으로 명시.

### 테스트

- 테스트 이름은 동작을 설명(길어도 OK). 가능하면 given / when / then.
