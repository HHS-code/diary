# step-build

합의된 계획 문서를 **자기완결적 step으로 쪼개고, 각 step을 깨끗한 컨텍스트의 새 세션으로 실행**하는 워크플로우. coplan의 짝 — coplan이 "합의 문서"를 만들면, step-build가 그걸 **분해 → 실행 → 사람용 리포트**로 옮긴다.

> **도메인 중립.** 이 커맨드는 어떤 프로젝트에도 쓴다. 아래 "레이어/모듈"은 일반어이며, 구체값(스택·검증 커맨드·허용 도구·리포트 독자 수준)은 **프로젝트가** 채운다. 차트/특정 도메인 용어를 이 워크플로에 박지 않는다.

## 전제 (없으면 멈추고 coplan으로)

- coplan으로 PRD/ADR/architecture(또는 그에 준하는 합의 문서)가 `docs/plan/<slug>/`에 있다.
- 열린 OQ(미해결 질문)가 0이다.

## A. 분해 설계 (대화형 — 사람 승인)

1. 합의 문서를 읽는다. 넓으면 Explore 병렬.
2. **의존순서**로 step 목록 초안을 만든다. 설계 원칙:
   - **한 step = 한 레이어/모듈.** 여러 모듈을 동시에 고쳐야 하면 step을 쪼갠다.
   - **자기완결성.** 각 step은 독립 세션에서 실행된다. "이전 대화처럼" 같은 외부 참조 금지 — 필요한 정보는 step 안에.
   - **사전 준비 강제.** 읽어야 할 문서·이전 step 산출 파일 경로를 명시한다.
   - **시그니처 수준.** 함수/클래스 인터페이스만 제시, 내부 구현은 에이전트 재량. 단 벗어나면 안 되는 핵심 규칙(보안·데이터 무결성 등)은 명시.
   - **AC = 실행 가능한 커맨드.** 프로젝트의 검증 커맨드(예: 린트+테스트). "~가 동작해야 한다" 같은 추상 서술 금지.
   - **금지는 구체적으로.** "X 하지 마라. 이유: Y".
   - **이름은 kebab-case slug.**
3. 사용자 승인을 받는다. **이 승인이 실행 게이트** — 막연한 "go"가 아니라 상세 명세 승인이다.

## B. 분해 파일 작성 (`steps/`)

러너가 안정적으로 읽도록 step을 `docs/plan/<slug>/steps/`에 둔다:
- `steps/index.json` — 상태머신.
  `{ "project": "...", "phase": "<slug>", "steps": [ {"step": 0, "name": "<slug>", "status": "pending"}, ... ] }`
- `steps/step{N}.md` — step별 지시. **상세 블록은 그 step 직전에 채운다(just-in-time)** — A에서 index.json 목록만 확정해도 된다.

step{N}.md 형식:
```
# Step N: <slug>
## 읽을 파일
- <문서·이전 산출 경로>
## 작업
<시그니처 수준 지시 + 벗어나면 안 되는 핵심 규칙>
## AC
<실행 가능한 검증 커맨드>
## 금지
- <X 하지 마라. 이유: Y>
```

## C. 실행 (러너 — 자동, headless)

실행 엔진: `$CLAUDE_PROJECT_DIR/.claude/scripts/step-build.py` (이 레포에 포함 — 현재 폴더의 프로젝트 대상)
```
python "$CLAUDE_PROJECT_DIR/.claude/scripts/step-build.py" <plan-dir>   # 예: docs/plan/<slug>
```

러너가 자동으로:
- `feat-<...>` 브랜치 생성/checkout (main 안 건드림)
- **가드레일 주입**: `CLAUDE.md` + `docs/*.md`를 매 step 프롬프트에 포함
- **컨텍스트 누적**: 완료 step의 한 줄 `summary`만 다음 프롬프트에 전달
- step마다 **새 headless 세션**(`claude -p`) → 깨끗한 컨텍스트
- **자가교정**: 실패 시 최대 3회 재시도(직전 에러를 프롬프트에 피드백)
- **자동정지**: 3회 실패 또는 `blocked`(사람 필요) → 멈추고 사람 호출
- **상태/재개**: `index.json`(pending/completed/error/blocked) — 중단 후 재실행 시 이어감
- step당 커밋

각 headless 세션이 step 완료 시:
1. (TDD) **실패 테스트 먼저** → 구현 → AC 실행
2. **사람용 리포트** 작성 (아래 레시피)
3. `index.json` status=completed + `summary`(한 줄, 다음 step용)
4. 커밋

## 안전층 (사람 승인 대신 *기계적* — 프로젝트 `.claude/`가 설정)

headless라 실시간 승인이 없다. 대신 프로젝트가 보장:
- **PreToolUse 거부 훅**: 위험 명령 차단(`rm -rf`, 네트워크 `curl`/`wget`, force push, `reset --hard` 등). **권한 모드와 무관하게 작동 = 진짜 안전망.**
- **tdd_gate 훅**: 실패 테스트 없이 소스 작성 차단.
- **Stop 훅**: 세션 끝마다 검증 커맨드(린트+테스트) 재실행 → 깨졌으면 세션을 못 끝낸다(에이전트 자가수정 강제).
- **권한**: `--dangerously-skip-permissions` + 위 훅 (신뢰 입력 빌드 전제). 빌드가 외부 신뢰 못 할 데이터를 만지면 화이트리스트로 조일 것.

## 리포트 레시피 (사람용 — 통제감)

각 step 리포트: `docs/plan/<slug>/reports/stepN-<slug>.md`.

- **독자**: 해당 도메인은 알고 개발 지식은 약한 사람(정확한 수준은 프로젝트가 지정). 도메인 용어는 정의하지 않고, **독자가 모를 개발 용어만 나오는 즉시 한 줄로 풀이**한다.
- **원칙**: 이야기 흐름 / 쉬운 말 / 구체적 숫자 예시 / 시그니처 대신 "무슨 일을 하는지" / 솔직함. **이모지 안 씀. 다이어그램은 ASCII.**
- **섹션**:
  ```
  # Step N: <제목>
  ## 이 단계를 왜 했나?            (배경·동기)
  ## 무엇을 만들었나?               (쉬운 말)
  ## 데이터가 어떻게 흐르나?        (ASCII 그림 + 풀이 + 숫자 예시)
  ## 쉽게 말하면                    (비유 한 개)
  ## 만든 파일
  ## 제대로 됐는지 어떻게 확인했나?  (AC 결과 = 기계적 사실)
  ## 전체 그림에서 지금 어디?       (진행 지도 + 다음이 왜 그건지)
  ## 아직 안 된 것 (솔직하게)
  ```
- **전체 줄거리 인덱스** `reports/README.md`는 러너가 `index.json`에서 **기계적으로** 생성(step명·상태·summary·링크) — LLM이 안 써서 항상 정확.

## 불변 규칙

- **합의 문서 밖은 구현하지 않는다.** 필요하면 멈추고 coplan으로 돌아가 합의를 갱신한다.
- **분해 설계(A) 승인이 실행 게이트다.** 러너 실행은 그 승인 위에서만 돈다.
